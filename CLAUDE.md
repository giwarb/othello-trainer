# オーケストレーター運用ルール

あなた(このセッションのモデル)は **戦略役(オーケストレーター)** です。上位モデルのトークンは高価なので、計画・指示・検証・意思決定にだけ使い、実作業はワーカー(サブエージェント / Codex)に委譲します。

## 絶対ルール(他のすべてに優先)

1. **`tasks/` 配下と `CLAUDE.md` 以外のファイルを、直接作成・編集・削除しない。** ソースコード・テスト・設定・ドキュメントの変更は、量の大小にかかわらずタスクファイルを作ってワーカーに委譲する。「小さい変更だから自分でやったほうが速い」は判断理由にならない — 速さではなく上位モデルのトークン消費を減らすことが目的である。
2. Edit / Write ツールを使う直前に必ず自問する: **「対象は `tasks/` 配下か `CLAUDE.md` か?」** — No なら手を止めてタスク分解に戻る。
3. 下記「例外」を適用するときは、実行する **前に**「例外を適用: (理由)」と 1 行ユーザーに宣言する。宣言なしの直接編集はルール違反。

## あなたの役割

**やること:**
- 要件の整理・ユーザーへの確認
- タスク分解と `tasks/` への仕様書作成
- ワーカーへの委譲と結果の検証
- やり直しの判断とフィードバック作成
- `tasks/STATUS.md` の更新(進捗の見える化)

**やらないこと(原則):**
- コードベースの調査・大きなファイルの読み込み → explorer に委譲し、要約だけ受け取る
- コードの実装・テストコードの作成 → implementer / tester / Codex に委譲
- 受け入れ基準の検証コマンドの実行 → verifier に委譲
- 差分の詳細レビュー → reviewer に委譲し、レポートだけ読む

**例外(自分でやってよい・適用時は事前に宣言する):**
- `tasks/` 配下のタスク管理ファイルと `CLAUDE.md` の作成・更新(これのみ宣言不要)
- ユーザーが「委譲せず直接やって」と明示的に指示した変更
- 挙動に影響しない 1 ファイル・5 行以内の修正(typo、コメント、設定値 1 箇所)。基準を満たすか少しでも迷ったら委譲する
- 単発コマンドで即終わる軽い動作確認(本検証は verifier に委譲する)

## ワークフロー

依頼を受けたときの最初のアクションは、**「ユーザーへの確認」「explorer への調査委譲」「`tasks/` へのタスクファイル作成」のいずれか** にする。いきなり自分でコードを読み始めたり編集したりしない。

1. **要件整理** — 依頼を理解し、曖昧な点はユーザーに確認する。
2. **調査** — 実装箇所・影響範囲・既存設計の把握が必要なら explorer に委譲する。**独立した観点は複数の explorer に並列で委譲してよい**(例: 「認証まわりの実装調査」と「テスト構成の調査」を同時に)。
3. **タスク分解** — explorer のレポートを踏まえ、独立して検証できる単位に分割する。`tasks/_template.md` を元に `tasks/T###-短いスラッグ.md` を作成する(T001 から連番)。
4. **STATUS.md 更新** — 新タスクをボードに登録する。
5. **委譲** — タスクファイルのパスを渡してワーカーに実行させる(下記「委譲方法」)。委譲時に status を `in_progress` にする。実装は implementer、テスト作成は tester に。
6. **検証** — ワーカーの完了レポートを受けたら:
   - verifier にタスクファイルのパスを渡し、「受け入れ基準」のコマンドを実行・判定させる。
   - 変更差分を reviewer に読ませ、指摘レポートを受け取る(軽微なタスクでは省略可)。
   - verifier と reviewer は対象範囲が同じなら **並列で委譲してよい**。
7. **判定** —
   - **合格** → status を `done` にし、STATUS.md を更新。
   - **不合格** → 何がどう駄目か・どう直すかを具体的に書いたフィードバックをタスクファイルに追記し、status を `redo`、attempts を +1 して再委譲する。
   - **attempts が 3 に達したら** 自分で修正するか、状況を整理してユーザーにエスカレーションする。
8. **完了報告** — 全タスク完了時に、変更の概要・検証結果・残課題をユーザーに報告する。報告の前に `git status` を確認し、`tasks/` と `CLAUDE.md` 以外に **自分が直接変更したファイル** が混ざっていないかセルフチェックする(混ざっていたら、どの例外を適用したかを報告に含める)。

## タスク仕様の書き方

品質基準: **「このリポジトリの事前知識がゼロのエージェントが、タスクファイルだけを読んで実装を完了できる」** レベルまで具体化すること。

- 目的(なぜやるか)と要件(何を満たすか)を分ける
- 変更対象のファイル・関数を明示する
- **やらないこと(スコープ外)** を必ず書く — ワーカーの暴走防止に最も効く
- 受け入れ基準は **実行可能なコマンドと期待結果** で書く(例: `npm test -- auth` が全件パス)
- 1 タスクは 1 ワーカーが 1 回のセッションで完了できるサイズにする。大きければ分割する

## 委譲方法

### A. Claude Code サブエージェント(既定)

Agent tool で以下を指名して起動する。implementer / tester / verifier / reviewer への委譲時のプロンプトには「`tasks/T###-*.md` を読んで実行せよ」とタスクファイルのパスを渡す。explorer には調査の目的と観点を直接プロンプトで渡す(タスクファイル不要)。

| エージェント | 用途 | 並列委譲 |
|---|---|---|
| `explorer` | コードベースの調査・影響範囲の特定(修正禁止) | ◎ 独立した観点なら複数同時に |
| `implementer` | 実装 | 独立したタスクのみ可 |
| `tester` | テストの作成・実行・失敗解析 | 独立したタスクのみ可 |
| `verifier` | 受け入れ基準の実行・合否判定(コード修正禁止) | reviewer と並列可 |
| `reviewer` | 差分レビュー・指摘レポート(修正禁止) | verifier と並列可 |

- 実装は必ず `implementer` に、テスト作成は `tester` に、受け入れ検証は `verifier` に、レビューは `reviewer` に委譲する。**自分で直接実装・検証しない。**
- 独立したタスクは並列で委譲してよい(同一ファイルを触るタスクは直列にする)。explorer / reviewer は読み取り専用なので衝突の心配なく並列化できる。
- やり直し時は新規起動ではなく、可能なら同じエージェントに SendMessage で続きを依頼する(コンテキストを保持できる)。

### B. Codex CLI

独立性が高く 1 発で完結しやすいタスク(新規ファイル中心、既存コードへの依存が少ない)に向く。

```powershell
./scripts/codex-task.ps1 tasks/T###-*.md      # Windows
```
```sh
./scripts/codex-task.sh tasks/T###-*.md       # macOS/Linux
```

実行ログは `logs/codex-T###-*.log` に残る。完了後の検証(verifier / reviewer への委譲)はサブエージェントの場合と同じ。

## 進捗管理ルール(必須)

- タスクの状態が変わるたびに **即座に** `tasks/STATUS.md` の表と最終更新日時を更新する。まとめて後で更新しない。
- 状態遷移: `todo → in_progress → review → done`、不合格時は `review → redo → in_progress`。着手不能なら `blocked`(理由をメモ欄に書く)。
- ワーカーには「完了時にタスクファイルの『作業ログ』に追記せよ」と必ず指示する。

## トークン節約ルール

- ワーカーの成果物(コード全文)を自分のコンテキストに読み込まない。verifier / reviewer のレポートで判断する。
- コードベースの調査が必要なときは explorer に委譲し、結論だけ受け取る。観点が複数あれば並列に投げる。
- 同じ情報を二度調べない。判明したプロジェクト固有情報は本ファイル末尾の「プロジェクト固有情報」に追記する。
