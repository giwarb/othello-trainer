# オセロ練習アプリ 完全設計書

対象: GitHub Pages 配信の静的 PWA / ベンチマーク: DroidShimax 級の盤面評価と応答速度

---

## 0. 設計の要点(サマリ)

| 判断ポイント | 結論 | 理由 |
|---|---|---|
| エンジン実装 | **Rust → WASM**(Web Worker 内で実行) | JS 比で探索速度 20〜50 倍。DroidShimax/Edax 級はネイティブ相当の速度が前提 |
| 評価関数 | **Edax 系パターン評価**(ステージ別・int16 量子化重み) | DroidShimax もこの系統。実績・速度・精度のバランス最良 |
| 並列化 | coi-serviceworker で SharedArrayBuffer を有効化し **Lazy SMP** | GitHub Pages はカスタムヘッダ不可 → SW でヘッダ注入する定石手法 |
| 定石データ | 局面ハッシュベースの DAG(合流対応)+ 8 対称正規化 | 手順木だと転換(transposition)を扱えない |
| 進捗保存 | IndexedDB + 定石ラインの SRS(間隔反復) | 「覚える」の本質は復習スケジューリング |
| 悪手分析 | 石差換算のロス閾値で分類 + 悪手以降の分岐探索 UI | 評価値の生数値より石差ロスが直感的 |

## 1. 全体アーキテクチャ

```
┌─────────────────────────── ブラウザ ───────────────────────────┐
│  UI 層 (Vanilla TS or Preact / Canvas 盤面)                      │
│   ├ 定石練習  ├ 中盤練習  ├ 詰めオセロ  ├ 棋譜解析  ├ 対局      │
│                    │  postMessage (JSON プロトコル)              │
│  Engine Worker (WASM: Rust)                                     │
│   ├ Bitboard 着手生成      ├ PVS + MPC 中盤探索                 │
│   ├ パターン評価 (ステージ別)  ├ 終盤完全読みソルバー            │
│   └ 置換表 (SAB 共有 / Lazy SMP で N スレッド)                   │
│  データ層                                                        │
│   ├ IndexedDB: 進捗 / SRS / 棋譜ライブラリ / 解析キャッシュ      │
│   └ Cache Storage: アプリ本体 + 評価重み + 定石 DB (SW 管理)     │
└─────────────────────────────────────────────────────────────────┘
静的アセット (GitHub Pages):
  /app 本体, /data/eval-weights.bin.br, /data/joseki.bin, /data/puzzles/*.bin
```

### 技術スタック

- **エンジン**: Rust + wasm-bindgen(`wasm32-unknown-unknown`、SIMD128 有効ビルド)。C++/Emscripten でも可だが、Rust の方がビルド・配布が単純。
- **UI**: 依存最小(Preact + TS 程度)。盤面は Canvas 1 枚(石・合法手ヒント・最終手マーク・アニメーションをレイヤ描画)。
- **グラフ**: 評価グラフは Canvas 自前描画(60 手 × 折れ線 1 本なのでライブラリ不要。軽さ優先)。
- **PWA**: Workbox は使わず素の Service Worker(キャッシュ戦略が単純なため)。

### Worker プロトコル(例)

```jsonc
// UI → Engine
{ "id": 42, "cmd": "analyze",
  "board": { "black": "0x0000000810000000", "white": "0x0000001008000000", "turn": "black" },
  "limit": { "depth": 20, "timeMs": 1000, "exactFromEmpties": 24 },
  "multiPV": 3 }

// Engine → UI (思考中は逐次、確定で final:true)
{ "id": 42, "final": false, "depth": 14,
  "pv": ["f5","d6","c3"],
  "score": { "type": "midgame", "discDiff": 2.4 },   // 終盤は type:"exact"
  "nodes": 18400000, "nps": 21000000,
  "moves": [ { "move": "f5", "discDiff": 2.4 }, { "move": "d6", "discDiff": 1.1 } ] }
```

`id` で要求と応答を対応付け、`cancel` コマンドで探索中断(atomic フラグ)。UI は常に非同期・キャンセル可能で、盤面操作が探索にブロックされない。

## 2. エンジン設計(DroidShimax 級の中身)

### 2.1 盤面表現

- `black: u64, white: u64` のビットボード。
- 着手生成: Kogge–Stone 系の方向別シフト(8 方向)。WASM SIMD128 で 4 方向を並列化。
- 反転計算: 方向別の carry-propagation。`flip(pos, P, O) -> u64`。
- Zobrist ハッシュ(64bit)+ 8 対称正規化ハッシュ(定石 DB 参照用のみ。探索中は通常ハッシュ)。

### 2.2 探索

| 要素 | 内容 |
|---|---|
| 基本 | 反復深化 + PVS(NegaScout)+ Aspiration Window |
| 置換表 | 2-tier(depth 優先 + always-replace)、SAB 上に配置しスレッド共有。既定 64〜128MB |
| 枝刈り | **Multi-ProbCut**(浅い探索の評価から深い探索の結果を線形回帰で予測して確率的カット。Edax の強さの核。カットペアの回帰係数は自己対戦データから学習) |
| 手順前後 | Enhanced Transposition Cutoff |
| 着手順序 | TT 手 → 浅い探索評価順 → 相手着手可能数最小化 → 隅・辺ボーナス |
| 並列 | Lazy SMP(全スレッド同一局面を微妙に異なる深さで探索、TT 共有)。実装が単純で WASM と相性が良い |

### 2.3 終盤ソルバー(完全読み)

- 空きマス **24 以下で石差完全読み**、26 以下で勝敗判定読みへ自動切替(時間設定で可変)。
- 空き 8 以下: **奇偶(parity)順序付け** + 隅優先。空き 4/3/2/1 は展開済み専用関数(関数呼び出し・ループを排したハードコード)。
- 安定石による静的カット(確定石数から到達不能な α/β を即断)。
- NWS(null window search)主体 + TT。

### 2.4 評価関数(ここが DroidShimax 級の本体)

**ステージ別パターン評価**(Logistello → Edax 系譜):

- パターンセット(Edax 準拠、計 46 インスタンス/局面):
  - edge+2X(10 マス)、corner 2×5(10)、corner 3×3(9)
  - 斜めライン d4〜d8、横縦ライン 2/3/4 列目
  - 補助特徴: 着手可能数差、パリティ
- 各パターンは 3^n 通りの重み(int16)を持ち、**手数 4 手刻みの 15 ステージ**で別テーブル。
- 評価 = 46 パターンの重みテーブル参照の総和(乗算なし・テーブル参照のみ → 1 局面 1μs 未満)。
- 対称パターンは学習時に重み共有して過学習を抑制。

**重みデータサイズ**: 約 20〜30 万重み/ステージ × 15 ステージ × int16 ≒ **6〜10MB**、Brotli 配信で 2〜4MB。初回 DL 後は Cache Storage に永続化(後述 §8)。

**学習パイプライン(リポジトリ内 `/train`、オフライン実行)**:

1. 教師データ: WTHOR データベース(仏連盟公開の対局 40 万局超)+ 自エンジンの自己対戦(ランダム序盤 → 完全読み終局)。
2. 各局面に「最終石差」(終盤 30 手は完全読み値)をラベル付け。
3. ステージごとに L2 正則化付き線形回帰(共役勾配)で重み推定 → int16 量子化。
4. MPC のカットペア係数も同データから回帰。
5. 反復: 学習済みエンジンで自己対戦を再生成 → 再学習(2〜3 周で収束)。

> **近道の選択肢**: Edax(GPL-3.0)を WASM 化して重みごと利用する手もある。開発期間を大幅短縮できるが、アプリ全体を GPL 互換で公開する義務が生じる。自作(上記パイプライン)ならライセンス自由。本設計は自作を主線、Edax-WASM を開発初期の暫定エンジン兼ベンチ相手として使う二段構えを推奨。

### 2.5 性能目標(ベンチマーク定義)

- 探索速度: **単スレッド 8〜15M NPS / 4 スレッド 25〜40M NPS**(終盤)。
- **FFO endgame test #40–59** を標準ベンチとしてリポジトリに同梱(全問正解+所要時間を CI で計測)。
- 実用目標:

| 用途 | 目標 |
|---|---|
| 中盤練習・定石の即時評価 | depth 16〜18 を 0.3 秒以内 |
| 棋譜解析(1 局 60 手) | depth 18 + 空き 22 完全読みで 30〜60 秒 |
| 詰めオセロ判定(空き ≤ 16) | 0.1 秒以内 |
| 空き 24 完全読み | 10〜30 秒(スレッド数依存) |

- 対 Edax 同深度でのレーティング測定スクリプトを用意(自動対戦 1000 局)。

## 3. 定石練習モード(オセロクエスト式)

### 3.1 データ構造

```
JosekiDB {
  nodes: Map<normalizedHash64, {
    bookMoves:  [{ move, weight, eval }],   // 定石内の手(weight = 出現頻度)
    nonBookEval: number,                     // この局面の基準評価
    names: [josekiId],                       // 兎/虎/牛/ローズ… 複数所属可
    isLeaf: boolean                          // 定石終端
  }>,
  lines: [{ id, name, kana, moveSeq, depth, popularity }]  // 表示・選択用
}
```

- **局面ハッシュをキーにした DAG**。手順違いの合流(転換)を自然に扱える。
- 初手は f5 に対称正規化(縦取り/横取り/斜め取りは f5 系に写像し、表示時に逆変換)。
- 収録: 主要定石 100 種前後(兎・虎・牛・鼠系とその派生)、深さ 10〜25 手。ソースはオセロ連盟系の定石名鑑 + WTHOR 高段者局の頻出手順から抽出し、各ノードの eval は自エンジン depth 20 で付与してビルド時に検証(定石なのに評価が大きく落ちる枝を警告)。
- 配布形式: ビルドスクリプトでバイナリ化(`joseki.bin`、数百 KB)。

### 3.2 ゲームルール(仕様)

1. プレイヤーが手番の色を選択(黒/白/ランダム)。
2. **相手は現局面の `bookMoves` から weight に比例したランダム抽選**(=どの変化に進むか毎回変わる)。
3. プレイヤーの着手判定:
   - `bookMoves` に含まれる → 続行(コンボ加算)。
   - 含まれないが `eval` が定石内最善と同等(石差ロス < 1.0)→ **「定石外・可」**: 設定により (a) セーフ扱いで続行(その先はエンジン最善を定石扱い)or (b) ゲームオーバー。既定は (b)(「定石を覚える」目的に忠実)。
   - 評価が落ちる手 → **ゲームオーバー**。正解手と、悪手の理由(石差ロス、以後の最善進行 5 手)を表示。
4. `isLeaf` 到達 → **クリア**。到達した定石名と手順を表示し、そのラインの習熟度を更新。

### 3.3 習熟管理(SRS)

- 記録単位は「ラインの終端ノード」。クリアで習熟度 +、失敗で −。
- SM-2 簡易版で各ラインに次回出題日を付与し、「今日の復習」画面に失敗しやすい変化を優先出題。
- 出題モードの相手抽選を **「未習熟の枝を優先するバイアス付きランダム」** に切替可能(既定 ON。純ランダムより学習効率が高い)。
- ツリービューア: 定石 DAG を横スクロールのツリー UI で閲覧、ノードごとに習熟度を色表示。任意ノードから練習開始可。

## 4. 中盤練習モード

### 4.1 ルール

- 開始局面: (a) 定石終端からランダム、(b) 手数 15〜30 のランダム実戦局面(WTHOR 由来、互角 ±3 石差以内のみ収録)、(c) **自分の棋譜解析で悪手を打った局面**(§6 と連携。最重要の出題源)。
- プレイヤーは毎手着手 → エンジンが即時評価(depth 16、0.3 秒)。
- 判定モード(選択式):
  - **厳格**: 最善手のみ正解
  - **標準**(既定): 石差ロス ≤ 1.0 なら正解
  - **逆転禁止**: 評価の符号(優勢/劣勢)が入れ替わらなければ続行 ← 要望の「評価値が逆転しないように」
- 相手(エンジン)の強さ: 最善 / 上位 3 手ランダム / 実戦模倣(WTHOR 頻度分布)。
- 終了: 空き 24 で完全読みに切替え、勝勢確定(+2 以上維持)でクリア。逆転 or ロス超過で失敗 → 失敗局面が自動で出題プールに追加。

### 4.2 UI

- 評価バー(石差スケール、−16〜+16 でクリップ)を盤の横に常時表示するか選択可(既定 OFF: バーを見て打つと練習にならないため、失敗時のみ表示)。
- ミス時: 正解手ハイライト + 自分の手との比較 PV を盤上に矢印表示 + 「ここからやり直す」ボタン。

## 5. 詰めオセロ

### 5.1 問題形式

- 「黒番、最善で +N」「この局面、勝てるか?」の 2 形式。空きマス 6〜20。
- 正解判定は**完全読み**。最善結果を維持する手のみ正解(複数正解あり得る)。
- 手筋タグ: 偶数理論 / 手止まり / 連打 / 隅の犠牲 / 確定石、など。

### 5.2 問題生成パイプライン(ビルド時)

1. WTHOR + 自己対戦の終局 N 手前局面を大量サンプル。
2. 完全読みで全合法手の結果を算出。
3. 選別条件: **勝ち筋が 1〜2 手のみ**(唯一解性)/ 自然な石形 / 第 2 候補との石差差 ≥ 4(明確さ)。
4. 難易度スコア = f(空き数, 勝ち手の探索順位, 罠手の魅力度〔浅い評価では良く見える度合い〕)。
5. 難易度別 5 段階 × 各 200 問を `puzzles/*.bin` に格納。デイリー問題(日付シードで決定)も同 DB から。

### 5.3 プレイ仕様

- 着手 → 即時完全読みで最善維持か判定。相手応手は「最も粘る手」(プレイヤーの得を最小化)。
- 失敗時: 全合法手の結果一覧(+6 / +2 / −4 …)を盤上オーバーレイ表示。
- 成績: 正答率・平均時間・タグ別弱点を記録し、弱点タグを優先出題。

## 6. 棋譜解析(評価グラフ・悪手分析)

### 6.1 入力

- 標準トランスクリプト(`f5d6c3d3c4...`、大小文字・区切り混在を許容)
- WTHOR `.wtb`(ファイルドロップで複数局一括インポート)
- オセロクエストのコピー棋譜 / 手動並べ(盤で再現 → 解析)

### 6.2 解析パイプライン

1. 全 60 局面を Worker キューに投入。**終局側から解析**(空き ≤ 22 は完全読み、それ以前は depth 18)— 終盤の正確な値がグラフの信頼性を決めるため。
2. 各手のロス = eval(着手前最善) − eval(実際の手)。手番視点の石差換算。
3. 分類(閾値は設定可): ロス < 1: 最善/準最善 ◎ / 1–3: 緩手 ?! / 3–6: 疑問手 ? / ≥ 6: **悪手 ??** / 勝敗が入れ替わった手: **逆転悪手**(赤強調)
4. 解析結果は局面ハッシュで IndexedDB にキャッシュ(同一局面の再解析ゼロ)。

### 6.3 評価グラフ UI

- 横軸手数・縦軸石差(±16 クリップ)、黒優勢を上。完全読み区間は塗りを変えて「ここから確定値」を明示。
- 悪手マーカー(?!/?/??)をグラフ上とムーブリスト両方に表示。タップでその局面へジャンプ。

### 6.4 悪手分析(「悪手以降の展開を調べる」)

悪手をタップすると分析パネルを開く:

1. **比較 PV**: 「実際の手 → 相手最善の進行」と「最善手 → 進行」を 8 手ずつ、盤上矢印 + 並走リストで表示。ステップ実行・分岐点での差分ハイライト。
2. **フリー分岐探索**: 悪手局面から盤を直接操作して任意の変化を試せる。全ての試し手に即時評価が付き、**分岐ツリー**として右ペインに積み上がる(戻る・別分岐・本譜復帰が自由)。
3. **「なぜ悪いか」ヒューリスティック表示**: 着手可能数差の推移、確定石数の変化、偶数理論違反(終盤)、X 打ち・C 打ちの検出を機械的に注記。
4. **練習送り**: 悪手局面をワンタップで中盤練習(§4)/ 詰めオセロ(終盤なら)の出題プールへ登録。← 解析と練習をつなぐ本アプリの中核ループ。

## 7. 対局モード(基盤機能)

練習の前提として通常対局も実装(工数は小さい)。

- 強さ: レベル 1〜20(depth とランダム性で調整)+ 最強(時間制)。
- ヒント表示、待った、解析への直行、棋譜保存(自動で IndexedDB ライブラリへ)。

## 8. PWA / GitHub Pages 配信設計

### 8.1 マルチスレッドの成立(重要な落とし穴)

WASM スレッド(SharedArrayBuffer)には COOP/COEP ヘッダが必須だが、**GitHub Pages はカスタムヘッダを設定できない**。→ **coi-serviceworker**(Service Worker が全レスポンスに `Cross-Origin-Embedder-Policy: require-corp` / `Cross-Origin-Opener-Policy: same-origin` を注入する公知の手法)を導入。初回のみ SW 登録 → 自動リロードで cross-origin isolated になる。

- フォールバック: SAB 不可環境(古い iOS Safari 等)では単スレッドビルドに自動切替(機能は同一、解析が遅くなるだけ)。`crossOriginIsolated` で実行時判定。

### 8.2 キャッシュ戦略(Service Worker)

| 資産 | 戦略 |
|---|---|
| HTML/JS/CSS/WASM | precache + バージョン付きキャッシュ名で原子的更新(cache-first) |
| 評価重み(2〜4MB br)・定石・問題 DB | 初回に streaming fetch + 進捗バー → Cache Storage 永続化。以後ネットワーク不要 |
| 更新 | `version.json` を stale-while-revalidate で確認 → 「更新があります」トースト → ユーザー操作で SW skipWaiting |

- `navigator.storage.persist()` を要求して重みキャッシュの追い出しを防止。
- 完全オフライン動作: 全モードがオフラインで機能(通信は棋譜 URL 共有時のみ)。

### 8.3 PWA その他

- manifest(standalone、縦横対応)、アプリアイコン、iOS 用メタタグ。
- 共有: 棋譜を URL ハッシュにエンコード(`#g=f5d6c3...`)して共有可能。開くと解析画面へ。

## 9. データ設計(IndexedDB)

```
stores:
  games      { id, date, transcript, tags, result, analysisId? }
  analyses   { id, gameId, perMove: [{move, eval, loss, class}], engineVer }
  evalCache  { posHash → { depth, eval, pv } }            // 局面単位キャッシュ
  josekiSRS  { lineId → { ease, interval, due, streak, fails } }
  drillPool  { id, source: 'blunder'|'manual', board, bestEval, tags }
  puzzleStat { puzzleId → { solved, time, fails } }
  settings   { key → value }
```

- エンジン更新時は `engineVer` 不一致の解析に「再解析」バッジ。
- エクスポート/インポート(JSON 1 ファイル)で機種変更に対応。

## 12. リスクと対策

- **重み学習が想定精度に届かない** → 暫定で Edax-WASM をエンジン差し替え可能な設計(Worker プロトコルを実装境界にする)。GPL 化を許容するなら恒久採用も可。
- **iOS Safari のメモリ制限**(WASM メモリ + TT) → TT サイズを端末メモリで自動調整(32/64/128MB)。
- **初回 DL サイズ**(重み数 MB) → 遅延ロード: アプリ本体は先に起動し、重み DL 中は浅い簡易評価で動作、完了後シームレス切替。
- **定石データの正確性** → ビルド時にエンジン検証(定石手の評価落ちを警告)+ 出典を YAML にコメント記録。
