# T153: Egaroucid public training data — acquisition, trainer support, and same-scale quality probe

## Summary

Egaroucid's publicly distributed training data (25,514,097 board positions with lv.17-derived
scores) was downloaded, verified, and fed into a new `--simple-corpus` input mode added to the
distillation trainer (`train/src/t090_distillation.rs`). Two same-scale probe trainings (v4
pattern set, teacher-only loss, seed 1) were run against known benchmarks from prior tasks:

| configuration | actual train samples | oracle regret (T096 60 positions) | comparable known point | delta |
|---|---:|---:|---:|---:|
| Egaroucid subset @900k | 899,442 | **1.8667** | v4 x distilled 1M (T127d) = 1.9000 | **-0.033** (slightly better) |
| Egaroucid subset @180k | 179,972 | **2.0667** | v4 x distilled 180k/K=1 (T126) = 2.7667 | **-0.700** (clearly better) |
| (reference) v2 x WTHOR full | - | 1.5667 | - | - |
| (reference) v4 x WTHOR full (T124) | - | 1.1111 | - | - |

Both M2 guards passed (`v2` oracle mean regret reproduced exactly as
`1.5666666666666667` in both sessions). At matched train-sample scale, the Egaroucid data
performs **at least as well as, and at 180k noticeably better than**, the existing
`corpus_expanded1m.jsonl`-derived distillation corpus used in T126/T127d. Neither probe is
statistically distinguishable from v2xWTHOR at n=60 positions (both 95% CIs straddle zero), so
these are point-estimate signals, not proven wins — but they are a positive, mildly encouraging
result for the "collect more/better teacher data" direction from the 2026-07-20 user instruction.
**Whether to proceed to the full 25.5M-record training and a game-play gate is a separate
decision, out of scope for this task** (per `tasks/design/public-training-data-2026-07-20.md`
stage 2).

### Format

Each line is `<64-char board a1..h8> <score>`, e.g. `-XO-OOXOOXX-...-OOOX-O- 4`. `X` = the side
to move's own stones, `O` = the opponent's, `-` = empty. The trailing integer is "the predicted
final disc-difference from the side-to-move's perspective" (README_JA.md).

**Important label-provenance caveat** (recorded for transparency, not acted on specially by the
trainer): positions with **15 or fewer stones** (opening moves 1-11) were generated by exhaustive
enumeration of all opening lines plus an Egaroucid console 7.4.0 level-17 negamax search — a
genuine "teacher search value" label, structurally similar to the existing distillation corpus.
Positions with **16 or more stones** (the large majority — essentially all of the 14-63 stone
buckets, i.e. ~24.6M of the 25.5M records) were instead generated by Egaroucid console 7.5.1
level-17 **self-play** with randomized moves in $7 \le N \le 59$, and the recorded score is the
**final game result of that self-play game**, attributed to the position immediately after the
random-move segment — i.e. an outcome-style label, not a fresh search value computed exactly at
that position. Per this task's instructions, all records are trained identically with
teacher-only Huber loss regardless of this distinction; the distinction is reported as context for
interpreting the quality probe results, since it means most of this corpus is philosophically
closer to WTHOR's "outcome" label than to the distillation corpus's "engine search" label.

## Trainer change (requirement 2)

`train/src/t090_distillation.rs` gained a `--simple-corpus <file|dir>` input mode. Design
constraints and how they were met:

- **Existing corpus path is untouched.** `pub fn run()` now starts with
  `if let Some(value) = arg("--simple-corpus") { return run_simple(PathBuf::from(value)); }` —
  a single early-return branch. Without the flag this line is a no-op and every line after it
  (the pre-existing `load_corpus`/`DistillRecord`/`run_one`/`run_all`/`--mixes`/
  `--outcome-matched-only`/etc. path) is byte-for-byte the same code that ran before this task.
  `cargo test -p train` confirms this: all 56 pre-existing tests pass unmodified, alongside 18
  new tests for the simple-corpus path (74 total, 0 failed).
- **New, fully independent record type.** `SimpleRecord { key, board, teacher_value }` has no
  relationship to `DistillRecord` (which requires non-empty `children` for its ranking/agreement
  metrics — a requirement `SimpleRecord` doesn't need to satisfy, since Egaroucid's data has no
  candidate-move information). `parse_simple_record` reuses the existing `parse_board` (X/O/-)
  validator and always uses `mover = Side::Black`; this is safe because `X`/`O` are already
  encoded from the side-to-move's perspective and `engine::patterns::pattern_state_index`
  normalizes "own vs. opponent" via the `mover` parameter regardless of which literal color it is
  given (verified against `engine/src/patterns.rs`'s own unit tests before relying on it).
- **Streaming + deterministic downsampling.** `load_simple_corpus` reads the file(s) as a single
  line stream (`BufRead::lines`, no `fs::read_to_string` of the full corpus) and applies
  Algorithm R (reservoir sampling) when `--simple-max-records` is given, producing exactly
  `min(total_lines, k)` records with a seeded xorshift64 PRNG (deterministic for a fixed input
  file set and seed). Every raw line (kept or not) is folded into an FNV-1a content hash used as
  the run's resume identity, so identical input files always produce the same hash regardless of
  the cap. Only accepted lines pay the cost of full board parsing + D4 canonicalization, so a
  25.5M-line scan with a 2M-record cap still completes in low single-digit seconds to under a
  minute (see Timing below) rather than requiring 25.5M canonicalizations.
- **Subset selection reuses the existing algorithm.** `select_simple_subset` is a direct port of
  `select_train_subset`'s nested stratified-by-phase subsampling (same floor-based cutpoints, same
  per-phase seeding via `subset_seed_for_phase`) onto the independent `SimpleRecord` type — no
  shared mutable state with the `DistillRecord` version, so it could not have altered its
  behavior.
- **Checkpoint/resume reuses existing, unmodified infrastructure.** `simple_run_one` calls
  `latest_checkpoint`, `ensure_metrics_header`, `truncate_metrics_after`, and `atomic_write`
  exactly as `run_one` does, with zero changes to those functions. `metrics.tsv` keeps the
  existing 7-column header (`METRICS_HEADER`); the unused ranking column is always written as
  `0.0` since simple-corpus records have no candidate-move ranking signal.
- Mix is always `teacher-only` in this mode (the corpus has no ranking or WTHOR-outcome signal to
  mix in); `--mixes` is not read in the simple-corpus branch.

## Same-scale quality probe (requirement 3)

Both runs draw from the **same deterministic 2,000,000-record reservoir pool**
(`--simple-max-records 2000000`, `simple_corpus_hash=8e14d3751185a95c` in both manifests,
default `--subset-seed 42`), so the 180k run's train split is a nested subset of the 900k run's
train split, matching the existing `select_train_subset` nesting convention. `--jobs 1`,
`--pattern-set v4`, `--seeds 1`, `--max-epochs 60` (patience-based early stop, as in prior tasks).

### Comparison to known benchmarks (task-specified, not recomputed)

| point | mean regret |
|---|---:|
| v4 x distilled 1M (T127d, `corpus_expanded1m.jsonl`, train=899,467) | 1.9000 |
| v4 x distilled 180k / K=1 (T126) | 2.7667 |
| v2 x WTHOR full | 1.5667 |
| v4 x WTHOR full (T124) | 1.1111 |
| **Egaroucid subset @900k (this task, train=899,442)** | **1.8667** |
| **Egaroucid subset @180k (this task, train=179,972)** | **2.0667** |

At matched scale, Egaroucid's data is a **slightly better** teacher than the existing
distillation corpus at ~900k (1.8667 vs 1.9000, a 0.0333-disc improvement) and a **clearly
better** teacher at 180k (2.0667 vs 2.7667, a 0.7-disc improvement). Both results remain worse
than v4xWTHOR full (1.1111) and are not statistically distinguishable from v2xWTHOR at n=60
(both CIs straddle zero), so this is encouraging but not conclusive evidence at this sample size.
The 180k improvement is the more suggestive of the two data points: the same 60 positions, same
pattern set, same loss, same nominal scale, and the only variable that changed is the data source
(Egaroucid self-play/exhaustive-search-derived vs. the existing engine-search teacher corpus).

## Background-run interruption note (long-running-task rule compliance)

Both training processes were launched with `nohup ... &` per the CLAUDE.md long-running-task
convention (per-epoch checkpoint/resume already exists via the reused `run_one`-style
infrastructure). An orchestrator check at one point found no `result.tsv`, no matching process,
and a log file that appeared to have stopped advancing — consistent with the known
Bash-tool-call-boundary interruption pattern seen in T127d's `resume-check`. Per instruction,
the 900k run's exact command was re-invoked in the foreground; it printed
`resume simple-corpus seed=1 epoch=46` and then exited immediately, because the resumed
checkpoint's saved `stale` counter had already reached the patience limit (5) — the training
`while` loop's condition was already false, so this resume performed only the idempotent
finalize step (reload `best.bin`, rewrite `final.bin`/`complete.txt`/`result.tsv`), not additional
training. The 180k run's original background process was independently found to have completed
on its own by the time this was checked (its `result.tsv` already existed). Both checkpoint
directories' `metrics.tsv` files were inspected for duplicate or out-of-order epoch numbers
(`awk` + `uniq -d`) and found clean (strictly increasing epoch numbers, no gaps, no duplicates),
confirming no concurrent-write corruption occurred despite the double-invocation on the 900k
directory.
