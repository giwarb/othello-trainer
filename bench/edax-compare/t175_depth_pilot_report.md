# T175: 深さベース探索+MPC ONパイロット報告(vs Edax lv12/lv10)

**本レポートは実測の提示までであり、採否判定は行わない。** ユーザー発案(2026-07-21夕)「ノード予算をやめて深さベースにすれば、Edaxと公平に戦える」の成立性を実測で確認するパイロット。本採用の設計(時間管理・深さ選択・UX)は別途Codex諮問(7/26予定)で行う。

## 0. 背景

T172の結論: MPCはノード予算だと深さに変換できないが、枝刈り自体は深さ12でノード比0.18まで強化済み。**深さベース(ノード予算を実質撤廃)+MPC ONの組み合わせ**が有望と判断し、本パイロットで実測した。

## 1. 実装(最小限の配線)

既存資産(`engine/src/search.rs`の`SearchPolicy{enable_mpc,...}`・公開API`search_with_eval_with_policy`・`engine/Cargo.toml`の`mpc_enabled` feature・T172再校正済みの`engine/src/mpc.rs`のCALIBRATIONS)により、新規の探索ロジック実装は不要で、CLI配線のみで足りた。

- `engine/src/bin/eval_cli.rs`(`cmd_best`): `--enable-mpc`フラグを追加。指定時のみ`SearchPolicy{enable_history:true,enable_aspiration:true,enable_mpc:true}`を組み立て、既存公開API`search_with_eval_with_policy`(`calibrate_mpc.rs`の`gate`サブコマンドで実績あり)を呼ぶ。`mpc_enabled` featureビルドでなければエラー終了(意図しない「OFFのまま計測」を防止)。JSON出力への`mpcStats`追加も`--enable-mpc`指定時のみ。
- `bench/edax-compare/vs_edax.py`: `--engine-enable-mpc`(既定False)を追加。指定時のみ`ensure_engine_built(enable_mpc=True)`が`cargo build --release -p engine --bin eval_cli --features mpc_enabled`でビルドし、`eval_cli best --enable-mpc`を呼ぶ。`settings.engine_enable_mpc`としてrun_key/metaに記録。

**既存経路(ノード予算・MPC OFF)の完全不変性の実証**: `git stash`で変更前のコードに戻し`cargo build --release`→2局面(初期局面、および中盤局面1つ)で`eval_cli best`を実行→出力を保存。`git stash pop`で復元・再ビルド・同一局面で再実行。この既存コードベース自身が非決定的と定義している`elapsedMs`/`nps`(`vs_edax.py`の`calibration_identity()`参照)を除く全フィールドが、フラグ未指定・`mpc_enabled` feature有無いずれの組み合わせでも完全一致した(スモークSHA一致)。加えて`cargo test -p engine --release`は**240 passed / 0 failed / 2 ignored**、`vs_edax.py --self-test-checkpoint`もPASSED。

コミット: `2e9bb88`(`engine/src/bin/eval_cli.rs`+`bench/edax-compare/vs_edax.py`)。

## 9. 深さベース化の示唆(事実ベース、判定はしない)

- **強さ**: 深さベース+MPC ONへの切り替えで、lv12・lv10いずれに対しても平均石差が約3.25〜3.27石改善した(lv12: -6.07→-2.82、lv10: -2.22→+1.05)。開幕単位のbootstrap CIはP1(vs lv12)で完全に0より上、P2(vs lv10)もわずかに0より上(下限+0.10)であり、方向として一貫した改善が観測された。局単位ではいずれも符号検定が有意(p<0.05)。ただし開幕単位の符号検定はいずれも非有意(p=0.136)であり、**この改善が真に有意で頑健かはより大きな標本(60局×2は限られたpaired検出力)でのみ確定できる**。
- **時間実行可能性**: 深さ12の探索は、100,000,000ノード上限・15,000ms時間予算のもとで**ほぼ常に(2987着手判断中2986回、99.97%)15秒以内に完了した**(wall保険発動1回のみ)。1局あたりの実測所要時間は平均約50秒(P1・P2ともほぼ同水準)で、60局が現実的な時間(約50分/60局)で完走した。「同じ時間でlv12と互角に戦えそうか」という問いに対しては、**現状の15秒/手という予算下では、-2.82石(ほぼ互角に近いがまだ劣勢)まで改善した**という事実が得られた。真の「公平な持ち時間比較」(Edax側の実消費時間との対称性)は本パイロットのスコープ外(Edax側は`-l 12`/`-l 10`の固定レベルで動いており、Edax自身の消費時間は計測していない)。
- **ノード分布の分散**: budgeted相番のノード消費はp50が68万前後・p90が6000万強と2桁近い分散があり、深さ12を「常に一定コストで」保証するものではない(局面依存性が大きい)。この分散の大きさは、本採用時の時間管理設計(7/26 Codex諮問)で考慮すべき材料になりうる。
