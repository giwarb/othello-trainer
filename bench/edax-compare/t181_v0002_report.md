## 2. 変換仕様と実装(`train/src/bin/egaroucid_v0002_convert.rs`)

1. **パース+再生**: f5d6の2文字着手記法をマスインデックス(0..63)へ変換し、`train::train_data::samples_from_game`(既存関数、WTHOR変換で実績あり)で再生・パス自動判定・最終石差ラベル付けを行う。不正棋譜0件・非合法手順0件(52,000,000対局全件スキャン)。
2. **ランダム序盤の除外**: フォルダ名の数値Nをそのまま先頭除外手数として使い、`samples[0..N)`を除外、`samples[N..]`(エンジン同士が実際に着手した局面)のみを採用候補とする。
3. **層化サンプリング**(目標25,000,000局面、`--target-count`でパラメータ化): 現行lv17コーパスの空きマス数分布を目標分布とし、バケットごとに決定的ハッシュ(ファイル相対パス・行番号・ステップ番号のみに依存する純粋関数)で採用判定する。採用が1件もない対局は再生自体をスキップする最適化により、52,000,000対局のスキャンが約700秒(約11.7分)で完了した。
4. **決定性・checkpoint/resume**: ファイル単位(1ファイル=1万局)でcheckpoint保存、resume時は設定一致を検証。実機テストで2件の不具合を発見・修正済み(詳細は`tasks/T181-v0002-outcome-label-experiment.md`作業ログ10節):
   - `BufWriter`の自動flushにより出力ファイル実バイト長がcheckpoint記録より先行する不整合 → `output_bytes_committed`追加+resume時`set_len`切り詰めで修正。
   - Windowsの`Path::join`によるパス区切り文字混在で採用判定ハッシュキーが不安定になる不具合 → `normalize_path_for_key`(`\`→`/`正規化)で修正。
   - 修正後、中断(`Stop-Process -Force`)→再開の実機テストで非中断実行と最終SHA-256完全一致を確認。

## 5. 結論・考察

1. **ユーザー仮説(強いAI同士の対戦結果由来の最終石差ラベルが有望)は、本実験の範囲では支持されなかった。** 静的frozen MAE・対Edax実戦成績のいずれの指標でも、v0002由来ラベル(E1単独・E2混合とも)は現行lv17探索値ラベル(v6)に劣った。
2. **lv17との混合(E2)はv0002単独(E1)より一貫して良い**(frozen MAE・対局成績とも)。これはE2の学習データの半分がlv17由来であることの効果であり、v0002ラベル自体が有益な追加情報を提供しているかは本実験だけでは切り分けられない(混合比の最適化はスコープ外、T181タスク仕様どおり)。
3. **最終石差ラベルの限界**: v0002は探索値(evaluationのターゲットとなる中間局面の評価値)ではなく対局の最終結果のみをラベルとするため、序盤・中盤局面に「その手が実際に良かったか」の情報が薄まりやすい(対局全体の勝敗に引きずられるノイズが大きい)。lv17コーパスの探索値ラベルの方が局面ごとの評価情報の密度が高く、これが両ラベル方式の性能差の一因と考えられる(本実験では直接検証していない仮説)。
4. 本番配線・再探索ラベル付け・混合比チューニングはいずれもスコープ外(T181タスク仕様どおり)。
