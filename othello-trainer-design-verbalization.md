# 拡張設計: 中盤の「言語化」支援 — 評価値を根拠に翻訳する

本書は `othello-trainer-design.md` の拡張。目的は「この手がなぜ良い/悪いか」を人間の概念語彙で説明し、かつ**ユーザー自身の言語化を訓練する**こと。

---

## 0. 設計思想

評価値は結論であって理由ではない。理由を作るには 3 つの独立した情報源を組み合わせる:

| 層 | 情報源 | 答えられる問い |
|---|---|---|
| A. 特徴量層 | 局面から直接計算する解釈可能な特徴(手数・開放度・壁…) | 「この手は盤面を**どう**変えたか」 |
| B. 評価内訳層 | パターン線形評価の寄与分解 | 「エンジンは**どこを**マイナスと見たか」 |
| C. 反証層 | 探索による具体的な咎めライン | 「その損は**いつ・どう**現実化するか」 |

A だけだと「開放度が高い=悪い」のような常に正しいとは限らない一般論に落ち、C だけだと「読み筋を見ろ」で終わる。**A×B×C を対比形式(なぜ最善手ではなくこの手が悪いのか)で提示**するのが本設計の核。

エンジンの評価関数が線形パターンモデル(本設計 §2.4)であることが、ここで効いてくる。ニューラル評価と違い、**線形モデルは寄与を厳密に分解できる**。これは説明機能のために評価関数アーキテクチャを選んだと言えるほどの利点。

---

## 1. 特徴量層(A): 解釈可能特徴の抽出

Worker に `explain` コマンドを追加し、任意局面・任意手について以下を計算(全てビットボード演算で μs オーダー):

### 1.1 特徴カタログ

| 特徴 | 定義 | 対応する用語 |
|---|---|---|
| 着手可能数差 | 自分−相手の合法手数(着手前後) | 手数、手得/手損 |
| 潜在手数 | 相手石に隣接する空きマス数の差 | 潜在着手可能数 |
| 開放度 | その手で返す石に隣接する空きマス総数 | 開放度理論、中割り(≤2 なら中割り判定) |
| フロンティア石数 | 空きに接する自石数の差 | 壁 |
| 新規に生む相手の手 / 消える自分の手 | 着手前後の合法手集合の差分(マス単位で特定) | 「相手に◯◯を与える手」 |
| 確定石差 | 辺・隅からの安定石計算 | 確定石 |
| 辺の形 | 各辺を 3^8 パターンで分類(ウィング/山/ブロック/一方空き…) | 辺の攻防 |
| X・C 打ちリスク | X/C 着手時、対応する隅の危険度(隅を取られた場合の確定石見積り) | X 打ち・C 打ち |
| 地域偶数 | 空きマスの連結成分ごとの奇偶と、自分が最後に打てるか | 偶数理論・手止まり |
| 余裕手 | 「打っても形が悪化しない手」の数(浅い評価でロス < 0.5 の手を数える) | 余裕手 |
| 種石 | 相手辺への着手を成立させている自石の特定 | 種石 |
| ライン | 主対角線・長ラインの占有状況 | 白ライン/黒ライン、通し |

### 1.2 対比テンプレート生成(オフライン・非 LLM)

最善手と対象手の特徴ベクトルの**差分が大きい軸だけ**を選び(上位 2〜3 軸)、テンプレートで文章化:

> **あなたの g6(−4.2)** は開放度 6 の全返しで、相手の手数を +3 増やします。
> **最善の c6(+1.8)** は開放度 1 の中割りで、自分の手数を温存しつつ相手の e7 を消します。
> 主な差: 手数(−2.1)、壁(−1.4)

要点: 全特徴を羅列しない。**差が説明力を持つ軸だけ**を語る。テンプレートは用語ごとに 3〜4 パターン用意し単調さを回避。全文に用語集リンク(§7)。

## 2. 評価内訳層(B): 線形評価の寄与分解

### 2.1 仕組み

評価値 = Σ(46 パターンの重み) + 手数項 + パリティ項。よって 2 つの局面(自分の手の後 vs 最善手の後)の評価差は、**パターングループ別の寄与差に厳密に分解できる**。グループを人間概念に写像する:

| パターングループ | 表示名 |
|---|---|
| edge+2X, 横縦 2/3/4 列 | 辺と辺周辺の形 |
| corner 3×3, corner 2×5 | 隅まわりの安全度 |
| 対角ライン d4–d8 | 斜めライン |
| 着手可能数項 | 手数 |
| パリティ項 | 偶数 |

### 2.2 UI: 損の内訳ウォーターフォール

悪手タップ時、評価差 −5.5 の内訳を小さな横棒で表示:

```
b2 はなぜ −5.5 か
 隅まわりの安全度  ████████ −4.1
 辺の形           ██ −1.0
 手数             ▌−0.6
 斜めライン        ▏+0.2
```

### 2.3 技術上の注意(正直に設計に織り込む)

手の良し悪しは静的評価でなく**探索結果**で決まるため、着手直後の静的評価差の分解は近似にすぎない。対策: 分解は**両 PV の末端局面(depth 先の到達局面)間**で行い、「◯手先の形で比較すると…」と明示する。さらに PV 中間局面でも分解し、寄与が急変した手(=損が形になった瞬間)を検出して層 C(§3)に接続する。

## 3. 反証層(C): 咎めラインと「損の回収点」

悪手に対し、エンジンは既に相手最善応手の PV を持っている。これを**物語として構造化**する:

1. **回収点検出**: PV を進めながら §2 の内訳を各手で再計算し、寄与が閾値以上動いた手を「損が現実化した点」としてマーク(例: 7 手後の相手 a1 で「隅まわり −6.2 が確定」)。
2. **咎め手のモチーフ分類**(§4 の検出器を PV 上の相手の手に適用): 「この悪手は 5 手後の**ストナー**で咎められる」「壁を作った結果、9 手後に**手詰まり**(良い手が 0 になる)」。
3. **UI**: PV 自動再生+回収点で一時停止し 1 行解説。「悪手 → …… → **ここで隅が確定**」という因果の見せ方にする。評価グラフ(§6.3)にも回収点マーカーを追加。

これにより「−5 だから悪い」ではなく「**この手は 7 手後の a1 を許すから悪い**」と言えるようになる。

---

## 4. モチーフ検出器(タグ体系)

手に付与するタグ。ルールベース検出+評価で妥当性検証(検出しても評価が支持しない場合は付けない):

**良い手系**: 中割り / 引っ張り(相手の好手を消す) / ブロック / 種石作り / 手止まり確保 / 余裕手の温存 / 辺の先着 / 犠牲打(一時損で偶数や隅を確保) / 通し

**悪い手系**: 全返し(開放度過大) / 壁作り / 手損 / X 打ち(無根拠) / C 打ち(無根拠) / 種石供給(相手の辺打ちを成立させてしまう) / 偶数放棄 / 自滅(自分の手数を自分で消す)

**罠系(専用検出器)**: ストナー(成立条件: 辺の形+X 筋の種石を厳密判定) / 爆弾 / 隅絡みの一方通行

タグは以下すべてで共通利用: 悪手分析の注記、中盤練習のフィードバック、言語化トレーニング(§6)の選択肢、弱点統計(§8)。

---

## 5. 可視化レイヤー(言語の前段)

言語化の半分は「見えるようにする」こと。盤上オーバーレイをトグル群として提供:

- **開放度マップ**: 各合法手を開放度で色分け(緑=中割り、赤=全返し)
- **手数マップ**: この手で新たに生まれる相手の手 / 消える自分の手を矢印表示
- **フロンティア表示**: 壁になっている石を縁取り
- **確定石シェーディング**
- **偶数マップ**: 空き領域を奇数=橙 / 偶数=青で塗り分け
- **種石ハイライト**: 相手の辺打ちを支えている自石を点滅表示

中盤練習中もトグル可(既定 OFF、失敗後の振り返りでは自動 ON)。**オーバーレイは常に §1 の特徴量と同一の計算を使う**ため、見た目と説明文が食い違わない。

## 6. 言語化トレーニングモード(新モード)

説明を「読む」だけでは言語化は身につかない。**予想 → 理由の宣言 → 検証**のループを回す:

### 6.1 出題フロー

1. 局面提示(出典: 中盤練習プール・自分の悪手局面・名局)。評価値は**非表示**。
2. ユーザーが手を選ぶ。
3. **理由タグを 1〜3 個選択**(§4 のタグ+特徴語彙から。自由記述メモ欄も任意)。
4. 検証画面: 手の正誤 × 理由の正誤を **2×2 で別々に採点**。
   - 手○理由○: 完全正解
   - 手○理由×: 「**正解だが理由が違う**」— 最重要のフィードバック。まぐれ当たりを検出できるのはタグ宣言があるからこそ
   - 手×理由○: 「着眼は正しい。ではその観点で最善はどれ?」と再出題
   - 手×理由×: 概念レッスン(§7)へ誘導
5. 理由の採点基準: 選んだタグが §1 特徴差・§2 寄与分解の上位軸と一致するか。

### 6.2 二択比較ドリル

「c6 と g6、どちらが良い? 理由は?」形式。説明生成が対比ベース(§1.2)なので出題も対比が自然。差が 1 概念に集約される局面ペアを puzzlegen 拡張で自動選別(寄与分解で 1 グループが差の 70% 以上を占める局面)。概念別の教材が無限に作れる。

### 6.3 自由記述の扱い

自由記述メモは局面とともに保存し、復習時に「過去の自分の言語化」と並べて表示。上達の記録になる(採点はタグで行い、自由文は資産として蓄積)。

---

## 7. 用語集と概念レッスン

- 全タグ・全用語に 1 ページの用語集(定義 3 行+インタラクティブな最小例局面 2 つ+反例 1 つ)。説明文・タグ・オーバーレイのどこからでも 1 タップで到達。
- 概念レッスン: 「開放度」「偶数理論」等 12 テーマ。各レッスン = 説明 1 画面 + §6.2 の二択ドリル 10 問。**教材局面はユーザー自身の棋譜から優先的に採用**(自分の失敗例ほど刺さる教材はない)。

---

## 8. 概念別の弱点統計(SRS の拡張)

既存の定石 SRS に加え、**概念単位の習熟トラッキング**を追加:

```
conceptStat { tag → { attempts, correct, reasonCorrect, lastSeen, due } }
```

- 悪手分析・中盤練習・言語化トレーニングの全結果がタグ経由でここに集約。
- ダッシュボード: 「あなたの負けの 42% は**壁**絡み。開放度の理由正答率 55%」のようなレーダーチャート+出題バイアスへの自動反映(弱い概念の局面を優先出題)。
- 棋譜解析にも反映: 悪手一覧を概念別にグルーピング表示(「今日の 3 敗、全部種石供給です」が見える)。

---

## 9. 任意拡張: LLM 解説レイヤー(既定 OFF・オフライン不要)

テンプレート文で足りない「読み物としての解説」が欲しい場合の追加層:

- ユーザー自身の API キーを設定画面で登録(BYOK)。GitHub Pages からブラウザ直接呼び出し。
- **入力は生の局面ではなく、§1〜4 の構造化出力**(特徴差分・寄与分解・回収点・タグ・PV)。LLM の役割は事実生成ではなく**文章化のみ**に限定し、プロンプトで「与えられた分析事実以外を述べない」と拘束。→ ハルシネーションで盤面と矛盾する解説が出る事故を構造的に防ぐ。
- 出力例: 悪手 1 つにつき 3〜4 文の講評、または 1 局まとめの感想戦テキスト。
- 未設定ユーザーには影響ゼロ(テンプレート説明が常に既定)。

## 10. 実装追加分

### Worker プロトコル拡張

```jsonc
{ "cmd": "explain", "board": {...}, "move": "b2", "compareTo": "best",
  "want": ["features", "attribution", "refutation", "tags"] }
→ { "features": { "mobilityDiff": [-1, +2], "openness": [6, 1], ... },
    "attribution": [ { "group": "corner", "delta": -4.1 }, ... ],
    "refutation": { "pv": [...], "criticalPly": 7, "criticalMotif": "corner_capture" },
    "tags": { "played": ["full_flip", "wall"], "best": ["uchiwari", "pull"] } }
```

### リポジトリ追加

```
/engine/explain   特徴量抽出・寄与分解・回収点検出(コア評価と同一クレート内)
/app/overlay      可視化レイヤー
/app/glossary     用語集(YAML → 静的生成)
/puzzlegen        二択比較ドリル選別を追加(寄与集中度フィルタ)
```

### ロードマップへの挿入

- フェーズ 4(棋譜解析)に: 特徴量層 + 寄与分解 + 損の内訳 UI
- フェーズ 6 に: モチーフ検出器 + 可視化レイヤー
- 新フェーズ 6.5: 言語化トレーニングモード + 用語集 + 概念統計
- フェーズ 7 に: LLM レイヤー(任意)

## 11. この設計が成立する理由(まとめ)

1. **線形パターン評価だから寄与分解が厳密にできる**(エンジン設計と説明機能が噛み合っている)。
2. オセロの中盤理論(手数・開放度・壁・偶数・種石)は**計算可能な特徴と 1 対 1 対応**しており、チェスや将棋より機械的言語化に向いた競技である。
3. 説明を読ませるだけでなく、**理由タグの宣言 → 採点**で「正解だが理由が違う」を検出できる。これが「評価値を見るだけの学習」との最大の差別化点。
4. 全機能(説明文・オーバーレイ・タグ・統計)が同一の特徴量計算を共有するため、実装コストの割に一貫した体験になる。
