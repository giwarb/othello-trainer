# T056: 棋譜解析の評価値モデルを「累積ロス方式」に作り直す(最善手は評価値維持、悪手のみ悪化)

## 目的

ユーザー指摘: 棋譜解析モードの評価値の考え方が根本的に間違っている。現状は各手ごとに独立したヒューリスティック探索値をそのまま「評価」欄・グラフに表示しているため、定石を外れた直後の最善手(スクリーンショットの13手目、分類は◎=最善なのに表示は「+18」で「逆転」フラグまで付く)のように、**最善手を打っただけなのに評価値が突然大きく動いて見える**という不整合が起きている。

ユーザー要望を要約すると:
1. スケールは最終的な石数差(そのゲームが終わった時の実際の石数差)になること。
2. 最善手を打った場合、評価値は**変化しない**(「良い手を打ったから改善する」という発想自体が誤り)。
3. 悪手を打った場合のみ、そのロス(既存の「ロス」列の値)分だけ評価値が悪化する。
4. Edaxベースの解析ツールで一般的な考え方に忠実に、評価値の算出方法を作り直す。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

### 現状の実装(何が間違っているか)

- `app/src/analysis/analyzeGame.ts`: 棋譜解析のコア。各手について、その局面でエンジンに全合法手を評価させ(`search_all_moves`相当)、`bestDiscDiff`(その局面での最善手の評価値、手番視点)と`playedDiscDiff`(実際に打たれた手の評価値、手番視点)を比較し、`lossDiscs = max(0, bestDiscDiff - playedDiscDiff)`(ロス、既存の「ロス」列はこれをそのまま表示)を算出している。**この「ロス」計算自体は同一局面・同一探索呼び出しで最善手と実際の手を比較しているため、相対的に信頼できる**(ユーザーもロス列・悪手判定自体には文句を言っていない)。
- 一方、ムーブリストの「評価」列・評価グラフに表示している数値(`blackAdvantageBefore`/`blackAdvantageAfter`、黒視点に変換した生の探索値)は、**局面ごとに独立に計算されたヒューリスティック探索の生の値**である。中盤のヒューリスティック評価は探索深さ・局面の複雑さによって本質的にノイズが大きく(T041-T045のパターン評価導入でも解消しきれない、探索アルゴリズムの限界)、たとえ全ての手が客観的に最善であっても、局面が進むごとに表示される生の数値は大きく上下しうる。これが「最善手なのに評価値が跳ねて見える」問題の直接の原因。
- 「逆転」判定も同様に、この生の値の符号変化を見て判定しているため、実際には形勢が変わっていない(最善手が続いている)のに「逆転」と誤表示されることがある。

### 正しい考え方(累積ロス方式、数学的に妥当なモデル)

以下の漸化式で評価値を再構成する。黒視点の評価値を`E[i]`(i手目まで進めた局面の評価、i=0は初期局面)とする:

- `E[0]` = 初期局面の評価値(黒視点)。**慣例として0(互角)を初期値として採用してよい**(T046で定石区間を0固定にしている既存の設計と一貫性を持たせるため)。他の妥当な初期値(エンジンによる初期局面の実探索値)を採用してもよいが、その場合は理由を作業ログに記録すること。
- i手目(手番側を`mover`とする)を打った後: `loss[i]` = 既存の`lossDiscs`(0以上、その手が最善からどれだけ劣っていたか、mover視点)。
  - `mover`が黒なら: `E[i] = E[i-1] - loss[i]`
  - `mover`が白なら: `E[i] = E[i-1] + loss[i]`

この漸化式は最善手が続く限り(`loss[i] = 0`)`E[i] = E[i-1]`となり評価値は変化せず、悪手を打った瞬間だけそのロス分だけ評価値が該当側に不利な方向へ動く。数学的には、`E[i]`は「その局面で双方が最善を尽くした場合に得られる石差の見積もり」を表しており、最終局面(`i = 総手数`)まで一貫して適用すれば、終盤の完全読み区間で計算される正確なロスと組み合わさり、**最終的に実際の最終石差に収束する**(終盤完全読みのロス計算が正確であるため)。

「逆転」判定も、この`E[i]`の符号変化(`E[i-1]`と`E[i]`で黒有利/白有利が入れ替わったか)を基準に再定義すること。

## 要件

1. 最善手(ロス0)を打った手について、評価値表示が直前の値から変化しないこと。
2. 悪手を打った手について、評価値表示が直前の値からロス分だけ悪化すること(符号の向きが手番と整合していること)。
3. 一連の対局を最後まで解析した場合、累積評価値の最終値が実際の最終石差と一致すること。
4. 「逆転」判定が、累積評価値の符号変化と一致すること(最善手を打ち続けている間は「逆転」が発生しないこと)。
5. T038(定石内の手は評価値表示なし)・T046(定石区間はフラット表示)の既存の挙動を壊さないこと。
6. 既存のテストが壊れないこと(挙動変更に伴う期待値の更新は行ってよい)。

### 設計判断の記録(仕様上「妥当な選択なら理由を記録」とされていた点)

- **`E[0]`の初期値**: タスク仕様が推奨する`0`(慣例上の互角)をそのまま採用した(T046の「定石区間は0固定」との整合性を優先)。
- **「逆転」判定の符号比較方法**: `Math.sign(before) !== Math.sign(after)`という単純な比較(ゼロを含む3値のいずれかが変わったら逆転)を採用し、「符号が厳密に+から-(またはその逆)に転じた場合のみ」という、よりゼロを特別扱いする代替案は採らなかった。理由: (1)タスク仕様の文言が「符号変化(`E[i-1]`と`E[i]`で黒有利/白有利が入れ替わったか)」であり、旧実装の比較方法(`Math.sign`の不一致)をそのまま新しい値に適用する読み方が自然であること、(2)要件4が要求する「最善手を打ち続けている間は逆転が発生しないこと」は、ロス0が続く限り`before === after`となるため単純な比較でも満たされること、(3)既存テスト(「1手だけの棋譜を解析」)がこの単純な比較方法を前提にした`reversal: true`を既に期待しており、変更しない方が既存の期待に忠実であること。なお、この選択には「`E[0]=0`の局面で最初の一手が(たとえ僅かなロスでも)非0の値に動くと、それだけで逆転とみなされる」というやや直感に反しうる副作用があることを認識している(実機確認では、酉定石を外れた直後の1手目が大きな悪手だった場合に(逆転)表示が付くケースで確認した)。もしオーケストレーター/ユーザーがこの挙動を望まない場合は、`analyzeGame.ts`の`applyCumulativeEvaluation`内の`reversal`計算式を`(before > 0 && after < 0) || (before < 0 && after > 0)`(厳密に+/-が入れ替わった場合のみ)に差し替えるだけで対応できる(該当箇所にコメントで両案を残していないため、変更時はこの作業ログを参照)。
