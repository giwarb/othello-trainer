# 設計諮問: オセロエンジンをEdaxに近づけるための強化方針(T083想定)

## 背景

本リポジトリはRust製オセロエンジン(`engine/`、wasm-bindgenでWASM化しPreactアプリ`app/`から使用)+学習パイプライン(`train/`)。ユーザーの依頼は「AIをもっと賢くし、Edaxになるべく近づけたい」。

現状のエンジン構成:
- 探索: 反復深化 + NegaScout(PVS/NWS) + 置換表(2-tier, 64MB) + ETC + 終盤完全読み(パリティ着手順序付け)。アスピレーションウィンドウ無し。killer/historyヒューリスティック無し(着手順序はTT手→隅優先→相手モビリティ最小)。MPCは実装済みだが**既定オフ**(`mpc_enabled`フィーチャ。評価関数の深さ間分散σが350〜660centi-discと大きすぎ、マージンを保守的にせざるを得ずプローブコストが割に合わないため。T048)。
- 評価: WTHOR学習のパターン評価v2(`train/weights/pattern_v2.bin`)。22パターンインスタンス(行8・列8・対角線2・隅3x3ブロック4)を対称性で6クラスに重み共有、13ステージ(空きマス/5)、WTHOR 2015-2024の19,119対局・約114万局面でSGD回帰(L2=1e-5、20エポック)。出力は±6400centi-discにクランプ(T059)。
- 性能: 単スレッドNPS実測約285万(FFO #40-44)。設計目標8-15Mには未達。安定石カット・浅い空きマス専用の終盤関数は未実装。
- 並列化(WASM Threads)は**ユーザー判断で実装しない**(COOP/COEP再導入が過去に事故多発領域のため。2026-07-11確定)。

## 実測データ(T082: Edaxとの自動対戦60局、2026-07-13)

`bench/edax-compare/vs_edax.py`(対戦ハーネス)、`vs_edax_results.json`(60局の棋譜・評価値)、`vs_edax_report.md`(レポート)を参照。

- 条件: 自作=depth10 / exact-from-empties18 / time-ms1000 / pattern_v2、Edax=book off。開始局面10種×黒白持ち替え。
- 結果: **Edax level 1 に16勝4敗(+15.4石)、level 5 に0勝20敗(-29.1石)、level 10 に0勝20敗(-37.2石)**。
- フェーズ別ロス(負け局14局・345手をEdax level16で再評価): **中盤(21-40手目)が平均+2.22石/手・累計311石と突出**。序盤+0.31石/手、終盤(41手目〜)は±0.00(完全読み領域は正確)。
- 大ロス局面トップ10の自エンジン着手: b2, g7, c1, g8, b7, b7, g2, b7, d2, d1 — **X/Cマス(隅隣接)への自殺的な着手が多数**(最大ロス+28石)。
- 静的評価のEdax比較(T043/T044、38局面): MAE 5.06石、符号一致率75%(序盤に限ると37.5%と特に低い)。
- 注意(データの解釈): 自作側はtime-ms1000のカットがあるため、中盤でdepth10に届いていない手が含まれうる。「level10=同深度」は厳密には成立していない(探索速度の差も結果に混ざっている)。

## 制約

- 実行環境はブラウザWASM・シングルスレッド固定(Threads不採用は確定判断)。アプリの1手あたり時間予算は対局モードで数百ms〜1秒程度。
- 重みファイルはGitHub Pagesから配信(現行pattern_v2.binは2.7MB)。数十MB級への肥大は初回ロード体感を悪化させるため、増やすなら根拠が必要。
- 学習データはWTHOR(FFO公式配布、ローカルのみ・非コミット)。年数の追加取得は可能。
- 定石DB(独自調査の定石体系、`bookgen/`)はアプリの練習モード用に存在するが、CPU対局の着手には未接続。
- 実装は複数タスクに分割してSonnetワーカーが1タスクずつ実施する。1タスク=1セッションで完結するサイズ。

## 判断してほしい点

1. **「Edaxに近づける」ための投資対効果の高い順の強化ロードマップ**(3〜6項目程度)。特に以下の仮説・選択肢の採否と優先順位:
   - 評価関数の表現力強化(Edax系のパターン構成への刷新: edge+2X、corner5x2、d5-d8対角線等の追加や、ステージ数の細分化、学習データ増、リッジ/ロジスティック回帰等の学習法改善) — 中盤ロスとX打ち血迷いの根本対策になるか
   - 探索の高速化(ムーブオーダリング強化(浅い探索での並べ替え・killer/history)、アスピレーションウィンドウ、評価改善後のMPC再挑戦、終盤ソルバーの安定石カット・浅空き特化) — time予算内での実効深さを稼ぐ
   - CPU対局への定石ブック接続(序盤ロスの削減、ただし序盤ロスは中盤より小さい)
2. 各項目の**期待効果の見積もり方**(何で測るか: vs_edaxハーネスのレベル別勝率を標準ベンチにしてよいか、局数は20局で足りるか)。
3. **評価関数を刷新する場合の具体設計**: パターン構成(どのパターンをいくつ)、ステージ数、学習データ量の目安、学習アルゴリズム、重みファイルサイズの見積もり。既存の`patterns.rs`/`pattern_eval.rs`/`train/`の資産をどこまで流用できるか。
4. X/Cマスの自殺的着手への**短期的な対症療法**(例: 評価への隅安全性項の追加、探索深さの動的延長)を、評価刷新とは別に先行実施する価値があるか。
5. タスク分割案(各タスクの検証可能な完了条件つき)。
