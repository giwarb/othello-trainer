# T149 設計依頼: 定石ブックフェーズ2(Edax評価値付き自前ブック)

status: 起票済み・設計待ち(Codex usage limit中。7/23 20:32復帰後にcodex-design諮問、またはユーザー回答が揃えばオーケストレーター設計で前倒し可)

## 背景(2026-07-20 explorer調査の要約)

- 現行フェーズ1: bookgen/joseki-research.json(手作り35ライン)→ app/src/joseki/generate.ts+buildDb.ts でDAG化 → app/public/joseki.json(615ノード・112ライン)。**eval/nonBookEvalフィールドは型として存在するが常にnull(未使用)**。重みは均等割り当て(出現頻度データなし)。
- 消費経路: 定石練習(judgeMove.tsはライブrequestAnalyzeAllで判定、事前evalは不使用)/対局CPU即着手(pickBookMove.tsの重み比例抽選)/T138定石トレース+ブックcap(weightとnamesのみ使用)。
- **Edax評価値を付けても、消費経路を設計しない限り死んだデータになる**(調査の核心)。
- 設計書§3の当初構想: 主要定石100種前後・深さ10-25手・WTHOR高段者局の頻出手順から抽出・重み=出現頻度・ビルド時エンジン検証(評価落ち警告)。現行はこの一部のみ実装。
- Edaxバッチ基盤(vs_edax.edax_solve_batch)は任意局面集合に流用可能(決定性はn_tasks=1指定)。WTHOR実体はtrain/data/*.wtb(2000-2024、74,024局)に存在。

## 設計してほしいこと

1. 対象局面の拡張方法: 既存112ノードへの後付けに加え、WTHOR頻出進行からのライン抽出(頻度閾値・深さ上限・既存35ラインとの整合/命名)をどう設計するか。
2. Edax評価値の付与: level選定(序盤局面は完全読み不可、level16〜24の近似)、決定性(n_tasks=1)、生成時間見積もり。
3. 消費経路の設計(最重要): (a)ビルド時検証=定石手の評価落ち警告(設計書338行の趣旨) (b)CPU着手選択への評価値の使い方(下記ユーザー確認事項) (c)重み=WTHOR出現頻度の導入(RawJosekiLine/JosekiBookMoveのスキーマ拡張)。
4. データ形式: joseki.json拡張で足りるか(binバイナリ化はサイズ問題が出た場合のみ)。SRS(ライン単位IndexedDB)との後方互換。
5. タスク分割案(生成パイプライン/評価付与/app消費/検証)。

## ユーザー確認事項(2026-07-20質問済み、回答あれば反映)

- Q1: CPU着手のランダム性。現行=均等ランダム(オセロクエスト式)。評価値導入後は「明白な悪手(例: ロス2石以上)をブックから除外した上で、残りは頻度重みランダム」をデフォルト案とする(強さと定石多様性の両立)。完全に評価値最良のみに絞る案もあるが対局の多様性が失われる。
- Q2: カバレッジ拡大の規模。デフォルト案=WTHOR 74,024局から頻出進行を抽出し、深さ〜16手・出現100局以上のラインを追加(推定数百〜千ノード規模)。
- Q3: 1M教師コーパスのシャードファイル(約1.6GB、mergedと重複)削除の可否(ハウスキーピング、本設計とは独立)。

## 制約

- 生成中の重い並行作業制約はEdax呼び出し量次第(数千局面×level16なら数分〜数十分で軽い)。
- 既存ユーザーのSRS進捗(lineId=name)を壊さない。
- ブックcap(T138)の表示仕様は維持。
