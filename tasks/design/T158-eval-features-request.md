# T158 設計依頼: 評価関数への追加特徴(モビリティ・囲い度)の導入設計

## 背景

- ギャップ分析(tasks/design/engine-gap-analysis-2026-07-20.md)の残候補。Egaroucid系の標準構成は「パターン+着手可能数(モビリティ)+囲い度(自石が空きマスに接する度合い)」だが、当プロジェクトの本番評価(v4)はパターン38形状×61段の線形和のみ。
- 学習: 本番トレーナー train_patterns_v3(WTHOR 74,024局・443万サンプル・石差ラベル・オンラインSGD 20epoch)。T155で簡易コーパス入力モードあり。
- 検証環境の教訓(T157): oracle 60局面は0.3石級の微差判定に無力。180局面版(t157系)でもWTHOR系重み間は有意差が出ない。**採用判定は対局ゲート必須**(重い、後回しリストでまとめて実施予定)。
- MPCシリーズ(T156)はGate 3不合格で撤退済み。探索側は現状維持(160kノード予算・決定性最優先)。

## 核心論点(諮問したいこと)

1. **eval hot pathのコスト**: 評価関数は探索の葉で毎回呼ばれる。モビリティ(両者の合法手数)を葉ごとに計算するコストはbitboardでも無視できない(合法手生成はKogge-Stone系で十数命令×2)。囲い度も同様。**NPS低下がどの程度見込まれ、特徴の寄与で相殺できる見込みがあるか**。Egaroucid/文献の実装(近似で十分とされる)を踏まえた「安い計算法」の設計(例: 正確な合法手数 vs popcount近似、囲い度の定義)。
2. **特徴の設計**: (a)モビリティ差・囲い度差をステージ別線形項として足す最小案 (b)値域・スケーリング・61段ステージとの関係 (c)学習側(train_patterns_v3)の拡張方法(パターン重みと同時学習)。
3. **実験プロトコル**: 学習→スクリーニングの段取り。T157の知見を踏まえ、oracle 180で差が出ない可能性が高い中で何をスクリーニング指標にするか(候補: frozen MAE/agreement、in-corpus指標、少数局対局スモーク)。最終採用ゲート(対Edax 60局paired)は後回しリストへ積む前提で、**「対局ゲートに送る価値がある候補」をどう安価に絞るか**。
4. **決定性・WASM整合**: 特徴計算のnative/WASM差、ANALYSIS_ENGINE_VERSION、NPS計測の条件。
5. **タスク分割案**(engine側特徴計算+eval統合 / train側拡張+学習 / 計測、各1セッション粒度)。

## 制約

- 探索の決定性(同一入力同一出力)を壊さない。既定挙動(特徴なしv4)は完全不変で切替可能に(T156cのSearchPolicy方式の前例)。
- 重い検証(対局ゲート)は本設計の範囲では実行しない(後回しリストへ)。
- NPS計測は専有状態で。
