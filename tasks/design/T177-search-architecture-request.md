## 諮問事項(設計してほしいこと)

### A. 反復深化+持ち時間制への移行設計(本丸)

- 現行: 1手=固定160kノード(+wall1500ms保険)、対局のみ空き20以下無制限exact。決定性が絶対要件(同一局面→同一手。実測NPSを制御に使わない past 方針: T085)。
- 目標: Edax同様の「深さ1→2→…と積み上げ、持ち時間で打ち切る」方式。1手平均2-3秒・最悪でも数秒に収まる時間管理。**決定性との両立が最大の設計論点**(時間打ち切りは本質的に非決定 → (a)決定的な代理予算〔ノード数〕を反復深化の打ち切りに使う (b)深さ完了単位でのみ採用し中断深さは捨てる (c)非決定を許容し解析キャッシュ側で対処、等の選択肢の比較を求む)。
- 対象経路の整理: 対局CPU(強)/評価バー/棋譜解析(analyzeAll)/詰め、それぞれの要件差(解析は再現性最重要、対局はUX応答性)。
- exact移行(空き20無制限)・quota制度との整合、ANALYSIS_ENGINE_VERSION運用。

### B. マルチスレッド化(フェーズ7、ブラウザWASM)

- 環境: GitHub Pages+coi-serviceworker(COOP/COEP注入済み)、Rust→wasm-bindgen。SharedArrayBuffer利用可の前提確認込み。
- 候補: Lazy SMP(共有TT)か、root分割か、YBWCか。決定性への影響(Lazy SMPは非決定が本質 → Aの決定性設計と連動)。
- ネイティブ計測経路(vs_edax/eval_cli)とWASM本番の二重実装コストの最小化。

### C. MPC本採用の位置づけ

- t=1.0再校正済み・margin_tパラメータ化済み。深さベース移行時にON前提でよいか、対局/解析での使い分け、aspiration併用の一致保証問題(T089a申し送り: 狭窓でのMPCカットはfull-window一致保証を崩す)の扱い。

### D. 増分評価(オプション、費用対効果の判断を求む)

- 現行score()は毎回46パターンを全計算(約540ns/eval)。Edax式の「着手で変化したセルだけ特徴インデックスを差分更新」の実装コスト・期待NPS・канonical index機構(T163)との整合。A/Bの後回しでよいかの判断。
