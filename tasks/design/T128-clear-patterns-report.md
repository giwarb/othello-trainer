# 設計レポート: 中盤練習「明確な悪手パターン」の追加候補(T128拡張)

- 依頼書: `tasks/design/T128-clear-patterns-request.md`
- 作成: 2026-07-18(設計コンサル代行。読み取り専用調査: `engine/src/explain.rs`、`app/src/analysis/motifs.ts`、`app/src/analysis/types.ts`、`app/src/midgame/clearBlunder.ts`、`tasks/T128-midgame-clear-blunder-compare.md`、`othello-trainer-design-verbalization.md`)

## 要旨

追加候補を10件評価し、**エンジン変更ゼロ(TSのみ)で実装できる4件を第1波(T128b)として推奨**する: ①隅の取り逃し ②相手パスの取り逃し ③自分の打てる場所の激減 ④石の取りすぎ(大量返し)。いずれも「盤面2枚対比+ハイライト+数を見せるだけ」で初中級者に一目で伝わり、既存5パターンが弱い「隅を取らない/自分が詰まる/取りすぎ」の3系統を埋める。第2波(T128c)に⑤危険な辺の形(ウィング) ⑥種石の献上を回す。パリティ系・ストナー・余裕手系は却下(平易さの絶対条件を満たさない)。**エンジン(Rust)の追加特徴量は不要**であり、教師コーパス生成走行中でも第1波は着手可能。

## (a) 推奨と理由

### 1. 追加候補の列挙と4軸評価(判断してほしい点1)

評価軸: (a)平易な言語化可能性 (b)静的検出可能性(1手先局面+合法手列挙の範囲) (c)初中級者(オセロクエスト帯)の実際の悪手での出現頻度(定性) (d)既存特徴量/モチーフとの重複。◎○△×の4段階。

| # | 候補 | (a)平易さ | (b)静的検出 | (c)頻度 | (d)重複 | 総合 |
|---|---|---|---|---|---|---|
| 1 | **隅の取り逃し**(最善手が隅なのに取らなかった) | ◎ 「隅を取れるのに取らなかった」の一文+隅ハイライトで完結 | ◎ `bestSquare`が隅か見るだけ。閾値すら不要 | 中〜高(初級は隅取りの見落とし・過剰な警戒が多い) | なし(既存5種はどれも「相手に隅を取られる」方向のみ) | **採用(最優先)** |
| 2 | **相手パスの取り逃し**(最善手なら相手がパスだった) | ◎ 「最善手なら相手はパスで、続けてあなたの番でした」 | ◎ `legalMoves(afterBest, opp).length === 0` | 低〜中(中盤後半で時々。発生時のロスは大きい) | opponent-mobilityの特殊ケース(差2以下だと既存閾値3で発火しない穴がある) | **採用** |
| 3 | **自分の打てる場所の激減**(手詰まり・パス危険) | ◎ 相手側と対称の「打てる場所の数」対比。数とマスを見せるだけ | ◎ `FeatureSet.moverMobilityAfter`が両手分既にある | 高(初中級の中盤崩壊の典型: 気づいたら自分だけ打つ所がない) | opponent-mobilityの裏面(既存は相手側しか見ていない)。motifs`jimetsu`と思想が近いが対比形式は未実装 | **採用** |
| 4 | **石の取りすぎ(大量返し)** | ◎ 「一度に{n}個も返した(最善手は{m}個)」+返した石のハイライト。初級指導の定番「石は少なく取る」そのもの | ◎ 着手前後の盤面差分で返した石を数えるだけ | 高(初級の最頻の悪癖。греedyな大量取り) | wall-frontier・openness(`zengaeshi`)と相関するが、「返した石の個数」という語り口は最も直感的で、frontier閾値4で漏れるケースを拾える | **採用** |
| 5 | **危険な辺の形(ウィング)を作る** | ○ 「隅だけ空けて辺に並んだ形は隅から反撃される」+辺8マスのハイライトで見せられる | ○ `FeatureSet.edgeShapes`にwing判定済み。ただし現行分類は**石の色を区別しない**ため「自分の石によるウィング」の色チェックをTS側で足す必要あり | 中(辺の取りすぎ・伸ばしすぎは初中級で頻出) | edgeShapes(特徴量7)は未使用資産。motifsに対応なし | **第2波** |
| 6 | **種石の献上**(相手の辺打ちで自石が返される) | △〜○ 概念はやや高度だが「相手が辺に打つと、この石が返される」+該当自石ハイライトなら見せられる | ◎ `FeatureSet.seedStones`が両手分既にある | 中 | motif`tanezukuriSupply`と同一概念(対比形式が未実装なだけ) | **第2波** |
| 7 | 開放度対比(中割りの逆) | △ 「返した石のまわりの空きマス数」は初中級への説明が回りくどい | ◎ `FeatureSet.openness` | 高(だが#4とほぼ同じ場面で発火する) | `isUchiwari`/`zengaeshi`と重複、#4と競合 | 却下(#4で代表) |
| 8 | 偶数領域への踏み込み・連打の献上 | × 領域パリティは盤面2枚対比でも初中級に一目では伝わらない | ○ `parityRegions`あり(ただし正確な適用は手番巡りの解析が必要で近似が粗い) | 中盤練習の手数帯では低 | motif`gusuuHouki`(近似実装済み) | 却下(終盤向け将来課題) |
| 9 | ストナー等の罠に掛かる | × 既存実装自体が「検証済み定義なし」の保守的近似(motifs.tsの注記)。誤検出時の説明が成立しない | △ | 低 | motif`stoner` | 却下 |
| 10 | 一方的に打てる領域・余裕手の損 | × 「余裕手」は浅い探索(ロス<0.5の手数え)が必要=静的でない。motifs.ts未実装5種の理由がそのまま当てはまる | ×(依頼書の制約「深い探索を要求しない」に抵触) | 中 | 未実装5種(余裕手の温存)と同根 | 却下 |

補足: 「連打の献上」は#2(パス)と#8(パリティ)に分解される。「山」等ウィング以外の辺悪形は#5に含めて段階的に扱う。

### 2. 推奨追加セット(判断してほしい点2)— 価値の高い順

共通前提: 既存`clearBlunder.ts`の`DetectContext`(boardAfterPlayed/boardAfterBest計算済み)に検出器を追加するだけで済む。**全候補ともエンジン変更不要**。

#### ① missed-corner(隅の取り逃し)— TSのみ・実装最小

```
if (CORNER_SQUARES.includes(bestSquare) && !CORNER_SQUARES.includes(playedSquare)) → 検出
severity = 8  // corner-gift(10)とx-c-danger(6)の間: 「取られる」より軽く「危険手」より重い
```
- 閾値不要(エンジンが「隅が最善」と言っている事実そのものが根拠。「隅=常に良い」という一般論ではない点で誤検出リスクがない)。
- 言語化テンプレ: 「**隅({座標})を取れるのに取りませんでした。隅は一度取るとひっくり返されない、いちばん強いマスです。**」
- ハイライト: best側盤面=取るはずだった隅(+その手で確定する石があれば`computeStableSquares`差分)、played側=同じ隅(空きのまま)。
- 実装コスト: 極小(10行程度+テスト)。

#### ② opponent-pass-missed(相手パスの取り逃し)— TSのみ・実装最小

```
if (legalMoves(boardAfterBest, opp).length === 0 && legalMoves(boardAfterPlayed, opp).length > 0) → 検出
severity = 9  // パス強制は隅級のテンポ利得。corner-giftに次ぐ
```
- 言語化テンプレ: 「**最善手なら相手は打てる場所がなくパスでした。続けてあなたの番になれたのに、この手だと相手は{n}か所に打てます。**」
- ハイライト: played側=相手の合法手、best側=なし(「打てるマスがない」ことが視覚情報)。
- 注意: 既存opponent-mobilityは差≥3で発火するため「2→0」のような小差パス逃しを取りこぼす。独立検出器にする(opponent-mobility内の特例分岐でも可だが、メッセージ・severityが別物なので分けるほうが素直)。
- 実装コスト: 極小。

#### ③ own-mobility-collapse(自分の打てる場所の激減)— TSのみ・小

```
playedOwn = playedFeatures.moverMobilityAfter   // FeatureSetに既にある
bestOwn   = bestFeatures.moverMobilityAfter
if (bestOwn - playedOwn >= 3 && playedOwn <= 4) → 検出
severity = bestOwn - playedOwn
```
- 閾値案: 差≥3は既存`OPPONENT_MOBILITY_THRESHOLD`と対称。**絶対数≤4の追加条件**が肝で、「10→7」のような余裕がある減少では発火させず「一目で苦しいと分かる」場面に絞る(定数コメントに明記)。
- 言語化テンプレ: 「**この手の後、あなたが打てる場所は{playedOwn}か所しかありません。最善手なら{bestOwn}か所ありました。**」 playedOwn=0なら「**…打てる場所がなくなり、パスになるおそれがあります。**」
- ハイライト: 両盤面に自分の合法手マス。
- 正直さの注意点: 着手直後は相手番なので「あなたが次に打てる場所」は相手の応手で変わる近似。ただし既存opponent-mobilityと同じ近似水準であり、静的モビリティ対比はオセロ指導の標準的な見せ方なので許容範囲(文言を「このままだと」と含みを持たせる選択肢はUI裁量)。
- 実装コスト: 小(FeatureSetの既存フィールドのみ)。

#### ④ mass-flip(石の取りすぎ)— TSのみ・小

```
flips(board, after, side) = (afterのside石マス) − (boardのside石マス) − {着手マス}   // 盤面差分、TSで数行
if (flipsPlayed − flipsBest >= 4 && emptyCount(preMoveBoard) >= 16) → 検出
severity = flipsPlayed − flipsBest
```
- 閾値案: 序中盤の最善手のフリップ数は1〜3個が典型(中割り志向)。差4以上なら「取りすぎ」と一目で言える。**空き16以上ガード**は「終盤に近づくと多く返すのが正しい局面が増える」ための適用範囲限定(中盤練習の手数帯なら実質常に成立するが、安全弁として定数化)。
- 言語化テンプレ: 「**この手は一度に{n}個も返しています(最善手は{m}個)。序盤・中盤で石をたくさん返すと、あとで相手に返され放題の形になりやすいです。**」
- ハイライト: 両盤面で「この手で返した石」。
- 重複制御: wall-frontierと同じ場面で発火しやすいが、既存のseverity降順・最大2件の仕組みで自然に整理される(取りすぎの説明としてはこちらの方が平易なので、同時発火時に本パターンを優先する調整も検討可)。
- 実装コスト: 小。

#### ⑤ bad-edge-wing(危険な辺の形を作る)— TSのみ・中【第2波】

```
wingByMover(edge, board): edgeShapes(FeatureSet)がwingの辺について、
  空いている隅側のC相当マス(と辺上の連なり)が自分の石であることをTSで色チェック
if (wingByMover数(afterPlayed) > wingByMover数(afterBest)) → 検出
severity = 7
```
- 言語化テンプレ: 「**辺のこの形(隅だけ空けて並んだ形)は、相手に隅から反撃される足がかりになります。最善手ならこの形になりませんでした。**」+辺8マス+対応する隅のハイライト。
- コスト/リスク: Rust側`classify_edge`は**石の色を区別しない**(占有のみ)ため、相手石によるウィングや混色を誤って「自分の悪形」と説明する誤検出があり得る。TS側の色チェック追加(中コスト)が必須。これが第2波に回す理由。エンジンに「色付き辺分類」を足す案もあるが、TSで十分なのでエンジン変更は不要(仮にエンジン側に寄せる場合はコーパス生成完走後)。

#### ⑥ seed-supply-contrast(種石の献上)— TSのみ・小〜中【第2波】

```
if (playedFeatures.seedStones.length >= 2 && bestFeatures.seedStones.length === 0) → 検出
severity = 5
```
- 言語化テンプレ: 「**この手の後、相手が辺(印のマス)に打つと、あなたの石(ハイライト)がひっくり返されてしまいます。最善手ならその心配はありませんでした。**」
- 閾値案: played≥2かつbest=0という非対称条件で「対比が鮮明な場合だけ」に絞る(1個差は日常的)。
- 第2波の理由: 概念がやや高度で、ハイライトの見せ方(相手の辺打ちマスと返される自石の両方)のUI設計が他より重い。頻度も#1〜4より下。

### 3. カバレッジの見立て(判断してほしい点3)

**定量の裏取りは不可**だった: 棋譜解析キャッシュはIndexedDB(ブラウザ内)でリポジトリに存在せず、`bench/`配下に対局棋譜の成果物も見当たらない(`bench/`直下は`ffo_positions.json`のみ、`edax-compare/`は生成走行中につき対象外)。また実測にはエンジン実行が必要で禁止事項に当たる。以下は理論的・定性的見積もり。

初中級者の「評価値上の悪手」を指導理論の系統別に分けると、おおよそ:

| 系統 | 想定割合 | 現行5種 | +第1波 | +第2波 |
|---|---|---|---|---|
| 隅絡み(X/C打ち・隅献上・**隅取り逃し**) | 25〜35% | 献上方向のみ | 取り逃しも捕捉 | 同左 |
| 手数系(相手を自由にする・**自分が詰まる**・**パス逃し**) | 20〜30% | 相手側のみ | 両面+パス | 同左 |
| 取りすぎ・壁・全返し | 15〜25% | frontier差≥4のみ | フリップ数でも捕捉 | 同左 |
| 辺の悪形・種石 | 約10% | なし | なし | ウィング+種石 |
| 深い差(パリティ・テンポ・通し筋・数手先の手筋) | 10〜20% | 対象外 | 対象外 | 対象外(方針どおり合格扱い) |

- 現行5種の捕捉率(悪手のうち≥1パターン発火)は**50〜65%**程度と見積もる(隅系と大きな手数差は拾えるが、「隅を取らない」「自分が詰まる」「取りすぎ(frontier閾値未満)」が丸ごと穴)。
- 第1波(①〜④)で**70〜80%**程度へ。特に①③④は初中級の最頻悪手系統を直接埋める。
- 第2波(⑤⑥)で**80〜85%**程度。
- 上限は85〜90%と見る。残りは本質的に「1手先の静的対比では説明できない差」であり、ユーザー裁定(説明できなければ合格扱い)の設計意図そのもの。**捕捉率100%を目指して閾値を緩めるのは本末転倒**(「わかりにくい説明を増やすくらいなら説明しない」の方針に反する)ことを明記しておく。
- 検証案(任意・後続): コーパス生成完走後に、practice/解析で使った局面をオフラインで再判定するNodeスクリプト(engineのWASMをheadless呼び出し)で「悪手判定→パターン発火率」を実測するタスクを切れる(下記T128d)。

### 4. 段階導入(判断してほしい点4)

- **T128b(T128直後・即時)**: ①missed-corner ②opponent-pass-missed ③own-mobility-collapse ④mass-flip。全てTSのみ・既存`clearBlunder.ts`への検出器追加+ユニットテスト+PracticeModeの文言/ハイライト配線。T128の受け入れ基準(合成盤面での陽性/陰性テスト等)をそのまま踏襲。エンジン変更ゼロなのでコーパス生成と競合しない。
- **T128c(後続)**: ⑤bad-edge-wing(TS色チェック実装込み) ⑥seed-supply-contrast。閾値の実戦調整(T128b運用後の感触を反映)も同梱。
- **後回し/やらない**: パリティ系(終盤ステージ導入時に再検討)、ストナー・余裕手系(平易さ条件を満たさない限り追加しない)。
- **T128d(任意)**: カバレッジ実測スクリプト(コーパス生成完走後)。

## (b) 代替案と却下理由

1. **開放度(openness)対比を第1波に入れる案**: `FeatureSet.openness`をそのまま使えて実装は最小だが、「返した石に隣接する空きマスの数」という概念の説明が初中級には回りくどく、発火場面が④mass-flipとほぼ重なる。④の「返した個数」の方が圧倒的に平易なため却下(④で代表)。
2. **エンジンに新特徴量(色付き辺分類・フリップ数等)を追加する案**: 依頼書は1〜2個まで許容するが、検討した全候補がTS側の盤面差分+既存FeatureSetで実装できるため不要。エンジン変更はビルド・再検証コストとコーパス生成走行中の制約(完走後まで着手不可)を負うだけで利得がない。将来⑤をRust側へ寄せたくなった時に再検討で足りる。
3. **既存パターンの閾値を緩めてカバレッジを稼ぐ案**(例: opponent-mobility差≥2): 発火は増えるが「一目で明らか」でない微差の説明が増え、方針(説明できないなら合格)と矛盾する。新系統の追加で穴を埋める方が筋が良い。却下。
4. **パリティ(偶数放棄)を対比形式で入れる案**: `parityRegions`資産はあるが、初中級者が盤面2枚を見比べても「偶数領域に踏み込んだ」ことの意味は自明でなく、既存motif実装自体が粗い近似。中盤練習の手数帯では頻度も低い。却下(終盤向け機能で再検討)。
5. **②を独立パターンにせずopponent-mobilityの文言分岐にする案**: 実装は最小になるが、severity(9 vs 差分値)とハイライト(空を見せる)が別物になり、既存検出器の単機能性(1関数1パターン+独立テスト)を崩す。独立検出器を推奨(ただし実装者裁量で分岐実装にしても致命的ではない)。

## (d) 確認事項

1. **mass-flipの適用ガード**: 空き≥16(案)で足りるか。中盤練習ステージの手数帯(開始局面の空きマス分布)を確認し、実質常時適用ならガード定数はコメント付きで残すだけでよい。
2. **own-mobilityの文言の正直さ**: 「あなたが打てる場所」は相手の応手前の近似値。文言を断定調のままにするか「このままだと」を添えるか(ユーザーの平易さ優先方針なら断定調+用語集で補足、を推奨)。
3. **MAX_PATTERNS=2の維持**: 検出器が9〜11種に増えると2件上限で重要パターンが隠れる場面が増える。2件維持(平易さ優先)を推奨するが、ユーザー確認の価値あり。
4. **①と corner-gift の同時発火**(隅を取らず、かつ相手に隅を取られる)は情報が重複気味。severity順で両方表示されるのを許容するか、①を抑制するか(推奨: 許容。メッセージの向きが「取れたのに」「取られる」で異なり学習上は相補的)。
