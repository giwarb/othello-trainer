## 背景

- MPCは実装済み(`engine/src/mpc.rs`: MIN_DEPTH=5、REDUCTION=2、T=1.5、深さ別σテーブル)だが、**本番では feature flag `mpc_enabled` で常時無効**。旧評価関数時代の校正で「同一深さノード数・同一時間予算の到達深さとも改善せず、対局も24局7勝17敗(平均-13.5)」となり既定オフが確定した経緯(mpc.rs:79、Cargo.toml:9-19 にメモ)。
- その後、評価関数は大きく進化: 現本番は **v4×WTHOR**(パターン38形状×61ステージ、oracle regret 1.111、T147で公開)。探索側も aspiration+history(ノード予算経路のみ有効)、NWS/PVS、TT bucket化、ETC、exactクオータ60%、空き20以下無制限完全読み、が入っている(T085〜T116)。
- 参考知見: Egaroucid作者の解説では MPC は「同時間で約3手深く読める」最大の探索改良(t≒1.5、深さペア(d,D)表、σはデータセットで事前計測、終盤向けは最終石差vs浅い探索値のσを別途計測)。当プロジェクトのギャップ分析(tasks/design/engine-gap-analysis-2026-07-20.md)でも最優先候補。
- **重要な制約・申し送り**:
  1. **決定性が最優先方針**: 探索制限はノード数予算(160k)が主・壁時計は保険のみ。同一入力→同一出力を壊す設計は不可。
  2. **T089a申し送り**: aspiration(狭窓)とMPCカットの併用は「full-window完全一致」保証を崩す。再有効化時は併用可否の再設計が必要と明記されている。
  3. analyzeAll(表示経路)はT139で手ごと独立TT化済み。MPCは現在この経路でも無効(featureフラグはビルド全体に効く)。
  4. 対局CPU「強い」はノード予算160k+exactクオータ60%+空き20以下無制限。校正基盤: `engine/src/bin/calibrate_mpc.rs`(--features mpc_enabled でビルドして比較検証する既存ツール)。
  5. ユーザー方針: 重い検証(60局対局ゲート)は後回し。まず軽い計測(σ再計測・ノード数/regret比較)で有望性を判定したい。

## 諮問したいこと

1. **旧校正が失敗した原因の分析**: リポジトリのmpc.rs・校正ログ・タスク記録(T052等、grep可)から、旧評価関数で「ノード数すら減らなかった」原因の仮説(σが大きすぎた? 評価関数のノイズ? (d,D)ペアの選択? 実装上の問題?)。
2. **v4でのσ再計測の設計**: 計測局面の分布(WTHOR中盤? どのply帯?)、ステージ(空きマス)別σにすべきか、(d,D)ペア表の設計、サンプル数、決定性の担保方法。
3. **aspiration・ノード予算決定性との整合**: (a)MPCとaspirationの併用可否、併用しない場合どちらを優先すべきか(定量判断の方法込み) (b)ノード予算経路でMPCを有効にしても決定性(同一入力同一出力)は保てるか(保てるはず=乱数不使用だが、TT/quota経路との相互作用の落とし穴があれば指摘)。
4. **段階的な採否設計**: 軽いゲート(例: 固定深さでのノード数削減率、同一ノード予算での到達深さ、oracle regret)の設計と合格基準案。対局ゲートは最後にまとめて行う前提。
5. **タスク分割案**(各タスク1セッションで完了できる粒度)。
