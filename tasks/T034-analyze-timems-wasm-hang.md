# T034: 【緊急】棋譜解析でWASM環境のtime_ms予算が機能せず数分間ハングする不具合の調査・修正

## 目的
T033のverifierが検証中に発見した重大な性能問題を調査・修正する。本番環境(GitHub Pages、実際のWASM/ブラウザ実行)で、24〜30手程度のごく普通の合法な棋譜を「棋譜解析」モードで解析すると、最初の1手の解析だけで**6分以上応答が返らない**現象が確認された。これは棋譜解析モード(T029〜T033)全体の実用性に関わる重大な不具合であり、優先的に修正する。

## 背景・コンテキスト
- 発見の経緯: T033のverifierが、自作の24手・30手のランダム合法手棋譜を本番URL(`https://giwarb.github.io/othello-trainer/`)で解析したところ、最初の1手分の解析(`cmd:"analyze", allMoves:true, depth:18, timeMs:1500`)が6分以上応答しなかった(バックグラウンドでポーリングしたが応答なし)。一方、8手のランダム棋譜は約23秒で解析完了し正常動作した。「対局」モード(depth8・単発、`allMoves`無し)は約20秒で正常応答しており、Worker/WASMパイプライン自体が完全に壊れているわけではない。
- `app/src/analysis/analyzeGame.ts`(T029)は`ANALYZE_LIMIT = { depth: 18, timeMs: 1500, exactFromEmpties: 22 }`を使っている。これは「反復深化+時間予算」方式で、T029実装時に「depth18を時間無制限で使うと初期局面の解析だけで60秒超かかる」ことが判明し、時間予算を追加して回避したはずだった(T029の作業ログに経緯あり)。
- `engine/src/search.rs`には`search_all_moves`が`time_ms`予算を守ることを検証するネイティブ単体テストが存在し(`analyze_with_all_moves_true_respects_time_ms_and_returns_promptly`等)、`cargo test`ではネイティブ実行で2秒以内に収まることを確認済み。**つまりロジック自体はネイティブでは正しく時間予算を守っている**。
- しかし本番のWASM/ブラウザ実行環境では、特定の(何ら特殊でない)中盤局面に対して、time_ms予算による打ち切りが機能していない、または著しく遅延している可能性が高い。考えられる原因(いずれも未検証、実装者が調査すること):
  1. WASM実行はネイティブよりノード評価が大幅に遅く(JIT最適化の差、シングルスレッド等)、時間チェックの頻度(何ノードごとにチェックしているか)が粗いため、1回のチェック間隔中に予算を大幅に超過してしまっている可能性。
  2. `search_all_moves`が全合法手それぞれに対して独立に深さを反復深化している場合、「全体でtime_ms」のつもりが「1手ごとにtime_ms」になっている、またはその逆で予算配分が意図と異なっている可能性。
  3. `web-time`クレート(T012で`std::time::Instant`のWASMパニック回避のために導入)がブラウザ環境で正しく経過時間を計測できていない可能性(以前の問題とは別の潜在バグ)。
  4. Worker側のメッセージ処理・タイムアウト機構に問題がある可能性(エンジン側は時間内に応答しているが、Worker⇔メインスレッド間のどこかで詰まっている)。
- **本タスクでの調査方針**: 推測で修正せず、まず実際に本番環境相当(WASMビルド)で問題を再現し、どこで時間予算超過が起きているか(Rust側のsearch.rsの反復深化ループなのか、Workerとの通信なのか)を特定してから修正すること。過去にこのプロジェクトで根拠のない修正を行って問題を悪化させた事例は無いが、原因不明のまま場当たり的にtime_msの値を下げる等の対処療法は避け、根本原因を特定すること。

2026-07-09 オーケストレーター(1回目のやり直し依頼):

verifierは合格でしたが、reviewerが独立コードレビューで1件の重大なデータ整合性バグ(修正必須)を発見しました。verifierの検証は受け入れ基準のコマンド実行が中心で、`protocol.rs`の`score.type`決定ロジックまでは踏み込んでいなかったため検出されませんでした。

### must: `search()`(単体API)がタイムアウトで静的評価にフォールバックした際、`score.type`が実態と異なり「exact(完全読み)」のまま返される

`engine/src/protocol.rs`の`score_kind`(`"exact"`/`"midgame"`)は、**着手前局面の空きマス数と`exact_from_empties`の比較のみ**で事前計算されており、`search()`が実際に完全読みを完走できたかどうかを見ていません。T034の修正により、`search()`のルート分岐で`solve_exact_bounded`がタイムアウトした場合、`negascout`経由で同一局面(既に予算切れ)に再突入して結局`best_move: None, depth: 0, score: evaluate_for(...)`という**ただの静的評価**を返す経路が生まれましたが、呼び出し元の`protocol.rs`は事前計算した`score_kind`(`"exact"`のまま)をレスポンスに使うため、**「完全読みで時間切れになった不正確な値」が「完全読み(exact)」だと偽装されて返ってしまいます**。

これはT034以前には存在しなかった不変条件の破壊です(以前はこの分岐が無条件に`solve_exact`を完走させていたため「exactラベル=実際にexact」が常に成立していました)。実際の影響箇所: `app/src/analysis/BlunderPanel.tsx`のフリー分岐探索機能が`ANALYZE_LIMIT`(`timeMs:1500, exactFromEmpties:22`)で単体`requestAnalyze`を呼び、`resp.score.type`をそのまま評価バッジの色分け(T019、確定=緑/中盤=黄)に使っています。空き22前後の重い局面(FFO#41級、約75秒かかりうる)でタイムアウトすると、ユーザーには「確定」の緑バッジで実は粗い静的評価が提示されてしまいます。

**修正**: T018で`allMoves`応答に対して既に行われている対応(`MoveEval::is_exact`、実際の探索結果に基づいて`score.type`を決定)と同じ方針を、`search()`の非`allMoves`応答にも適用してください。`SearchResult`に実際の探索結果を反映する`is_exact`相当のフィールドを追加し、`protocol.rs`はこれを見て`score.type`を決定するようにしてください(事前計算した`score_kind`をそのまま使わない)。

### should(余力があれば対応、必須ではない)
- `search()`のタイムアウト時フォールバックが`search_all_moves()`より粗く、`best_move`が完全に失われる(`None`になる)。`search_all_moves()`と対称的に、タイムアウト時も静的評価+実在する合法手を返すようにすることを推奨。

**原因特定**

1. まず`negascout`(中盤探索)の反復深化ループに一時的なノード数・経過時間ログ(`eprintln!`、`T034_DEBUG`環境変数でガード)を追加して調査。上記局面の最初の合法手について、depth=10で156ms・depth=11で399ms(累計556ms)と、depthが1増えるごとの所要時間の増加は緩やかだったが、depth=12のイテレーションが30秒経過しても完了しなかった。
2. さらに詳細なログ(ノード数のチェックポイントごとの経過時間、`solve_exact`呼び出しの直前直後のログ)を追加したところ、**`negascout`の再帰の奥深くで`empties<=exact_from_empties(22)`の局面に到達し、`endgame::solve_exact`(完全読みソルバー)を呼び出した直後に処理が返ってこなくなっている**ことを特定した。
3. `endgame.rs`のモジュール冒頭ドキュメント、および`engine/tests/ffo_bench.rs`のドキュメント(T009)に明記されている通り、`solve_exact`(`negamax`)は**安定石カット等の高度な枝刈りを持たない素朴なalpha-beta+TT**であり、空き22前後の局面でもFFO#41(空き22)で約1.9億ノード・75.8秒、FFO#42(空き22)で約3.2億ノード・116.0秒かかることが公式に記録されている。`search_all_moves`の探索木は深さ最大18まで反復深化するため、ルート局面(空き30〜34)から探索中の様々な局面で頻繁に空き22の境界を跨ぎ、その都度この「重い」`solve_exact`が(無条件・無制限に)呼び出される。**時間予算(`time_ms`)は反復深化の「1つのdepthが完了するごと」にしかチェックしておらず、かつ`solve_exact`自体は一切の時間予算を持たず1回の呼び出しを最後まで完走するまで戻らないため、`solve_exact`が「重い」局面に当たった瞬間、時間予算は完全に無視されて数十秒〜数分単位で処理がブロックされる。**これがハングの直接原因である。
4. 結論: 当初タスクで挙げられていた仮説(1: WASMのノード評価が遅くチェック頻度が粗い/4: Worker側の通信問題)は主要因ではなく、**「反復深化の時間予算チェックの粒度が粗すぎる」ことに加え、「時間予算チェックの対象外である`solve_exact`(完全読み)が、探索木の内部で無条件かつ無制限に呼び出されている」という2段構えのロジックバグ**が根本原因だった。ネイティブ・WASMどちらでも起こりうる(WASM環境はノード評価が遅い分、同じ「重い局面」への到達確率・被害はさらに大きくなりうるが、今回確認された6分ハングは主にこの構造的バグによるもの)。

**修正内容**

1. `engine/src/search.rs`
   - `SearchCtx`に`start: Instant`(反復深化1イテレーションの起点)と`timed_out: &mut bool`(時間切れフラグ)を追加。
   - `negascout`の再帰の**冒頭**(各ノード呼び出しのたび)で、`limit.time_ms`が指定されていれば`TIME_CHECK_NODE_INTERVAL`(1024)ノードに1回、経過時間をチェックし、超過していれば`timed_out`を立てて即座に`0`を返す(従来は「1つのdepthの探索が完了するごと」にしかチェックしていなかった)。子の呼び出しから`timed_out`が伝播してきた場合も、置換表に格納せず即座に展開する(不完全な探索結果でTTを汚染しないため)。
   - `negascout`が空きマス数のしきい値に到達した際の完全読み呼び出しを、`limit.time_ms`が指定されていれば`solve_exact`ではなく新設の`solve_exact_bounded`(下記)に切り替え、時間切れなら`timed_out`を立てて呼び出し元にイテレーション全体を破棄させる。
   - `search()`のルート分岐(局面自体がしきい値以下の場合)・`search_all_moves()`の各手の直接分岐(着手後の局面がしきい値以下の場合)も同様に、`time_ms`があれば`solve_exact_bounded`を使うよう変更。前者は時間切れ時に通常の反復深化ループへフォールバック、後者は時間切れ時にその手だけ静的評価(`evaluate_for`)にフォールバックし`is_exact=false`として報告する。
2. `engine/src/endgame.rs`
   - `negamax`に`budget: Option<TimeBudget>`と`timed_out: &mut bool`引数を追加(`TimeBudget{start, time_ms}`は新設の小さな値型)。`budget`が`None`なら従来と完全に同じ挙動・性能(既存の`solve_exact`/`solve_exact_with_nodes`はそのまま`None`を渡すため無変更)。
   - `budget`が`Some`の場合、`search.rs`と同じ「1024ノードに1回チェック」方式で経過時間を確認し、超過したら即座に`0`を返して巻き戻る(TTには格納しない)。
   - 新設の`pub fn solve_exact_bounded(board, side, tt, budget) -> Option<i32>`: 時間切れなら`None`(結果は使ってはならない不完全な値)、完走すれば`Some(score)`を返す。
3. `engine/src/bin/eval_cli.rs`: `moves`サブコマンドに`--time-ms`オプションと実行時間のログ出力を追加(調査・今後の同種調査に恒久的に有用と判断し維持)。
