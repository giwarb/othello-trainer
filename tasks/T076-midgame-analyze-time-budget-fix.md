# T076: 中盤練習の複数候補手評価で時間予算共有により最善手判定が不正確になる問題の修正

## 目的

ユーザー報告(2026-07-12、スクリーンショット付き):中盤練習モードのある局面で、実際に打った手(b4)が「失敗(最善手ではない)」、b2が「正解手」と判定されたが、ユーザーは「b2(隅a1に隣接するX打ち)の方が明らかに悪い手のはずで、評価関数がバグっているのでは」と指摘。詳細調査の結果、静的評価関数自体にバグは無いが、**中盤練習モードが実際に使う探索時間予算の設定に起因する、実用上の誤判定バグ**が見つかったため修正する。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

explorerによる詳細調査の結果:

- 報告された局面(黒番、盤面は下記参照、合法手12箇所)を実際にビットボード化し、`eval_cli`(本エンジン)とEdax(`bench/edax-compare/edax-extract/wEdax-x86-64.exe`、既にインストール済み)の両方で、b2着手後・b4着手後それぞれの評価値を比較検証した:
  - **本エンジンdepth10/11(時間無制限)**: b2=-18.7〜-15.8石、b4=-2.5〜-4.3石(黒視点)。
  - **Edax level10**: b2=-26石、b4=-4石。
  - → **深い探索では本エンジン・Edaxともに「b4の方がb2より10〜20石相当良い」と一致**しており、ユーザーの直感(b2=X打ちは良くない)を裏付けている。静的評価関数の符号バグ・パターンインデックス誤り等は見当たらない(隅の危険度を直接評価する項自体が存在せず、隅a1の帰趨はどちらのラインでも同じ)。
  - **ところが、`app/src/midgame/PracticeMode.tsx`(58行目付近)の`MIDGAME_ANALYZE_LIMIT = { depth: 16, timeMs: 300, exactFromEmpties: 24 }`という、アプリが実際に使う設定(`depth16, timeMs300ms`)では、b2=-4.6石、b4=-11.35石と、深い探索・Edaxと矛盾する形で「b2の方が良い」と誤判定していた**。
- 原因: `engine/src/search.rs`の`search_all_moves_with_eval`(414行目付近)は、`limit.time_ms`で指定された時間予算を、**全合法手を通じた累積の経過時間**として消費する設計になっている(441行目付近のコメント「全合法手を通じた累計の経過時間を計測する」、513〜514行目の`start.elapsed()`によるループ内タイムアウトチェック)。これはT034(棋譜解析でWASM環境のtime_ms予算が機能せず数分間ハングする不具合)の修正で導入された、**意図的な設計**(合計の探索時間に上限を設けることで、多い候補手数×重い完全読み等による無制限のハングを防ぐ)。
- しかし、この「時間予算を候補手数で分け合う」設計は、**候補手数が多い局面ほど1手あたりの実効探索深さが浅くなる**という副作用を持つ。本局面は合法手12箇所あり、300msを分け合うと1手あたり実質20〜30ms程度しか探索できず、この局面ではそれが致命的な誤判定を招いた。
- **重要**: これはT034の修正が誤っていたということではない(ハング防止という目的は正しく達成されている)。今回問題なのは、ハング防止のための時間予算の「総量」または「配分方法」が、精度とのトレードオフにおいて中盤練習モードには厳しすぎる(短すぎる)可能性がある、という点。

## 変更対象

- `engine/src/search.rs`の`search_all_moves_with_eval`の時間予算配分ロジック、および/または`app/src/midgame/PracticeMode.tsx`の`MIDGAME_ANALYZE_LIMIT`の値。どちらを・どう変更するかは実装判断でよいが、以下の要件を満たす必要がある。

**原因特定**

- `engine/src/search.rs`の`search_all_moves_with_eval`(T034で導入された`solve_exact_bounded`経由の時間予算チェックを保持したまま)に一時的なデバッグ出力(`T076_DEBUG`環境変数ガード、作業後に削除済み)を追加して調査したところ、`limit.time_ms=300`のとき、**合法手のうち先頭(マス番号が最も若い、この局面ではg1)の1手だけがdepth=12まで反復深化し300msの予算をほぼ完全に消費し、残り11手(b2を含む)は全てdepth=1(ほぼ静的評価1回分)しか探索されていない**ことを直接確認した。
- 原因は`search_all_moves_with_eval`の時間予算チェックロジックにあった: 合法手を通じて単一の`start`(経過時間の起点)と`limit.time_ms`(合計予算)を共有し、各合法手の反復深化ループは「（合計の)経過時間が予算を超えたら打ち切る」という早い者勝ち方式になっていた。深さ1のイテレーションは時間チェックの前に必ず1回完走するため、予算が尽きた後に評価される合法手は必ずdepth=1だけになる。今回の局面はたまたま先頭の合法手(g1)が「重い」部分木を持っていたため、そこで300msの予算がほぼ完全に消費され、b2を含む残り11手全てがdepth=1という、実質ノイズに近い浅い評価だけで順位付けされていた。これが「合法手が多い局面ほど1手あたりの深さが浅くなる」という想定内のトレードオフを大きく超え、たまたま先頭に来た1手以外は事実上ほぼ探索していないに等しい、という設計不良だった。
- T034が導入した「1024ノードごとの時間チェック」「`solve_exact_bounded`によるタイムアウト対応」自体は正しく機能しており、本タスクではこのロジックを一切変更していない(既存のT034回帰テスト・作業ログのコメントを確認済み)。今回変更したのは「合計予算をどの合法手が消費してよいか」という配分方針のみ。

**修正内容**

1. `engine/src/search.rs`
   - `search_all_moves_with_eval`のループを`for mv in moves`から`for (i, mv) in moves.into_iter().enumerate()`に変更し、各合法手を評価する直前に、その時点で残っている時間予算(`limit.time_ms`から共有起点`start`以降の経過時間を引いたもの)を「まだ評価していない合法手の数(自分を含む)」で均等に割った、その手専用の予算を計算するようにした。
   - この計算を`fn fair_share_time_ms(total_time_ms: u64, elapsed_ms: u64, moves_left: usize) -> u64`という純粋関数として切り出した(壁時計に依存しないユニットテストのため)。`remaining_ms = total_time_ms.saturating_sub(elapsed_ms)`、`share_ms = remaining_ms / moves_left.max(1)`。
   - 各合法手の反復深化ループ(NegaScout経由)・完全読み直接呼び出し(`solve_exact_bounded`)の両方に、この手専用の`per_move_limit`(`time_ms`だけをこの手の持ち分に差し替えた`SearchLimit`のコピー)と、この手専用の新しい起点`move_start`(`Instant::now()`、共有の`start`とは別)を渡すようにした。これにより、ある合法手が自分の持ち分より早く完走すればその余りは自動的に後続の合法手に回り(次の合法手の計算時に`elapsed_ms`が小さいまま渡る)、逆にある合法手が予算を使い切ってもそれ以上は進めず後続の取り分を奪わない。
   - 全合法手の持ち分の合計は常に元の`limit.time_ms`以下に収まる(各ステップで「残り予算/残り手数」以下しか割り当てないため)ため、T034が防いだ「全体のハング」のリスクは変わらない。`time_ms`を指定しない既存の呼び出し(FFO・詰めオセロ・対局モード等)は`per_move_limit = limit.clone()`となり挙動・性能とも完全に無変更。
