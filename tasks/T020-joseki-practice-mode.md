# T020: 定石練習モード本体(ゲームロジック+SRS+UI+ナビゲーション)

## 目的
設計書§2.6「定石練習モード」(オセロクエスト式)を実装する。プレイヤーが定石の手順を辿れるか練習でき、外れると理由が表示され、覚えた定石はSRS(間隔反復)で復習スケジューリングされる。

## 背景・コンテキスト
- 前提: T016(`bookgen/joseki-research.json`)・T017(`app/src/joseki/`、`app/public/joseki.json`、DAG+8対称正規化)・T018(`requestAnalyzeAll`)・T019(`EvalBadge`, `isBlunder`, `app/src/joseki/lookup.ts`)すべて完了・コミット済み。
- 設計書 `othello-trainer-design.md` §2.6.2「ゲームルール(仕様)」を参照:
  1. プレイヤーが手番の色を選択(黒/白/ランダム)
  2. 相手は現局面の`bookMoves`から**weightに比例したランダム抽選**で着手(毎回進行が変わる)
  3. プレイヤーの着手判定:
     - `bookMoves`に含まれる → 続行
     - 含まれないが評価が定石内最善と同等(石差ロス<1.0) → 「定石外・可」(既定は**ゲームオーバー**。設計書は「定石を覚える目的に忠実」として既定をゲームオーバーとしている)
     - 評価が落ちる手 → **ゲームオーバー**。正解手と、悪手の理由(石差ロス、以後の最善進行)を表示
  4. `isLeaf`到達 → **クリア**。到達した定石名と手順を表示し、そのラインの習熟度を更新
- 設計書§2.6.3「習熟管理(SRS)」: 記録単位は「ラインの終端ノード」。SM-2簡易版で次回出題日を付与。「今日の復習」画面で失敗しやすい変化を優先出題。
- **本タスクでの簡略化**: ツリービューア(定石DAGを横スクロールで閲覧するUI)は本タスクではスコープ外とする(下記参照)。出題モードの「未習熟の枝を優先するバイアス付きランダム」も本タスクでは簡易実装(未習熟のラインをランダムに1つ選ぶ程度)でよい。

## 変更対象(新規作成/変更)
- `app/src/joseki/srs.ts`: SM-2簡易版のSRSロジック(次回出題日・ease・interval・streak・failsの計算)
- `app/src/joseki/db.ts`(または既存`indexeddb`関連ファイル): IndexedDBに`josekiSRS`ストア(ラインID→SRS状態)を作成・読み書きする関数群
- `app/src/joseki/PracticeMode.tsx` + `.css`: 定石練習モードの画面(色選択、進行中のボード表示、判定結果・理由表示、クリア/ゲームオーバー画面)
- `app/src/joseki/pickBookMove.ts`: `bookMoves`の`weight`に比例した重み付きランダム抽選ロジック
- `app/src/joseki/judgeMove.ts`: プレイヤーの着手を判定するロジック(定石内/定石外だが可/悪手)。定石外判定には T018 の `requestAnalyzeAll` を使う(現局面の全合法手評価の中で、プレイヤーが打った手の石差ロスを計算)

- テストファイル一式

## 要件
1. **色選択画面**: 黒/白/ランダムの3ボタン。選択後、定石DB(`app/public/joseki.json`)のルートノード(初手前の状態。あるいは初手そのものを起点にする設計でもよい)から練習を開始する。
2. **相手の着手**: 現局面に対応する定石DBノードの`bookMoves`から、`weight`に比例した確率でランダムに1手選び着手する(T017データは均等重みなので、実質は均等ランダム)。
3. **プレイヤーの着手判定**:
   - 現局面の定石DBノードの`bookMoves`に含まれる手を打った → 続行。
   - 含まれない手を打った場合: `requestAnalyzeAll`で現局面の全合法手を評価し、プレイヤーの手の評価値と最善手の評価値の差(石差ロス)を計算する。ロス<1.0なら「定石外・惜しい」メッセージと共にゲームオーバー、ロス>=1.0なら「悪手」メッセージと共にゲームオーバー(設計書の既定方針どおり、いずれの場合も本タスクではゲームオーバーとする。「定石外・可」を続行させるオプションは本タスクではスコープ外)。
   - ゲームオーバー時、正解手(`bookMoves`の中で最も重みが高い手、または全て表示)と、プレイヤーの手との評価差を表示する。
4. **クリア判定**: 定石DBノードの`isLeaf`が`true`になったら「クリア」。到達した定石名(`names`配列)と、ここまでの手順を表示する。
5. **SRS更新**: クリア時またはゲームオーバー時、そのラインの終端ノード(クリア時は到達したisLeafノード、ゲームオーバー時は出題対象だったライン)のSRS状態をIndexedDBに保存・更新する。SM-2簡易版: 正解(クリア)なら`interval`を伸ばし次回出題日を先に、失敗(ゲームオーバー)なら`interval`をリセットして近い将来に再出題されるようにする(具体的なパラメータは実装者の判断で構わないが、一般的なSM-2の考え方(ease factor 1.3〜2.5程度、正解時はease×interval、失敗時はinterval=1日に戻す等)を踏襲すること)。
6. **出題ライン選択**: 練習開始時、IndexedDBのSRS状態を参照し、「本日出題すべき(dueが今日以前、または未挑戦)」ラインの中からランダムに1つを選んで出題する。該当が無ければ全ラインからランダムに選ぶ。
7. **ナビゲーション**: アプリ起動時に「対局」「定石練習」を選択できる簡単な画面(タブ・ボタン等)を追加する。既存の対局モード(T013/T019)の動作を壊さないこと。
8. **レスポンシブ**: 本タスクの新規UIもレスポンシブ必須(375px幅で崩れないこと)。
9. 単体テストで以下を検証する:
   - `pickBookMove`が`weight`に比例した分布になること(統計的検証。多数回試行して比率を確認する程度でよい)
   - `judgeMove`が「定石内」「定石外・惜しい」「悪手」を正しく判定すること(モックした`requestAnalyzeAll`結果を使う)
   - `srs.ts`のSM-2簡易ロジックが、正解時に次回出題日が延びること、失敗時に近い将来に戻ることを検証する
   - IndexedDBへの読み書きが正しく動作すること(テスト環境でのIndexedDBモック、または`fake-indexeddb`等のライブラリを追加してよい)
10. 実機確認: 実際にブラウザで定石練習モードを開始し、(a)相手がランダムに定石内の手を選ぶこと、(b)定石内の手を続けるとクリアまで到達できること、(c)わざと外れた手を打つとゲームオーバーになり理由が表示されること、(d)ページをリロードしてもSRS進捗が保持されること(IndexedDBの永続化確認)、(e)375px幅でも崩れないこと、を確認し作業ログに記載する。

verifier・reviewerが**独立に同じ致命的な設計不備**を発見しました。typecheck/test/build/レスポンシブ/SRS永続化/ナビゲーション自体は合格でしたが、**クリア判定ロジックが実データでは機能しない**ため、本タスクの目的(定石を覚える練習)を達成できていません。

### 発見された問題

現在の`PracticeMode.tsx`は「現在の局面ノードの`isLeaf`が`true`になったら即座にクリア」というロジックです。しかし実際の定石DB(35ライン)では、f5への3つの合法応手(d6/f6/f4)がそれぞれ2手だけの短い定石名(縦取り/斜め取り/並び取り)として登録されており、**同じノードに深さ18〜25手に及ぶ本格的な定石(虎・ローズ系・シャープローズ・手塚システムなど)の`bookMoves`(継続手)も同居しています**(T017の`buildDb.ts`の仕様上、1つのノードが「あるラインの終端」と「別ラインの分岐点」を同時に満たせるため)。

結果、**実際のプレイでは黒番プレイヤーは実質1手(f5)しか打てず、その直後(2手進行時点)に強制的に「クリア!」画面になってしまいます**。白番でも同様に、プレイヤーの初手を打った直後にクリアします。22件もの本格的な定石(本来10〜25手かけて辿るべきライン)が、実際にはその手順を一度も辿らないまま「習得済み」としてSRSに記録されてしまい、「定石外・悪手」のゲームオーバー判定パスも実データでは到達不能です(作業ログの実機確認でも、ゲームオーバーの検証は本番の`joseki.json`ではなく差し替えた偽データでしか行えていませんでした)。

### やり直しの要件

**クリア判定ロジックを以下のように修正してください:**

1. 現在ノードの`isLeaf`が`true`でも、そのノードの`bookMoves`が**非空**であれば、即座にクリアにせず**セッションを続行**してください(相手は引き続き`bookMoves`からランダムに次の一手を選ぶ)。
2. `isLeaf=true`のノードを通過した際は、そのノードの`names`(定石名)を「このセッションで通過したライン名」のリストに蓄積してください(重複を除く)。
3. セッションが本当に終了するのは、**現在ノードの`bookMoves`が空**(その先の定石データが存在しない、真の終端)になった時点、またはプレイヤーが定石外の手を打った時点(既存のゲームオーバー判定はそのまま)です。
4. 真の終端(`bookMoves`が空)に到達してクリアした場合、**セッション中に通過したすべての`isLeaf`ノードのライン名**についてSRS成功を記録してください(最終ノードの`names`だけでなく、途中で完了を通過した短いライン名も含む)。クリア画面には、到達した全ライン名(または代表的な最長のライン名)を表示してください。
5. ゲームオーバー時のSRS記録は、既存の「出題対象として選んだライン(`targetLineId`)」への失敗記録のロジックのままで構いません。ただし`targetLineId`が実際のゲーム進行(相手の着手選択)に反映されていない場合は、この点も見直してください(セッション開始時に選んだ出題対象ラインが、実際のプレイでは無関係な別ラインに終わってしまうと、SRSの復習スケジューリングの意味が薄れます。相手の着手選択で、`targetLineId`のラインがまだ辿れる状況では優先的にそちらを選ぶ、といった改善も検討してください。必須ではありませんが、可能であれば対応してください)。
6. 修正後、**本番の`bookgen`/`joseki.json`データ(モックやfetch差し替えではなく)を使った実機確認で、少なくとも1回は10手以上のセッションが成立すること、かつ定石外の手を打った際のゲームオーバーが本番データで再現できることを確認し、作業ログに記載してください**。
7. 単体テスト・統合テストで、「isLeafだがbookMoves非空のノードでは継続すること」「真の終端(bookMoves空)でクリアすること」「通過した複数ライン名がSRSに記録されること」を検証するテストを追加してください。
