# T180: 速度ボトルネック分析

## 目的

ユーザー裁定(2026-07-22未明):「他のAIと比べてまだまだ遅い。ボトルネックを分析すべき」。現状の断片的な事実(実探索NPS約89万/秒 vs Edax級エンジンは数千万/秒とされる=数十倍差の疑い、score()1回約540ns、深さ12の1手最大10秒)を、**体系的な計測で「どこで何倍差がついているか」の内訳に分解**し、最適化の優先順位リストを作る。分析タスクであり大規模な最適化実装はしない(T177諮問=7/26の材料にする)。

## 計測項目

1. **対Edaxの同条件NPS実測**: 手元のEdaxバイナリ(bench/edax-compare/edax-extract/)で同一局面集合を解かせ、ログからnodes/timeを抽出してNPSを実測(lv10/lv12の中盤探索・終盤解き両方)。うちのエンジン(深さ12、MPC on/off両方)の同一局面NPSと並べ、**倍率を確定**する。
2. **ノード単価の内訳**: うちの探索1ノードあたりの時間(現状約1.1μs)を、(a)評価(score、深さ0呼び出し頻度×540ns) (b)合法手生成・flip計算 (c)TT probe/store (d)増分hash・状態管理 (e)その他、に分解する。方法は計測しやすいもので良い(コンポーネント別マイクロベンチ+呼び出し回数カウント、またはサンプリングプロファイラ)。**「評価が支配的か、盤面操作が支配的か」を確定**させることが最重要。
3. **葉評価の呼び出し構造**: 深さ12探索でscore()が何回呼ばれるか(=葉+内部ノードの静的評価回数)。評価1回540ns×回数が総時間に占める割合。
4. **Edaxとの構造差の整理**(コード調査+文献): Edaxが速い既知の理由(増分特徴更新=着手時に変化パターンだけ更新/最適化されたflip(カーネル化・分岐レス)/TTの軽さ/評価の呼び方)と、うちの実装の対応状況の差分表。
5. **最適化候補の優先順位リスト**: 各候補(増分評価・flip最適化・TT軽量化・SIMD等)について「期待削減幅(計測に基づく上限)・実装コスト・リスク」を1表に。T177諮問(反復深化・マルチスレッド)との関係も記載。

## スコープ外

- 大規模な最適化の実装(分析が成果物。ただし計測に必要な軽い計測コード・カウンタは可、既存経路ビット不変で)
- WASM側の計測(まずネイティブで倍率と内訳を確定。WASM差は既知の係数として言及のみ)

## 受け入れ基準

1. 対Edax NPS倍率の実測値(中盤・終盤別)がある
2. ノード単価の内訳(a-e)と評価呼び出し構造の計測結果がある
3. 構造差の差分表と優先順位リストがレポート(bench/edax-compare/t180_bottleneck_report.md + meta)にある
4. 計測用に足したコードは既存経路ビット不変(実証)またはテスト専用。`cargo test -p engine` 全パス
