# T066: UI磨き込み(4): 石の反転アニメーション

## 目的

UI/UX監査で洗い出した優先度順8項目のうち、(1)盤面サイズ統一(T061)・(2)デザイントークン一元化(T061)・(3)タイトル/ホーム画面(T065)が完了済み。本タスクは4番目の項目「石の反転アニメーション」を実装する。着手時に相手の石が瞬時に色変更されるだけの現状を、実際のオセロのように石がひっくり返る演出に変える。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

- `app/src/components/Board.tsx`: 盤面を描画する**唯一の**Canvas 2Dコンポーネントで、対局・定石練習・中盤練習・詰めオセロ・棋譜解析(悪手分析パネル・反証層含む)・言語化トレーニング(練習/二択ドリル/概念レッスン/用語集)の全モードから共通で使われている(個別実装なし)。**本タスクはこのコンポーネント内にアニメーションを実装すれば、全モードに自動的に反映される。**
- 現状の描画方式: `useEffect`内の`draw()`関数(29〜106行目付近)が`[board, sideToMove, lastMove]`の変化のたびに一括再描画する同期的・命令的な単発描画。`requestAnimationFrame`はリポジトリ全体で1箇所も使われておらず、トランジション/イージングの仕組みは皆無。各マスの石は`occupant === 'black'/'white'`で単純に塗り分けられるだけ(瞬時反転)。
- `BoardProps`(8〜17行目付近)は`board: BoardState`(現在の1盤面のみ)・`sideToMove`・`lastMove?`・`onMove?`のみを受け取る設計で、**着手前後2つの盤面を比較する仕組みは無い**。「どのマスがひっくり返ったか」は、コンポーネント内部で前回の`board`を`useRef`等で保持し、新旧のビットボード(`board.black`/`board.white`、`bigint`)をXOR比較して導出する必要がある。
- **重要な判別方法**: 新旧盤面が「1手が正常に適用された結果」なのか、それとも「別局面へのジャンプ(棋譜解析でのply選択・分岐探索・新規対局開始・待った等)」なのかを区別する必要がある(後者ではアニメーションさせず瞬時に切り替えるべき)。判別には**石の総数**を使う: 1手の着手では石の総数が必ず旧盤面+1になり(パスは総数不変、着手時に置いた石+反転石はどれも総数を変えない)、それ以外の変化(総数が減った、+1以外の差、初期局面へのリセット等)は「ジャンプ」とみなしてアニメーションなしで即座に描画してよい。
- `app/src/game/othello.ts`の`applyMove`(153〜177行目付近)は純粋関数で、旧`Board`と`side`/`square`から新しい`Board`を返すのみで、反転マスの差分情報自体は返さない。
- 盤面セル評価オーバーレイ(`MoveEvalOverlay.tsx`、T039/T042)やモチーフオーバーレイ(`analysis/BoardOverlay.tsx`、T032)はCanvasとは別レイヤー(CSS Gridを`position:absolute`・`pointer-events:none`で重ねる方式)なので、Canvas内でのアニメーション追加はこれらのDOM製オーバーレイと直接の描画干渉は起きない(触れる必要はない)。
- 各モードの自動進行(CPU応手・相手の定石内進行等)は`setTimeout(..., 300〜350ms程度)`(例: `analysis/BlunderPanel.tsx`130行目、`midgame/PracticeMode.tsx`44行目、`joseki/PracticeMode.tsx`33行目、`tsume/PlayMode.tsx`38行目)で次の着手を遅延実行している。アニメーション所要時間はこれらの遅延値と厳密に同期させる必要はない(要件参照)が、極端に長くしてテンポを損なわないよう、既存の遅延値(300ms程度)を超えない範囲に収めること。

## 要件

1. 合法手が適用され石が反転する際、瞬時の色変更ではなく、石が横方向に潰れて(エッジが見える状態を経て)反対の色で戻る、実際のオセロの反転を模したアニメーションで表示すること。
2. 新しく置かれた石(反転ではなく新規配置)も、瞬間出現ではなく軽い出現演出(例: 小さい状態から本来の大きさへ拡大)を行うこと。
3. アニメーションの総所要時間は300ms程度以内に収め、テンポを損なわないこと(具体的な値は実装判断でよい)。
4. 「1手適用による通常の盤面遷移」ではない盤面変化(新規対局開始、棋譗解析での別局面へのジャンプ、分岐探索での局面切り替え、待った等、背景記載の判定方法を参照)では、アニメーションさせず即座に最終状態を描画すること(誤ってジャンプ時にも反転演出が走らないようにする)。
5. アニメーション中もクリック/タップによる着手入力(`onMove`)がブロックされず、これまで通り機能すること。
6. `prefers-reduced-motion: reduce`が有効な環境では、アニメーションを行わず即座に最終状態を描画すること(アクセシビリティ配慮)。
7. 新旧盤面比較・「1手適用かどうか」の判定ロジックは`Board.tsx`から分離した純粋関数として実装し、vitestの単体テストを追加すること(通常の1手着手ケース・パスケース・新規対局リセットケース・棋譜解析でのジャンプケースなど、複数パターンを検証する)。
8. 全モード(対局・定石練習・中盤練習・詰めオセロ・棋譜解析・言語化トレーニング)で、既存の盤面表示・クリック操作・評価オーバーレイ表示に回帰が無いこと。
9. 既存のテストが壊れないこと。

## フィードバック(やり直し時にオーケストレーターが記入)

**不合格(2026-07-11、verifier・reviewer双方が同一バグを独立に確認)**:

`app/src/components/Board.tsx`の`useEffect`内(140〜194行目付近)で、以下の重大なバグにより**アニメーションが実質発火しない**:

```
resize()                                    // 同期呼び出し。この中でdraw()が呼ばれ hasDrawnOnce=true になる
const observer = new ResizeObserver(resize)
observer.observe(container)
```

`ResizeObserver`は仕様上、`observe()`を呼ぶと実際のサイズ変化の有無に関わらず**初回通知が保証されて非同期に必ず1回発火**する。この初回通知が`resize()`→`draw()`を再度呼ぶ際、`hasDrawnOnce`が既に`true`のため「実際のリサイズが起きた」と誤認し、`cancelAnimation()`で進行中のアニメーションを即座にキャンセルして`drawFrame(1)`(最終状態)を強制描画してしまう。

verifier・reviewerともに、Canvas描画API呼び出し(`translate`/`scale`)のフック計装や`ResizeObserver`コンストラクタのラップによる独立した実測で、着手のたびにこの初回通知が着手後数十ms以内(2フレーム程度)に発火し、220msかけて進行するはずのアニメーションが実質「進行度ほぼ0→強制的に進行度1」の2回描画に潰れてしまうことを、`npm run dev`・`vite preview`・本番URLの3環境すべてで再現確認した。ユーザー視点では本タスク導入前とほぼ変わらない「瞬時に色が変わるだけ」の体感になっている。`Board.tsx`は全モード共通コンポーネントのため、影響は全モードに及ぶ。

なお、`diffBoards`(`boardDiff.ts`)のロジック自体・その単体テスト(`boardDiff.test.ts`)・アニメーションの数式(`drawFlippingDisc`/`drawAppearingDisc`)自体は両者から妥当と評価されている。**問題は「初回描画かどうか」の判定(`hasDrawnOnce`)とResizeObserverの保証された初回コールバックとの統合部分のみ**。

**修正方針(reviewer提案)**: 以下いずれかの方式で、「ResizeObserverの保証された初回通知」と「実際にコンテナサイズが変わった場合の通知」を区別できるようにすること。
- (a) `resize()`内で直前に記録したコンテナの実サイズ(幅・高さ)と比較し、変化が無ければ`draw()`を呼ばずスキップする。
- (b) `ResizeObserver`インスタンス自体をコンポーネントのマウント時に1つだけ生成・`observe()`し(`board`/`sideToMove`/`lastMove`が変わるたびに再生成しない)、その最初のコールバック呼び出しだけを明示的にスキップするフラグを別途持つ。

修正後は、reviewer・verifierが用いたのと同種の直接的な検証手法(Canvas描画API呼び出しの引数を記録する、または連続フレームのキャンバス内容差分を取る等)で、実際にアニメーション中の中間フレーム(潰れた状態のスケール値)が観測できることを必ず確認すること(キャンバスの塗り面積比較(dipRatio)のような間接的な指標だけでは、今回のように偽陽性を検出できなかった実例があるため)。
