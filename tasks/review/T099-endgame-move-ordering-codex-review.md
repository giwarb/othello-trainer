## (b) 中（次タスクで対応すべき）

1. **`MoveInfo::flips` と `MoveInfo::mv` が保存されるだけで、探索時に一度も利用されていません。**

   `MoveInfo` は `mv` と `flips` を保持していますが、ソート後の探索で使うのは `next_board` と `square` だけです。特に flip mask は、`Board::apply_move` 内部で一度算出された後、子盤面との差分から改めて復元され、そのまま未使用です。

   候補ごとの `apply_move` が二回から一回になったという本タスクの主目的は達成していますが、「flip mask を一度だけ計算して再利用する」という設計意図には十分沿っていません。また、固定長配列の要素を不必要に大きくし、各候補で余分な演算も行っています。

   今後 flip mask や増分 hash を使う段階で、flip mask から子盤面を構築して双方に再利用する設計へ整理するか、現段階で不要ならフィールドと復元処理を除去すべきです。

## (c) 軽微（記録のみ）

1. **モジュール冒頭の着手順序説明が更新されていません。**

   冒頭コメントは依然として「隅 → mobility → parity」とだけ説明しており、今回追加された最上位の TT move と最終マス番号タイブレークが反映されていません。実装の正しさには影響しませんが、設計説明とコードがずれています。

2. **追加ユニットテストは実経路を直接検証していません。**

   `move_info_sort_key_prioritizes_tt_move_and_breaks_ties_by_square` は人工的な `MoveInfo` 同士のキー比較のみです。非合法・範囲外 TT move が実際の候補生成で無視されることや、合法な TT move が先頭になることはコード上確認できましたが、回帰テストとしては実局面と TT probe を通す方が堅牢です。

3. **独立したテスト再実行は環境制約により実施できませんでした。**

   `cargo test -p engine` は read-only サンドボックスで `target/debug/.cargo-build-lock` を作成できず、開始前に失敗しました。これはコードのテスト失敗ではありません。作業ログには全 engine テスト、naive differential、FFO #40–44、決定性確認の成功が記録されており、静的レビューでもスコア計算・論理ノード加算・abort契約を壊す変更は見つかりませんでした。
