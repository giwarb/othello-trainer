## 観点1: diag4_patterns生成の正しさ — 問題なし

- `diag4_patterns()`(patterns.rs:232-240)は `diagonal_offset_patterns()`(同203-215)のループ本体を len=4 で単独実行したものと完全同型。base セルは `i*8+i+offset`(offset=4)= 4,13,22,31 → 盤座標 (r0,f4)(r1,f5)(r2,f6)(r3,f7)、すなわち e1-h4 の長さ4対角。手書きセル列なしの要件を満たす。
- `symmetry_orbit`(ソート済みセル集合の HashSet で重複除去)により D4 で移り合う長さ4対角4本(主対角平行±、反対角平行±)が得られ、`assert_eq!(orbit.len(), 4)` が実行時にも本数を担保。長さ5-7の既存 `diagonal_offset_patterns` と長さが異なるため重複はあり得ない。
- 回帰テスト(V3Corner5x2=46/11、V3Corner5x2Diag4=50/12)・形状テスト(D2先頭46=D1、末尾4本が長さ4)・D1/D2各100局面の全8対称一致テストの3点が押さえられており、クラス数(corner5x2で+1、diag4で+1)も理論どおり。
- `generate_patterns_for` の matches! 拡張は追加式(V3系の既存分岐に新variantを足すだけ)で、既存configの生成結果に影響する変更はない。

## 観点2: ScalarFeatureKind::Constant と ALL_SCALAR_FEATURE_KINDS 一元化 — 後方互換は保たれている(中1件・軽微2件)

**後方互換の確認(既存PWV4/PWV6ファイルの読み込み不変):**

- パース側(pattern_eval.rs:770)は `1..=2` → `1..=ALL_SCALAR_FEATURE_KINDS.len()`(=3)への**拡大のみ**。既存の1特徴(t158-b1/b2系)・2特徴(b3系=現行本番v5)ファイルは従来どおり受理され、読み取りループ・scale_shift検証・schema_hash照合のコード経路は不変。
- `seen` 配列は `[false;3]` → `[false;4]`。kind_value(1..=3)を直接添字にする設計は従来と同じで、旧ファイル(kind 1-2)の重複検出動作は不変。
- `score()`(同443-465)は固定2種配列の走査から `ALL_SCALAR_FEATURE_KINDS` 走査に変わったが、**列挙順は旧2種と同順+末尾にConstant**。既存2特徴ファイルではConstantの `find` が None → スキップとなり、浮動小数の加算順序も含めビット等価。`regression.rs::sgd_step` のmatchアーム追加も同様に既存2種の更新経路不変。
- `scale_shift()=0`(値1がそのまま)・score/sgd両側で `raw=1` の実装は一貫しており、D4不変テスト(`t168_constant_scalar_feature_is_always_one_and_d4_invariant`)はパターン寄与ゼロ+定数項のみ非ゼロという分離設計で「重み=そのままスコア寄与」を直接確認している。良い。
- schema_hash_v4(同284-305)はパターン形状+scalar kind/scale_shift を含むため、D1/D2ファイルは自動的に別スキーマになり取り違え不能(タスク要件2の「確認のみ」も妥当)。

**【中-1】`with_zeroed_scalar_features()` の非retain呼び出しが1つ既存経路に残る(eval_cli):**
`engine/src/bin/eval_cli.rs:128` の `make-zero-feature-model` サブコマンドは retain せず直接シリアライズするため、出力PWV4が**2特徴→3特徴に変わる**(かつ kind=3 を含むため、旧版バイナリではパース拒否される)。スコアは係数ゼロで数値等価であり、このツールはT158aベンチのアドホック用途のみ(CI・本番不使用)なので実害は小さいが、**「既存経路の完全不変」の実証(stash SHA)がカバーするのは t158-b3 学習経路のみで、この経路には既知の挙動変化がある**ことは記録に残すべき。doc上「呼び出し元がretainで絞る前提の候補全種リスト」と明記されたので設計としては一貫しているが、前提を満たさない既存呼び出し元が1つ残った形。

**【軽微-1】** `engine/tests/eval_features_nps_bench.rs:93` の candidate も暗黙に3特徴化し、「scalar特徴のオーバーヘッド計測」の測定対象がConstant1個分(find+乗加算)増える。過去計測値との経年比較時に注意。
**【軽微-2】** T164テストヘルパーの暗黙3特徴化は作業ログに正直に記録済みで全テストパスだが、意図しない波及だった点は同上の「非retain呼び出しの危うさ」の傍証。`ALL_SCALAR_FEATURE_KINDS.len()` とenum定義の整合を守るstatic assert(または網羅match)があるとより堅い。

## 観点3: 既存t158系configの不変 — コード読解で確認、問題なし

- `t158_config` は `scalar_feature_config(name, PatternConfig::V3, scalar_features)` を呼ぶだけの薄いラッパーに変更。`scalar_feature_config` の構築フィールド(num_stages=V4_NUM_STAGES、stage_empty_divisor=V4_STAGE_EMPTY_DIVISOR、t158:true、canonical:false)は旧 `t158_config` と完全同一で、変わったのは `pattern_config` が固定値V3から引数になった点のみ。**純粋な引数抽出リファクタ**。
- identity文字列(train_patterns_v3.rs:1429-1438)は config名・feature_schema(scalar kind名+scale)等から組み立てられ、t158-b0〜b3では入力が一切変わらないため文字列不変。resume時の `verify_identity` も既存checkpointと整合する。
- stash SHA実証は t158-b3-canonical の1構成のみだが、b0〜b2 は b3 と同じ構築関数・同じ走査コード(score/sgd_stepの共有経路、上記観点2でビット等価確認済み)を通るだけなので、コード読解上も他構成に影響する差分は存在しない。`config as u64`(テスト内seed導出)のようなdiscriminant依存も、新variantが既存variantの**後**に追加されているため既存値は不変。
- 新config名(t168-d1/d2)は既存名前空間と衝突せず、`scalar_kind_name` へのアーム追加は網羅matchなのでコンパイル時に強制された変更。

保険として、本対局前に `eval_cli budget-regression`(または数局のspot-check)で新バイナリ×v5重みの決定性・既存挙動一致を1回確認しておくと監査痕跡になる(60局フル再実行より遥かに安い)。

**(b) NPS-21%の対局への影響見立て:** -21%は `score()` 単体のマイクロベンチ値であり、探索全体(合法手生成・盤面更新・TT参照は不変)のNPS低下はこれより小さいはず。160kノード固定なら1手あたりwall時間は最悪でも約+27%(1/0.787)、典型的にはそれ未満。現行の1手時間が数百ms級ならwall 1500ms保険の常時発火は考えにくいが、**評価呼び出し密度の高い中盤局面で発火頻度が増える可能性はある**ため、T169では両アームのwall保険発火回数・発火時ノード数をログして比較すること(発火増=実力低下は製品でも起きる現象なので、ゲート判定としてはそのまま織り込んでよい)。終盤無制限exactは末端が石差計算でパターン評価依存が小さく(並べ替え・浅い部分のみ)、影響は軽微の見立て。ただしexact移行局面の総時間は計測して記録すること。

## 観点6: 使い捨てテスト方式の監査性 — 恒久化を推奨(中)

**【中-2】** 健全性チェック(`t168_health_check.rs`)・NPS参考ベンチ(`t168_nps_reference.rs`)を「書いて実行して削除」する方式が3タスク連続(T165/T166系→T168)で使われている。レポートに手法・数値は残るが**検証コードそのものが残らないため、第三者(将来のverifier含む)が同一手順を再実行できず、手法記述と実際のコードの乖離も検出不能**。次回以降は、重みファイルパスを引数(env var)で受ける恒久 `#[ignore]` テスト(例: `engine/tests/weights_health_check.rs`)に一般化して1回だけコミットすることを推奨する。健全性チェック(全8対称一致・finite・発散grep)は今後の全学習タスクで再利用される定型手順であり、恒久化の投資対効果は高い。NPSマイクロベンチも同様(`eval_features_nps_bench.rs` という前例が既にあるので、形状比較版として統合可能)。推奨度: 高(ただしT168の合否には影響しない)。
