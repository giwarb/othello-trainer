# T041: フェーズ3着手(2): パターン特徴量抽出 + 回帰学習パイプライン(train/クレート拡張)

## 目的

T040でWTHOR棋譜データの取得・パースが完了した。次のステップとして、盤面パターン特徴量の抽出と、それを使った回帰学習パイプラインを実装し、学習済み重みファイルを生成する。これが完了すれば、後続タスク(T042想定)で`engine/src/eval.rs`にパターン評価を統合できる。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

- 現状の評価関数(`engine/src/eval.rs`)は手作りの軽量線形評価(モビリティ・隅・安定石の3項)。今回、Logistello(強豪オセロAI)等で採用されている「盤面の局所パターン形状ごとに、その状態(空/自石/相手石の組み合わせ)に対応する重みを学習する」方式(パターン評価)を導入する。
- T040で`train/`クレートを新設済み(`train/Cargo.toml`が`engine`をpath依存に追加、`train/src/wthor.rs`がWTHORファイルパーサー、`train/src/main.rs`がCLI)。`train/data/`にWTHOR実データ(2023・2024年、計5,369ゲーム)がダウンロード済みだが`.gitignore`で除外されコミット対象外。**追加の年のデータをダウンロードしてよい**(パラメータ数に対してデータ量が少ないと学習の質が下がるため、後述の要件2を満たすため必要なら追加ダウンロードすること)。ダウンロード元は`https://www.ffothello.org/wthor/base/WTH_[年].wtb`(T040のimplementerが実際に確認済みのURLパターン)。
- 盤面座標系: `index = rank0*8 + file0`(0-63、両方0始まり)。`engine/src/bitboard.rs`の`Board`型(`legal_moves`, `apply_move`等の既存API)を使う。T040の`train/src/wthor.rs`に、WTHOR棋譜を先頭から再生して各手ごとの盤面(`Board`)・手番(`Side`)・実際の着手を復元するロジック(`replay`関数)が既にあるので、これを再利用して「学習用の(局面, 手番, その対局の最終結果)」のペアを大量に生成できる。
- **本タスクでは`engine/src/eval.rs`は一切変更しない**(統合は後続タスク)。あくまで`train/`クレート内で特徴量抽出・学習・重みファイル出力までを完結させる。

## パターン形状の定義(このタスクで実装する範囲。手作業でのセル座標転記は行わず、必ずコードで機械的に生成すること)

過去のタスク(定石データ収集T016)で「記憶に頼った手作業転記がデータ誤りを生んだ」事故があったため、パターンの座標は**ループ演算で機械的に導出**し、ハードコードした個別セル番号のリストを手で書き並べない設計にすること。以下の22パターンをv1のスコープとする(対称形による重み共有は本タスクでは行わない、各インスタンスは独立した重みを持つ。将来の改善候補として作業ログに記録すればよい):

1. **行パターン(8個)**: `rank0`を0から7まで固定し、`file0`を0から7まで動かした8セル。`indices = [rank0*8 + f for f in 0..8]`。
2. **列パターン(8個)**: `file0`を0から7まで固定し、`rank0`を0から7まで動かした8セル。`indices = [r*8 + file0 for r in 0..8]`。
3. **主対角線(1個)**: `file0 == rank0`となる8セル(a1-h8方向)。
4. **反対角線(1個)**: `file0 + rank0 == 7`となる8セル(a8-h1方向)。
5. **隅3x3ブロック(4個)**: 4つの隅それぞれについて、隅を含む3x3(9セル)。例えばa1側は`rank0 in 0..3, file0 in 0..3`。h1側は`rank0 in 0..3, file0 in 5..8`。a8側は`rank0 in 5..8, file0 in 0..3`。h8側は`rank0 in 5..8, file0 in 5..8`。いずれも2重ループで`indices`を生成すること。

合計22パターン。各パターンは3進数エンコード(各セルの状態: 0=空、1=着目手番(mover)から見た自石、2=相手石)で状態インデックスを持つ(8セルパターンなら3^8=6561状態、9セルパターンなら3^9=19683状態)。

## ステージ分割(ゲーム進行度ごとの重みテーブル分離)

石差の意味合いは序盤・終盤で大きく異なるため、空きマス数によってステージを分割し、ステージごとに独立した重みテーブルを持つこと。`stage = empty_count / 5`(空きマス60個なら12ステージ、0〜12の13ステージ程度になる想定。厳密な境界値は実装時の判断でよい)。

## 変更対象

- `train/src/patterns.rs`(新規) — 上記22パターンの定義(セルインデックス生成、機械的なループで導出)と、与えられた`Board`・`mover`から各パターンの状態インデックスを計算する関数。ユニットテストで「各パターンのセル数が期待通り(8か9)」「パターン内にセルの重複が無い」「境界の隅パターンが実際に隅マスを含む」ことを検証すること。
- `train/src/train_data.rs`(新規) — WTHOR棋譜(`wthor.rs`の`replay`を利用)から学習サンプルを生成する処理。1対局につき、初手以降の各局面(手番側視点)について「局面(`Board`, `mover`)」と「その対局の最終結果(mover視点の最終石差 = mover石数 − 相手石数)」のペアを生成する。全対局からサンプルを集める。
- `train/src/regression.rs`(新規) — オンライン確率的勾配降下法(SGD)による学習。特徴は「アクティブな(パターンID, ステージ, 状態インデックス)の組」の集合(1局面につき22個)。予測値はアクティブな重みの合計。目的関数は(予測値 − 実際の最終石差)の二乗誤差 + L2正則化。学習率・エポック数・正則化係数は実装時に適切な値を選び、作業ログに記録すること。データの一部(例: 対局の最後10%)をホールドアウトとして分離し、学習に使わず検証用に残すこと。
- `train/src/bin/train_patterns.rs`(新規、CLIバイナリ) — `train/data/`配下のWTHORファイル群を読み込み、学習を実行し、学習済み重みを`train/weights/pattern_v1.bin`(新規ディレクトリ、コミット対象)に出力する。実行後、学習誤差(訓練誤差)とホールドアウト誤差(平均二乗誤差または平均絶対誤差)を標準出力に表示する。
- `train/weights/pattern_v1.bin`(新規、バイナリ形式) — 学習済み重みファイル。フォーマットは実装時に決めてよいが、後続タスク(engine/eval.rsへの統合)が読み込めるよう、フォーマット仕様を`train/weights/README.md`(新規、簡潔でよい)に明記すること。
- `.gitignore` — `train/weights/`は**除外しないこと**(重みファイルはコミット対象、ユーザー方針により生データのみ除外)。

## 要件

1. `train/src/patterns.rs`のパターン定義が、ハードコードされた個別セル番号の手書きリストではなく、ループ演算による機械的生成であること(コードレビューで確認可能な形にする)。
2. 学習に使うWTHOR対局データが、パラメータ数に対して明らかに不足しない規模であること(目安: 少なくとも数万局面規模。2年分(約32万手)で不足する場合は追加の年のデータをダウンロードして増やしてよい。ダウンロードしたデータは`train/data/`に置き、引き続きコミットしないこと)。
3. `train_patterns`バイナリを実行すると学習が完了し、`train/weights/pattern_v1.bin`が生成されること。
4. ホールドアウト誤差が、自明なベースライン(例えば「常に0(互角)と予測する」場合の誤差、または「常に訓練データの平均値を予測する」場合の誤差)より明確に小さいこと(学習が実際に信号を捉えていることの確認)。この比較結果を作業ログに数値で記録すること。
5. `cargo test --workspace`でパターン定義・学習ロジックのユニットテストが通ること。

## やらないこと(スコープ外)

- `engine/src/eval.rs`へのパターン評価統合、WASMビルド、アプリ側(`app/`)への反映は行わない(後続タスクで対応)。
- 対称性(回転・鏡映)による重み共有・パラメータ削減は行わない(v1は各パターンインスタンスが独立した重みを持つシンプルな設計。将来の改善候補として記録すればよい)。
- 隅3x3以外のコーナー系パターン(2x5等)、対角線以外の斜めパターン、短い対角線(長さ4-7)の追加は行わない。
- 学習アルゴリズムをSGD以外(バッチ最小二乗法、ニューラルネット等)に変更することは不要(SGDで十分)。
- Edaxとの比較検証は本タスクでは行わない(後続のengine統合タスクで、実際にプレイに使う段階になってから行う)。
