# T029: 棋譜解析モード(入力+解析パイプライン+評価グラフ+ナビゲーション拡張)

## 目的
設計書 `othello-trainer-design.md` §6「棋譜解析(評価グラフ・悪手分析)」のうち、§6.1「入力」・§6.2「解析パイプライン」・§6.3「評価グラフUI」を実装する。ユーザーが自分の(または任意の)対局棋譜を入力し、全手について評価値を解析、評価グラフで可視化できるようにする。悪手分析パネル(比較PV・フリー分岐探索・ヒューリスティック理由表示・練習送り)は次タスク(T030)で実装する。

## 背景・コンテキスト
- 前提: T012(Worker+WASM統合)・T018(`requestAnalyzeAll`)・T021(`app/src/db/appDb.ts`共通IndexedDBモジュール)・T028(詰めオセロ、4タブ構成)完了・コミット済み。
- 設計書§6.1「入力」:
  - 標準トランスクリプト(`f5d6c3d3c4...`、大小文字・区切り混在を許容)
  - WTHOR `.wtb`(ファイルドロップで複数局一括インポート)
  - オセロクエストのコピー棋譜 / 手動並べ(盤で再現 → 解析)
- **本タスクでのスコープ縮小**: WTHOR `.wtb`インポートは、バイナリフォーマットのパーサー実装コストに対し、本アプリの主目的(自分の対局を解析する)への寄与が薄いため**実装しない**(将来必要になれば別タスクで追加)。標準トランスクリプト入力(テキストボックス)と、盤面クリックによる手動並べの2方式を実装する。「オセロクエストのコピー棋譜」は、実際の書式が座標記法の羅列であれば標準トランスクリプト入力でカバーできる想定(実装者が確認し、明らかに異なる書式であれば対応可否を作業ログに記載する)。
- 設計書§6.2「解析パイプライン」:
  1. 全60局面をWorkerキューに投入。**終局側から解析**(空き≤22は完全読み、それ以前はdepth18)— 終盤の正確な値がグラフの信頼性を決めるため。
  2. 各手のロス = eval(着手前最善) − eval(実際の手)。手番視点の石差換算。
  3. 分類(閾値は設定可): ロス<1: 最善/準最善 ◎ / 1–3: 緩手 ?! / 3–6: 疑問手 ? / ≥6: **悪手 ??** / 勝敗が入れ替わった手: **逆転悪手**(赤強調)
  4. 解析結果は局面ハッシュでIndexedDBにキャッシュ(同一局面の再解析ゼロ)。
- 設計書§6.3「評価グラフUI」:
  - 横軸手数・縦軸石差(±16クリップ、**T024較正後の評価値スケールでは実際の石差に近いため、クリップ範囲は既存の`EvalBar`の定数を参考にしつつ実装者判断で調整してよい**)、黒優勢を上。完全読み区間は塗りを変えて「ここから確定値」を明示。
  - 悪手マーカー(?!/?/??)をグラフ上とムーブリスト両方に表示。タップでその局面へジャンプ。
- 「終局側から解析」の実装上の注意: 60手全てを最初から順番に解析すると、終盤(石差が正確)の情報が出るのが最後になり、グラフの信頼性の説明(設計書の意図)が損なわれる。実際の着手順とは逆順(最終局面→初手方向)に解析キューを処理すること。
- 既存の`app/src/db/appDb.ts`(T021, T028で`APP_DB_VERSION`管理を一元化済み)に、本タスクの解析結果キャッシュ用ストアを追加する。**T028の教訓により、バージョン管理は必ず`appDb.ts`側で一元的に行うこと(独自のバージョン定数を作らない)**。

## 要件
1. **入力画面**: テキストエリアへの標準トランスクリプト貼り付け、または盤面クリックによる手動並べのいずれかで棋譜を入力できる。
2. **トランスクリプトパース**: 大文字/小文字混在(`F5D6C3...`/`f5 d6 c3...`等の区切り文字混在)を許容し、`a1`〜`h8`の座標列に変換する。不正な文字列はエラー表示する。
3. **解析パイプライン**: 入力された着手列から全局面を再生し、**終局側から順に**各局面の評価値を解析する(空き22以下は完全読み、それより前はdepth18相当。既存`AnalyzeLimit`の使い方はT018/T020を参考にする)。解析中は進捗表示を行う。
4. **ロス計算・分類**: 各手について、着手前局面の最善評価値との差(石差)を計算し、閾値に基づき分類する(◎/?!/?/??/逆転悪手)。分類閾値はユーザーが調整できる設定UIを用意する(既存の`BlunderSettings`パターンを参考にしてよいが、閾値の意味が異なるため独立した設定として実装してよい)。
5. **キャッシュ**: 局面ハッシュ(T017の`hashBoard`等、既存ロジックを再利用可能なら再利用)をキーに解析結果をIndexedDBにキャッシュし、同一局面の再解析を避ける。
6. **評価グラフ**: 横軸=手数、縦軸=石差のグラフを表示する。完全読み区間とヒューリスティック区間を視覚的に区別する。悪手マーカーをグラフ上とムーブリスト上両方に表示し、クリックでその局面にジャンプできる。
7. **ナビゲーション拡張**: 5タブ構成(対局/定石練習/中盤練習/詰めオセロ/棋譜解析)にする。既存4モードの動作を壊さないこと。
8. **レスポンシブ**: 375px幅で崩れないこと(グラフの表示方法も含む)。
9. 単体テストで以下を検証する:
   - `parseTranscript`が様々な区切り文字・大小文字混在の入力を正しくパースすること、不正な入力を検出すること
   - `classifyMove`が閾値に応じて正しく分類すること(境界値のテストを含む)
   - キャッシュの読み書きが正しく動作すること(`fake-indexeddb`使用)
10. 実機確認: 実際にブラウザで棋譜解析モードを開始し、(a)トランスクリプト入力で解析が開始できること、(b)解析が終局側から進むこと(進捗表示で確認)、(c)評価グラフが表示され悪手マーカーが正しい位置に出ること、(d)グラフ/ムーブリストのクリックで局面ジャンプができること、(e)同じ棋譜を2回解析した際にキャッシュが効いて高速化すること、(f)375px幅でも崩れないこと、を確認し作業ログに記載する。

**探索条件について(重要な設計変更)**

当初、要件どおり`{ depth: 18, exactFromEmpties: 22 }`(時間予算なし)で実装したが、実機(Playwright)検証で**初期局面1手の解析だけで60秒超**かかることが判明した。`engine/src/search.rs`は`time_ms`未指定だと反復深化を`depth`まで時間無制限に進める仕様であり、開局面のように合法手の分岐が広い局面ではdepth18到達に非常に長い時間がかかる。60手全てを解析する本パイプラインでこれを放置すると、まさにタスクで警告されていた「完全読みハング事故」と同種のリスクになるため、`midgame/PracticeMode.tsx`の`MIDGAME_ANALYZE_LIMIT`(depth16, timeMs300)と同じ「反復深化+時間予算」方式に合わせ、`ANALYZE_LIMIT = { depth: 18, timeMs: 1500, exactFromEmpties: 22 }`とした(depth18はあくまで上限)。この変更後、1手あたり平均約2.5秒程度で解析が完了することを確認した。
