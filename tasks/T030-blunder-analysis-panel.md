# T030: 棋譜解析の悪手分析パネル(比較PV+フリー分岐探索+理由表示+練習送り)

## 目的
設計書 `othello-trainer-design.md` §6.4「悪手分析(「悪手以降の展開を調べる」)」を実装する。T029の棋譜解析モードで悪手をタップした際に開く分析パネルを追加する。

## 背景・コンテキスト
- 前提: T029(棋譜解析モード、`app/src/analysis/`一式・`AnalysisMode.tsx`)完了・コミット済み。T021(`app/src/midgame/pool.ts`、出題プール、`source: 'blunder-review'`が既に設計されている)・T018(`requestAnalyze`の`pv`フィールド)を再利用する。
- 設計書§6.4の4項目:
  1. **比較PV**: 「実際の手 → 相手最善の進行」と「最善手 → 進行」を8手ずつ、並走リストで表示。ステップ実行・分岐点での差分ハイライト。
  2. **フリー分岐探索**: 悪手局面から盤を直接操作して任意の変化を試せる。全ての試し手に即時評価が付き、**分岐ツリー**として右ペインに積み上がる(戻る・別分岐・本譜復帰が自由)。
  3. **「なぜ悪いか」ヒューリスティック表示**: 着手可能数差の推移、確定石数の変化、偶数理論違反(終盤)、X打ち・C打ちの検出を機械的に注記。
  4. **練習送り**: 悪手局面をワンタップで中盤練習(§4)/詰めオセロ(終盤なら)の出題プールへ登録。
- **本タスクでのスコープ縮小**:
  - 比較PVの「盤上矢印表示」は、T021同様テキスト/座標リストでの並走表示で足りる(Canvas矢印描画は必須ではない)。
  - ヒューリスティック理由表示は、機械的に判定しやすいもの(着手可能数差・確定石数変化・X打ち/C打ちの検出)に絞る。「偶数理論違反」は、判定ロジックが複雑なため、実装可能であれば追加してよいが、難しければ省略してよい(判断根拠を作業ログに記載)。
  - 練習送りのうち「中盤練習の出題プールへ登録」はT021の`app/src/midgame/pool.ts`(`addPoolEntry`、`source`フィールド)がまさにこの連携のために設計されているのでそのまま使う。「詰めオセロの出題プールへ登録」は、詰めオセロの問題は唯一解性・明確さフィルタを満たす必要があり(T027)、任意の悪手局面がこの基準を満たすとは限らない。**本タスムでは、悪手局面の空きマス数が20以下であれば、その場で全合法手を完全読みし、唯一解性・明確さフィルタ(T027の`app/src/tsume/filters.ts`のロジックを再利用)を満たす場合のみ即席の`Puzzle`としてプレイできるようにする(事前生成プールへの永続登録はスコープ外、その場でのプレイのみでよい)。満たさない場合は「この局面は詰めオセロとして出題するには条件を満たしません」と表示すればよい**。
- 分岐ツリーの状態管理は、React側でシンプルなツリー構造(ノード=局面、子=試した手)を保持する設計でよい。永続化(IndexedDB保存)は不要(セッション内のみ)。

## 変更対象(新規作成/変更)
- `app/src/analysis/BlunderPanel.tsx` + `.css`: 悪手分析パネル本体(比較PV・フリー分岐探索・理由表示・練習送りボタンをまとめた画面)
- `app/src/analysis/comparePv.ts`: 「実際の手の進行」と「最善手の進行」を並走リストとして構築する純粋関数(`requestAnalyze`の`pv`フィールドを使う)
- `app/src/analysis/whyBad.ts`: 着手可能数差・確定石数変化・X打ち/C打ち検出の純粋関数(悪手判定の理由テキストを生成)
- `app/src/analysis/branchTree.ts`: フリー分岐探索用のツリー構造・操作関数(ノード追加・移動・本譜復帰)

- `app/src/analysis/AnalysisMode.tsx`: 悪手マーカーのタップで`BlunderPanel`を開くよう統合

## 要件
1. **比較PVの表示**: 悪手局面から「実際に打たれた手→(以後の本譜進行、最大8手)」と「最善手→(最善進行、最大8手)」を並べて表示する。手番ごとにテキストで進行を表示すればよい(矢印描画は任意)。
2. **フリー分岐探索**: 悪手局面(または比較PVの任意の局面)から、実際に盤面をクリックして試し手を打てる。打つたびに即時評価(`requestAnalyzeAll`または`requestAnalyze`)を表示する。試した手は分岐ツリーとして記録され、ツリー上のノードをクリックすればその局面に戻れる。「本譜に戻る」ボタンで元の棋譜の進行に復帰できる。
3. **ヒューリスティック理由表示**: 悪手局面と着手後局面を比較し、着手可能数の変化・確定石数の変化・X打ち(隅隣接かつ隅が空いている手)/C打ちの検出結果をテキストで表示する。
4. **練習送り**: 「中盤練習に送る」ボタンで、その局面(board/turn)を`midgame/pool.ts`の出題プールに`source: 'blunder-review'`として登録する。「詰めオセロとして解いてみる」ボタンで、空きマス数条件を満たせばその場で唯一解性・明確さを判定し、満たせば即席のパズルとしてプレイできる画面に遷移する(満たさない場合はその旨を表示する)。
5. **統合**: T029の評価グラフ・ムーブリストの悪手マーカー(?!/?/??/逆転悪手)をクリックすると、`BlunderPanel`が開く(モーダルまたは画面内パネル、実装者判断)。
6. **レスポンシブ**: 375px幅で崩れないこと(分岐ツリーの表示方法も含む)。
7. 単体テストで以下を検証する:
   - `comparePv`が正しく並走リストを構築すること
   - `whyBad`が人工的な局面ペアに対して正しく着手可能数差・確定石数変化・X打ち/C打ちを検出すること
   - `branchTree`のノード追加・移動・本譜復帰が正しく動作すること
   - `sendToPractice`の中盤練習登録・詰めオセロ即席判定が正しく動作すること
8. 実機確認: 実際にブラウザで棋譜解析モードから悪手をタップし、(a)比較PVが表示されること、(b)フリー分岐探索で試し手を打つと分岐ツリーに追加され、ノードクリックで局面を行き来できること、(c)ヒューリスティック理由が表示されること、(d)「中盤練習に送る」で実際に中盤練習の出題プールにIndexedDB経由で登録されることを確認すること、(e)375px幅でも崩れないこと、を確認し作業ログに記載する。

### 実装内容
- `app/src/analysis/comparePv.ts`(+test): 悪手局面から「実際に打たれた手→本譜の以後の進行(最大8手)」と
  「最善手→PVによる進行(最大8手)」の並走リストを構築する純粋関数`buildComparePv`。両リストの重複区間で
  最初に手が食い違うインデックス(`firstDivergenceIndex`)と各インデックスの一致/不一致(`diverges`)も返す
  (分岐点ハイライト用)。エンジン呼び出し(`requestAnalyze`で最善手後のPVを取得)は`BlunderPanel.tsx`側の責務とし、
  本モジュールは純粋関数のまま。
- `app/src/analysis/whyBad.ts`(+test): 着手可能数差(着手前の自分の着手可能数 vs 着手後の相手の着手可能数)、
  確定石数の変化(`countStableDiscs`: 隅から固定点反復で安定集合を広げる簡易判定。全ラインが埋まっている、
  または両方向が盤端/既に安定な同色石に隣接、を条件とする。過小評価はあっても過大評価はしない設計であることを
  コード内コメントで明記)、X打ち/C打ち検出(隅がまだ空いている隅隣接マスへの着手)を計算し、人が読める理由
  テキスト配列(`reasons`)を返す純粋関数`analyzeWhyBad`。「偶数理論違反」はタスク仕様どおりスコープ外とした
  (判定ロジックが複雑なため。理由は元タスク仕様の記載どおり)。
- `app/src/analysis/branchTree.ts`(+test): フリー分岐探索用の不変(immutable)ツリー構造。`createBranchTree`
  (悪手局面をルートに)・`addBranchMove`(試し手を追加、既存の子があれば重複させず移動するだけ、非合法手は例外)・
  `goToNode`/`goToRoot`(ノード間の移動・本譜復帰)・`currentMover`(`midgame/resolveMover.ts`でパス自動処理)・
  `setNodeEval`(ノードへの即時評価付与)を提供。永続化はしない(タスク仕様どおりセッション内のみ)。
- `app/src/analysis/sendToPractice.ts`(+test): `sendToMidgamePractice`は`midgame/pool.ts`の`addPoolEntry`を
  `source: 'blunder-review'`で呼ぶだけの薄いラッパー(既存のIndexedDB一元管理`db/appDb.ts`をそのまま経由、
  新規ストアは追加していない)。`buildInstantTsumePuzzle`は空きマス数が`MAX_INSTANT_TSUME_EMPTIES`(20)を
  超える場合はエンジンを一切呼ばずに却下し(完全読みハング対策、`tsume/PlayMode.tsx`の`puzzleAnalyzeLimit`と
  同じ`{depth: empties, exactFromEmpties: empties}`を空き20以下でのみ使う設計を踏襲)、それ以外は全合法手を
  完全読みして`tsume/filters.ts`の`analyzeMoveOutcomes`(唯一解性・明確さ)で判定し、満たせば`Puzzle`相当の
  データを構築する(永続登録はしない)。
- `app/src/analysis/BlunderPanel.tsx`(+css): 上記4モジュールを統合したモーダルパネル。
  1. 着手前局面・実際の手/最善手の評価バッジ表示
  2. 比較PV(ロード中表示・分岐ハイライト付き)
  3. なぜ悪いか(`whyBad`の理由テキストのリスト表示)
  4. フリー分岐探索(`Board`コンポーネントで試し手を打つと`branchTree`に追加され即時評価を取得、
     再帰コンポーネント`BranchTreeView`でツリー表示、ノードクリックで移動、「本譜に戻る」ボタン)
  5. 練習送り(「中盤練習に送る」ボタン、「詰めオセロとして解いてみる」ボタン→受理なら`tsume/PlayMode.tsx`と
     同様のミニ対局ループ(`judgePuzzleMove`・相手は「最も粘る手」を自動着手)をパネル内に埋め込み、
     却下なら理由文を表示)
- `app/src/analysis/AnalysisMode.tsx`: ムーブリストの分類セル(悪手マーカー: 分類が◎以外、または逆転)を
  ボタン化し、クリックで`BlunderPanel`を開く(`key={ply}`で悪手を切り替えるたびに再マウントし、パネル内部の
  state をリセット)。`gameMoves`は`results.map(m => m.move)`から再構成(棋譜全体を保持する専用stateは追加不要)。
- `app/src/analysis/EvalGraph.tsx`: 「変更対象」ファイル一覧には無かったが、要件5(評価グラフの悪手マーカー
  クリックでもパネルを開く)を満たすため、既存の`onSelectPly`とは別に任意の`onMarkerClick`propを追加した
  (省略時は従来どおり`onSelectPly`にフォールバックする後方互換設計、既存呼び出し元は無変更で動作)。
  小さな統合用の変更のため直接編集した。
