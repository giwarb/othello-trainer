# T062: Service Workerの更新伝播を修正(デプロイ後も古いアプリ本体が表示され続ける不具合)

## 目的

ユーザー報告: T059(評価値スケール異常の修正)をデプロイ・本番確認済みと報告したにもかかわらず、ユーザーが実際に本番URLにアクセスすると、修正前の異常値(石差の理論上限を超える値)がまだ表示された。調査の結果、Service Workerの更新伝播に根本的な欠陥があることが判明した。これは今回に限らず、過去にユーザーが指摘した「デプロイしても結果が変わらない」という問題(T060で対応したのはIndexedDBの解析結果キャッシュのみ)の、より深刻な形の再発である。**通常のユーザーは、手動でブラウザのキャッシュをクリアしない限り、新しいデプロイ内容を見られない可能性がある。**

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように、調査済みの根本原因)

- `app/public/sw.js`のfetchハンドラ(105行目以降)は、**ページ本体(`index.html`、`request.mode === 'navigate'`)も含めて全リクエストをcache-first戦略で処理している**(127行目`caches.match(request, { ignoreVary: true })`をまず試し、無ければネットワーク取得)。つまり、一度アプリシェルがキャッシュされると、新しいデプロイがあっても、次回訪問時にまず古いキャッシュ済みのindex.htmlが返され続ける。
- `app/src/registerServiceWorker.ts`は`navigator.serviceWorker.register(swUrl)`を呼ぶだけで、新しいService Workerが利用可能になったことを検知してユーザーに通知・リロードを促す仕組み(`updatefound`イベント、`controllerchange`イベントのリスニング)が一切実装されていない。
- `sw.js`自体は`self.skipWaiting()`(87行目)・`self.clients.claim()`(100行目)を呼んでおり、新しいService Worker自体は比較的早く有効化・クライアントの制御を奪う設計にはなっているが、**既に開いているタブのJavaScript実行内容(メモリ上のコード)は新しいService Workerが有効化されても自動的には入れ替わらない**。ページの再読み込みが必要だが、その再読み込みを促す仕組みが無いため、ユーザーは気づかない限り古いアプリを使い続けることになる。
- **重要な検証プロセス上の教訓**: これまでの各タスク(T057〜T061等)の本番確認では、実装者・verifierが「本番URLでPlaywrightにより確認する際、事前にIndexedDB・Cache Storageを手動でクリアしてから確認する」という手順を踏んでいた。これは実質的に「通常のユーザー体験では発生しうる更新伝播の問題を、検証時に毎回手動で回避していた」ことを意味し、この問題自体を検知できていなかった。今後の本番確認では、**キャッシュを事前クリアしない「素の再訪問」のシナリオでも動作確認する**ことを検証手順に含めるべきである。

## 変更対象

- `app/public/sw.js` — fetchハンドラの戦略を見直す。少なくとも、ナビゲーションリクエスト(`request.mode === 'navigate'`、実質的に`index.html`)については、**ネットワークを優先し、失敗時(オフライン時)のみキャッシュにフォールバックする**戦略に変更する(オフライン動作の維持と、最新版の即時反映を両立させる)。ハッシュ付きの静的アセット(JS/CSS/WASM、ビルドごとにファイル名が変わるため内容が変わらない限りキャッシュして問題ない)は引き続きcache-first(または現状の戦略)のままでよい。
- `app/src/registerServiceWorker.ts` — 新しいService Workerが検出・インストールされた際に、ユーザーに更新を通知するか、自動的にページをリロードする仕組みを追加する。具体的な実現方法(自動リロード vs 「新しいバージョンがあります、更新する」というプロンプト表示)は実装時に判断してよいが、**無限リロードループを起こさないようガードすること**、**対局中・練習中にユーザーが気づかないうちに突然リロードされてゲーム進行状況を失うことがないよう配慮すること**(例えば、対局中は通知のみ表示し強制リロードはしない、または次回のページ遷移時に適用する等)。

## 要件

1. 新しいデプロイが行われた後、ユーザーが通常の操作(ページの再読み込み、再訪問)を行った際に、古いキャッシュされたアプリ本体(index.html・JS・WASM)ではなく、新しいバージョンが表示されるようになること。
2. オフライン時は引き続き動作すること(PWAのオフライン対応を壊さないこと)。
3. 対局・練習の途中で突然リロードされてユーザーの作業内容が失われることがないよう配慮すること。
4. 既存のテストが壊れないこと。
5. 今後の本番確認手順として、**キャッシュを事前クリアしない状態での動作確認**を検証項目に加えること(作業ログに、この観点での確認結果を記録すること)。

## やらないこと(スコープ外)

- 解析結果キャッシュ(IndexedDB、T060で対応済み)自体の変更は行わない。
- PWAのマニフェスト・アイコン等の見た目の変更は行わない(別タスク)。
