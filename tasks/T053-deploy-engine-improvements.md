# T053: エンジン強化(ETC・終盤パリティ)をWASM再ビルドで本番反映

## 目的

T048(MPC、既定オフ)・T051(ETC)・T052(終盤パリティ着手順序付け)はいずれも`engine`クレート(Rust)のみの変更で、各タスクの受け入れ基準にWASM再ビルド・GitHub Pagesへのデプロイが含まれていなかった。そのため、これらの改善はまだ本番公開中のアプリには反映されていない(直近の本番デプロイはT050時点のコミット)。`app/`のビルドプロセスは`engine/src`から都度WASMを再ビルドする設計(`prebuild`スクリプトで`wasm-pack build`が走る)のため、`app/`側で何もコード変更をしなくても、ビルド・デプロイを実行するだけでET C・終盤パリティ順序付けが本番に反映される。これを実施し、実際に問題なく動作することを確認する。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

- `app/src/engine/pkg/`(wasm-pack生成物)は`.gitignore`対象で、ビルド時に`engine/src`から自動再生成される(コミットされた古い成果物ではない)。つまり`app/`側で何もソース変更をしなくても、`npm run build`(またはCI上のビルド)を実行すれば、現在の`engine/src`(ETC・終盤パリティ導入済み、MPCは既定オフのまま)を反映したWASMが生成される。
- ETC・終盤パリティは「探索結果を変えない安全な高速化」として検証済み(T051・T052でFFO正解値の完全一致を確認済み)なので、本番反映によってゲームプレイの挙動(着手選択・評価値)が変わることは想定されない。純粋に探索速度(同一時間予算内での読みの深さ)が向上する。
- 過去の教訓(T034): 探索の時間管理まわりの変更はWASM環境固有の問題(ネイティブでは再現しない)を引き起こした実績がある。ネイティブの`cargo test`で検証済みであっても、**実際にWASMビルド・ブラウザ上での動作確認を必ず行うこと**。

## 変更対象

- `app/`配下のソースコードは変更不要(このタスクは「ビルド・デプロイの実行」が本体)。
- 何かのファイルにダミーの変更を加える必要は無い。`git commit --allow-empty`等でトリガーするか、CI設定上デプロイに必要な最小限の操作を行うこと(具体的な実施方法は実装時に判断してよい)。

## 要件

1. 現在の`main`ブランチ(T052まで反映済み)から、GitHub Actionsのデプロイを実行させ、本番URL(`https://giwarb.github.io/othello-trainer/`)に反映させること。
2. 本番環境で実際に対局を行い、着手・CPU応答が正常に機能すること、コンソールエラーが出ないことを確認する。
3. 詰めオセロ・棋譜解析等、終盤完全読みを使う機能が正常に動作すること(結果の正しさに変化が無いことを念のため確認する)。
4. WASMビルド・ロードにかかる時間や、対局中のCPU応答速度に明らかな異常(遅すぎる、フリーズする等)が無いことを確認する(T034のような事故の再発が無いことの実機確認)。

## やらないこと(スコープ外)

- エンジンのコード自体の変更は行わない(既に完了しているT048/T051/T052の反映のみ)。
- MPCを有効化することは行わない(既定オフのまま)。
