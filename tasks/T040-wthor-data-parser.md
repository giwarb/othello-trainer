# T040: フェーズ3着手(1): WTHOR棋譜データ取得 + パーサー実装(train/クレート新設)

## 目的

ユーザー承認により、評価関数をWTHOR棋譜データによるパターン評価学習(フェーズ3、設計書§2.14の後回しにしてきた項目)に本格着手する。その第一歩として、WTHOR形式の棋譜データを実際に取得し、正しくパースできるパーサーをRustで実装する(学習・特徴量抽出・エンジン統合は後続タスクで行う)。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

- 現状の評価関数(`engine/src/eval.rs`)は手作りの軽量線形評価(モビリティ・隅・安定石の3項、T024でEdax較正済み)。設計書上、本格的なパターン評価(WTHOR学習)は「段階的フル実装方針」でフェーズ3として後回しにされてきた。ユーザーが今回、本格着手を指示した。
- リポジトリ構成は`/engine` `/train` `/bookgen` `/puzzlegen` `/app` `/bench` `/.github`を想定しているが、`/train`は現時点で**存在しない**(本タスクで新設する)。ルートの`Cargo.toml`は`[workspace] members = ["engine"]`のみ(1行目)。`train`を新規crateとしてworkspaceに追加すること(`members = ["engine", "train"]`)。
- **WTHORデータの入手可否(調査済み)**: フランスオセロ連盟(FFO)公式サイト https://www.ffothello.org/informatique/la-base-wthor/ で現在も無料配布中(累計137,548局、年ごとに`WTH_[年].wtb`または圧縮版`WTH_[年].ZIP`)。**利用規約・再配布可否の明記は一切ない(グレーゾーン)**。ユーザー方針により、**生のWTHORデータファイル自体はリポジトリにコミットしない**(`.gitignore`で除外)。学習によって生成される重みファイルのみを成果物としてコミットする(重みは学習済みパラメータのみで元の棋譜を復元できないためライセンスリスクは低いという判断)。
- **WTHORファイルフォーマット(調査済み、概ね確認)**: 全ファイル共通で16バイトヘッダ(作成日の世紀/年/月/日、レコード数N1/N2、対象年、盤サイズP1、種別P2、深さP3、予約X1)+固定長ゲームレコードの並び。ゲームレコードは大会番号・黒白選手番号・黒石数(実測スコア)・理論スコア・手順(着手ごとに1バイト、`列+10*行`エンコード。例: a1=11, h1=18, a8=81, h8=88)。全体リトルエンディアン。**1ゲームレコードの正確なバイト長・終端条件は未確認**。実装時は一次情報として `https://github.com/LimeEng/wthor` (Rust実装のWTHORパーサー、構造体定義・レコード長が明記されているはず)のソースコードを直接参照し、正確な仕様を確認してから実装すること(参考記事: http://ledpup.blogspot.com/2012/03/computer-reversi-part-15-thor-database.html も補助的に使ってよい)。
- 付随ファイル`.JOU`(選手名簿)・`.TRN`(大会名簿)は本タスクでは扱わなくてよい(対局の手順さえパースできれば学習には十分)。
- 既存のRustワークスペース(`engine/`)のビルド・テスト方法は`cargo build` / `cargo test`(ワークスペースルートで実行)。`train`クレートも同様の作法に従うこと。
- 盤面・座標表現は`engine/src/`の既存のビットボード実装(`bitboard.rs`等、正方形64マスの座標系)と揃えること。WTHORの「列+10*行」記法(a1=11など)から、エンジン内部の座標(0-63、詳細は`engine/src/bitboard.rs`または`engine/src/notation.rs`相当のファイルを確認)への変換関数を用意する。

## 変更対象

- ルート`Cargo.toml` — `members = ["engine", "train"]`に変更。
- `train/Cargo.toml`(新規) — 新規Rust binクレート(またはlib+binの構成)。`engine`クレートを依存に加え、盤面表現・合法手判定ロジックを再利用できるようにする。
- `train/src/wthor.rs`(新規) — WTHORファイルパーサー。`.wtb`ファイルのバイト列を読み、ヘッダ情報+各ゲームの手順(着手マス列、黒石数、理論スコア等)を構造体として返す関数を実装する。
- `train/src/main.rs`(新規) — CLIエントリポイント。指定したWTHORファイル(または複数年分のディレクトリ)を読み込み、パース結果のサマリ(総ゲーム数、総手数、パース失敗件数、先頭数ゲームのサンプル手順)を標準出力に表示する。
- `train/data/`(新規ディレクトリ) — ダウンロードしたWTHOR生データの格納先。**`.gitignore`に`train/data/`を追記し、コミット対象から除外すること**。
- `.gitignore` — `train/data/`を追記。

## 要件

1. FFO公式サイト(https://www.ffothello.org/informatique/la-base-wthor/ )から、少なくとも直近1〜2年分のWTHORデータ(`.wtb`または解凍後の`.wtb`)を実際にダウンロードし、`train/data/`配下に配置する(コミットはしない)。
2. `train/src/wthor.rs`が、ダウンロードしたファイルのヘッダを正しくパースし(レコード数・対象年等がファイル内容と整合すること)、各ゲームの手順を「着手マスの列」として復元できること。
3. パースした手順が実際に合法なオセロの着手列になっていることを検証する(例: `engine`クレートの合法手判定ロジックを使い、各ゲームの手順を先頭から再生して「常に合法手のみが指されている」ことを確認するテスト、またはCLIでのサマリ表示で異常な手が無いことを目視確認できるようにする)。
4. `train/src/main.rs`を実行すると、パースしたゲーム数・総手数等のサマリが表示され、少なくとも数百〜数千ゲーム規模のデータが正しく読めていることを確認できること。
5. `cargo test`(ワークスペースルート、`train`クレート含む)でWTHORパーサーのユニットテスト(ヘッダパース・手順デコード・座標変換)が通ること。

## やらないこと(スコープ外)

- パターン特徴量の設計・抽出、回帰学習、重みファイルの生成は行わない(後続タスクで対応)。
- `engine/src/eval.rs`の変更は行わない(既存の3項評価はそのまま)。
- WASMビルド・アプリ側(`app/`)への統合は行わない。
- `.JOU`(選手名簿)・`.TRN`(大会名簿)ファイルのパースは行わない。
- WTHORデータのライセンスについてFFOへの問い合わせは行わない(ユーザー方針により「生データはコミットしない」運用で進める)。
- 複数年分すべて(137,548局全件)のダウンロード・パースは必須ではない(まずは1〜2年分で十分。多いほど良いが必須要件ではない)。
