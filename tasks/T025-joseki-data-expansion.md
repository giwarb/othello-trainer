# T025: 定石データの大幅拡充(既存35ラインでは網羅性不足、虎系のローズビル・酉等を含め収集)

## 目的
ユーザーが実際に定石練習モード(T020)を遊んだところ、「定石が少なすぎる。虎系統でローズビルや酉に進行しないのはおかしい」という指摘を受けた(2026-07-08)。T016で収集した`bookgen/joseki-research.json`は35ラインのみで、実在するオセロの定石体系(虎・牛・兎系統から派生する多数の名前付きライン)を十分に網羅できていない。本タスクでは、公開されている定石名鑑・戦術wiki等からより広範囲に定石データを追加収集し、DAG(T017)を再構築する。

## 背景・コンテキスト
- 前提: T016(`bookgen/joseki-research.json`)・T017(`app/src/joseki/buildDb.ts`・`app/public/joseki.json`)・T020(定石練習モード)完了・コミット済み。
- **重要な制約(T016の教訓)**: T016では当初、出典URLに実際には存在しないデータを記載してしまう「データ捏造」問題が2回発生し、やり直しを経てcurl+grepによる文字列レベルの検証で解消した経緯がある(詳細は`tasks/STATUS.md`のT016メモ・`tasks/T016-joseki-data-research.md`参照)。**本タスクでも同じ過ちを繰り返さないこと**: 「この名前の定石が存在するはず」という記憶・推測だけで着手列を書かない。必ず実際にWebページを取得し(`curl -A "Mozilla/5.0" <URL>`等)、文字列レベルで着手列・名称を確認してから`joseki-research.json`に追記すること。WebFetchツールの内部AI要約は事実確認には使わない(過去に不正確な要約が誤ったデータを生んだ実績があるため)。
- 現在のデータ(35ライン)の内訳: 虎/猫/羊/虎C/虎D/虎E/兎/馬/野兎/縦取り(f5-d6系統)、牛/牛A/牛B/バッファロー/闘牛/蛇/狸/闘牛C/暴走牛/斜め取り/並び取り(f5-f6系統)、野ウサギ定石(3種)、ローズ基本形/シャープローズ(2種)/フラットローズ(2種)/手塚システム/こうもり定石/戦車定石/Sローズ(3種)。**ユーザーの指摘通り、虎(f5-d6-c3-d3-c4)からさらに深く進む「ローズビルド」系統や、"酉"を含む系統など、実在する有名な定石ラインが欠落している。**
- 出典として過去に使用した実績のあるサイト: `bassy84.net`、`hasera.net`(`hasera.net/othello/jouseki-tate.html`、`saizen40.html`、`jouseki-naname.html`、`jouseki-narabi.html`等の複数ページに定石データが分散している)。これら以外にも、公開されている定石名鑑・オセロ戦術wiki・攻略サイトがあれば広く調査すること(サイト名を事前に指定しない。実装者が実際に検索・調査して見つけること)。
- 定石DB方針(2026-07-08ユーザー承認、CLAUDE.md記載): オセロクエストのデータを直接複製せず、公開されている定石名鑑・戦術wiki等から同じ定石体系(兎・牛・虎など)を独自に調査・データベース化する。本タスクでもこの方針を継続する。

## 変更対象(変更/新規作成)
- `bookgen/joseki-research.json`: 既存35ラインに加え、追加のラインを収集・追記する(既存エントリは正しければ変更不要。誤りがあれば修正してよいが、その場合は理由を作業ログに明記する)
- `app/src/joseki/generate.ts`(既存): 変更不要な想定だが、データ拡充後に再実行して`app/public/joseki.json`を再生成する
- `app/public/joseki.json`: 再生成(コミット対象)
- 拡充後のデータに対するテスト調整(`app/src/joseki/buildDb.test.ts`等、ライン数を前提にしたテストがあれば更新)

## 要件
1. **追加収集の観点**: ユーザーが名指しした「虎系統のローズビルド」「酉」を最優先で調査すること。加えて、既存収集済みの系統(虎・牛・兎)についても、公開情報からさらに深い/派生した進行が見つかれば追加する。目安として、**最終的に合計70〜100ライン程度**(現状35ラインの倍以上)を目指す。ただし件数を追うために不正確なデータを混ぜないこと。実際に裏付けが取れた範囲で構わない(信頼できる情報源が尽きたなら、その時点の件数で構わない。件数はあくまで目安)。
2. **データ形式**: T016の`bookgen/joseki-research.json`の既存スキーマ(`name`, `moves`, `firstMoveBasis`, 出典情報等)をそのまま踏襲する。スキーマの詳細は既存ファイルを実装者自身で確認すること。
3. **検証手法(必須、T016の教訓を厳守)**: 追加する各ラインについて、実際に取得したページの生テキスト(curl等で取得したHTML/テキストをgrep等で検索)から、着手列・名称が実在することを確認する。WebFetchの要約のみに頼った追記は禁止。既存の`tasks/T016-joseki-data-research.md`の作業ログに記載された検証手法・スクリプトを参考にしてよい。
4. **座標記法の統一**: 既存データと同じ記法(`a1`〜`h8`、初手`f5`基準)で統一する。初手が`f5`以外のソースから収集した場合は、T017の`app/src/joseki/symmetry.ts`/`normalize.ts`のロジックで`f5`基準に変換してから記録する(既存データの慣習に合わせる)。
5. **DAG再構築**: データ追加後、`cd app && npm run joseki:build`を実行して`app/public/joseki.json`を再生成する。合流(複数ラインが同じ局面を経由する)が正しく機能しているか、`app/src/joseki/buildDb.test.ts`のテスト(または新規テスト)で確認する。
6. **重複・矛盾のチェック**: 同じ着手列に異なる名前が付けられていないか、逆に同じ名前が異なる着手列に付けられていないか(データ入力ミス)を確認する。
7. 実機確認: `npm run dev`で定石練習モードを起動し、拡充後のデータで複数回セッションを行い、以前は到達できなかった系統(虎から分岐する新規ライン等)に到達できることを確認する。
