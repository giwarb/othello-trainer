# T032: 言語化支援 モチーフ検出タグ + 盤面可視化オーバーレイ

## 目的
`othello-trainer-design-verbalization.md` §4「モチーフ検出タグ」・§5「可視化オーバーレイ」を実装する。T031の特徴量層(12特徴量)・評価内訳分解層の出力を使い、着手を人間になじみのある概念タグ(中割り・全返し・壁作り等)で注記し、盤面上に該当マスをハイライト表示する。

## 背景・コンテキスト
- 前提: T031(特徴量層・評価内訳分解層、`engine/src/explain.rs`・`app/src/analysis/attribution.ts`)完了・コミット済み。`app/src/analysis/whyBad.ts`(T030)がX打ち/C打ち検出の前身として既に実装済み。
- 設計書§4「モチーフ検出タグ」の正確なリスト(`othello-trainer-design-verbalization.md`より引用):
  - **良い手系(9種)**: 中割り / 引っ張り(相手の好手を消す) / ブロック / 種石作り / 手止まり確保 / 余裕手の温存 / 辺の先着 / 犠牲打(一時損で偶数や隅を確保) / 通し
  - **悪い手系(8種)**: 全返し(開放度過大) / 壁作り / 手損 / X打ち(無根拠) / C打ち(無根拠) / 種石供給(相手の辺打ちを成立させてしまう) / 偶数放棄 / 自滅(自分の手数を自分で消す)
  - **罠系(3種、専用検出器)**: ストナー(成立条件: 辺の形+X筋の種石を厳密判定) / 爆弾 / 隅絡みの一方通行
  - タグの利用先: 悪手分析の注記、中盤練習のフィードバック、言語化トレーニング(T034で実装予定)の選択肢、弱点統計(T035で実装予定)
- **重要な制約(必ず理解してから着手すること)**: 「爆弾」「隅絡みの一方通行」の2種は、設計書内に検出条件の記載が一切ない(名称のみ)。**これらを推測・創作で実装しないこと**(このプロジェクトではT016の教訓により、根拠のないデータ・ロジックの捏造を厳しく禁じている)。実装しない場合はその旨を「やらないこと」として明記し、作業ログにも理由を記載すること。
- T031の特徴量との対応関係(実装の指針、必須ではなく参考):
  - 中割り: 開放度(特徴量3) ≤ 2
  - 全返し: 開放度(特徴量3)が過大(閾値は実装者判断)
  - 壁作り: フロンティア石数(特徴量4)が増加
  - 手損: 着手可能数差(特徴量1)が悪化
  - X打ち/C打ち(無根拠): T030の`whyBad.ts`のロジックをそのまま再利用(「無根拠」の判定は、隅を取り返せる/取り返せない等の文脈判定が必要なら実装者判断で追加してよい)
  - 種石作り/種石供給: 種石(特徴量11)
  - 通し: ライン(特徴量12)
  - 偶数放棄: 地域偶数(特徴量9)
  - ストナー: 辺の形(特徴量7)+種石(特徴量11)の組み合わせ
  - ブロック・引っ張り・手止まり確保・余裕手の温存・辺の先着・犠牲打・自滅: 明確な機械的定義が設計書に無いため、実装者が特徴量の変化パターンから合理的な近似ロジックを設計してよい(判断根拠を作業ログに必ず記載すること。無理に定義できないものは省略してよい)。

## 要件
1. **モチーフ検出**: 上記リストのうち、設計書に検出条件の記載がある/合理的に近似できるモチーフを実装する(「爆弾」「隅絡みの一方通機」を除く)。各モチーフは、着手前後の局面と特徴量セットを入力に、該当するかどうかの真偽値(またはタグ文字列の配列)を返す純粋関数として実装する。
2. **タグの表示**: `BlunderPanel`(T030)で、着手に該当するモチーフタグをバッジ・ラベル形式で表示する。
3. **盤面オーバーレイ**: 少なくとも以下を盤面上に視覚的に表示できるようにする: フロンティア石(特徴量4)・確定石(特徴量6)・種石(特徴量11)・X/C打ちで危険なマス(特徴量8)。表示するオーバーレイの種類を切り替えられるUIを用意する(チェックボックス等、シンプルでよい)。
4. **統合**: `BlunderPanel`にモチーフタグ+盤面オーバーレイを統合する。
5. **レスポンシブ**: 375px幅で崩れないこと。
6. 単体テストで、各モチーフ検出関数が人工的な特徴量データに対して正しく判定することを検証する。
7. 実機確認: 実際にブラウザで棋譜解析→悪手分析パネルを開き、(a)モチーフタグが表示されること、(b)盤面オーバーレイの切り替えで該当マスがハイライトされること、(c)375px幅でも崩れないこと、を確認し作業ログに記載する。

  **実装したモチーフ(15種)と判断根拠**:
  - 中割り(good): `FeatureSet.isUchiwari`をそのまま使用(特徴量そのもの、判断余地なし)
  - X打ち/C打ち・無根拠(bad×2): `whyBad.ts`(T030)の`analyzeWhyBad`をそのまま再利用(タスク指示通り)。追加の「無根拠」文脈判定は行わず、隅が空きというT030の既存条件のみを使用(スコープを絞った判断)
  - 種石供給(bad): `FeatureSet.seedStones.length >= 1`(特徴量そのもの)
  - 通し(good): `FeatureSet.lines`のいずれかで相手石0かつ自分石4以上(閾値4=8マス中の過半数、実装者判断)
  - ブロック(good)/手損(bad): `FeatureSet.mobilityDiff`の閾値+4/-4(対称な実装者判断の閾値)
  - 全返し(bad): `FeatureSet.openness >= 6`(中割りの閾値2の倍を「過大」とみなす実装者判断)
  - 壁作り(bad): 自分のフロンティア石数(TS側で独自計算、`frontierSquares`)が着手前後で+2以上増加(閾値2=着手自体による不可避な+1を除いた「明確な増加」、実装者判断)。単体テストで delta=+2(検出)とdelta=-2(非検出)の両方を手計算で検証済み
  - 種石作り(good): `FeatureSet.seedStones`の逆方向(相手の石が自分の将来の辺打ちで挟み返せるか)をTS側で`compute_seed_stones`と同アルゴリズムで独自計算(`computeVulnerableSeeds`)。Rust側は片方向しか計算していないため
  - 辺の先着(good): 着手先が辺のマスで、着手前その辺8マス全てが空き(=最初の着手)。機械的で判断余地の小さい定義
  - 引っ張り(good): 相手が着手前に持っていたX打ち/C打ちの合法手(隅が空き)が着手後に失われた場合に検出。「相手の好手」を一般的に定義するのは困難なため、機械的に判定できる対象(X/C打ち)に絞った実装者判断の近似
  - 偶数放棄(bad): 着手前局面で空き領域が2つ以上に分かれており、着手先が「最大かつ偶数」の領域に属し、他に選択肢があった(強制手でない)場合に検出。厳密な偶数理論(残り空き全体のパリティ等)はスコープ外とし、「最大の偶数領域への不必要な着手」に単純化(実装者判断)
  - 自滅(bad): `FeatureSet.lostOwnMoves.length >= 2`(1個の消失はありふれているため閾値2、実装者判断)
  - ストナー(trap): 設計書のヒント「辺の形+X筋の種石を厳密判定」を文字通り機械的に組み合わせ(`edgeShapes`のwing形状+対応するX打ちマスが`seedStones`に含まれる)。コード内コメントに明記した通り、伝統的な「ストナーの罠」の正確な成立条件の再現を主張するものではなく、設計書のヒントの機械的翻訳である

  - 犠牲打(good): 「一時損と引き換えの後続獲得」の判定には数手先の実現確認が必要で、1手先の特徴量差分だけでは通常の石損と区別できない。誤検出リスクが高いと判断し省略
