# T022: 評価関数の妥当性検証(Edaxとの比較基盤構築+初回比較)

## 目的
ユーザー指示(2026-07-08):「評価値に関しては、ちゃんと意味のある値になっているか、ほかのオセロAIの実装(Edax?)と必ず比較しておいてほしい」への対応。本エンジン(`/engine`)の評価値(特に中盤ヒューリスティック評価。終盤は完全読みでありT009でFFO問題の公式正解値と照合済みのため優先度は相対的に低い)が、既知の強豪オセロAI実装であるEdax(https://github.com/abulmo/edax-reversi)と比較して「意味のある値」になっているかを検証する比較基盤を構築し、初回の比較結果をレポートする。

## 背景・コンテキスト
- 本エンジンの評価関数(`engine/src/eval.rs`相当。T004で実装)は、モビリティ・安定石・隅・パリティの手作り軽量パターン評価であり、Edaxのような機械学習済みパターン評価(数百万局の自己対戦から学習した特徴量テーブル)とは設計思想が異なる。**完全に同じ値になることは期待しない**。本タスクの目的は「大きく間違っていないか(符号が逆、明らかな悪手を良い手と評価する等)」の妥当性チェックであり、Edaxとの完全一致を求めるものではない。この前提を作業ログ・報告に明記すること。
- 終盤(空きマス数が少なく完全読みが効く局面)の正しさは、T009でFFO endgame test(#40-48)の公式正解値と照合済みであり、これは「客観的な正解」との比較であるためEdaxとの比較より強い保証である。本タスクは主に**中盤ヒューリスティック評価(完全読みが及ばない局面でのdepth制限探索の評価値)**の妥当性確認に重点を置く。
- Edaxの入手方法(優先順): 
  1. **公式GitHubリリースページ(https://github.com/abulmo/edax-reversi/releases)の配布済みWindowsバイナリを使う**(ビルド不要で最も確実)。ダウンロードしたファイルは改変せず、取得元URLを作業ログに記録する。
  2. リリースにWindows向けビルドが無い、または動作しない場合は、ソースからのビルドを試みる(Edaxは標準的なC言語プロジェクト。Windows上でのビルドにはMSYS2/MinGWまたはWSLが必要になる可能性が高い。開発機の環境(このタスク実行時点でgcc/make/MSYS2/WSLが利用可能か)をまず確認し、無ければインストール要否をユーザーに確認してから進める、または探索的に別の実行方法(Docker等、利用可能なら)を検討する)。
  3. 上記いずれも困難な場合、Edax以外の公開されている検証済みオープンソースオセロエンジン(例: Zebra等、由緒あるもの)で代替してよいが、その場合は理由と代替エンジン名を作業ログに明記し、タスクファイルのタイトル趣旨(著名な強豪エンジンとの比較)を満たすことを確認する。
  4. どの方法でも実行環境の確保が困難だと判断した場合は、実装を進めず状況を整理してユーザーに相談すること(無理に不正確な代替手段(例: 架空の比較データ)を作らないこと。T016のデータ捏造問題の教訓を参照)。

## 変更対象(新規作成)
- `bench/edax-compare/`(ディレクトリ新規作成): 比較用スクリプト・データ一式
  - `positions.json`(または類似): 比較対象局面のセット。以下を含めること:
    - 初期局面から数手進めた代表的な序盤局面(例: 5〜10局面)
    - ランダムまたは自己対戦で生成した中盤局面(空きマス20〜40程度の範囲で、例: 20局面程度)
    - T009で使用したFFO問題の局面(既存の完全読み結果と3者比較(本エンジン完全読み・Edax・FFO公式値)できると理想的だが、必須ではない)
  - `run-comparison.*`(PowerShellまたはNode/Pythonスクリプト、実装者判断): 各局面について、(a) 本エンジン(WASMではなくネイティブ`cargo run`または既存のCLI/テストハーネスがあればそれを使う。無ければ簡易CLIを追加してもよい)で評価値を取得、(b) Edaxで同一局面・同等の探索深さ(depth)で評価値を取得、(c) 結果を突き合わせて比較表を出力する
  - `report.md`(または類似): 比較結果のレポート。各局面ごとの本エンジン評価値・Edax評価値・符号一致有無・大きな乖離の有無、および全体の考察(妥当と判断できるか、要調整点があるか)をまとめる
- 本エンジン側に比較用の局面評価CLIが不足している場合、`engine/src/bin/`配下等に簡易的な評価値出力バイナリを追加してよい(既存の`cargo test`/wasm APIとは別に、局面(着手列 or FEN的表記)を渡すと評価値を標準出力するだけの最小限のツール)

## 要件
1. 上記「背景」のEdax入手方法を優先順に試し、実際に動作するEdax実行環境を用意する。入手・ビルド手順を作業ログに再現可能な形で記載する。
2. 比較対象局面セット(序盤・中盤・可能ならFFO局面)を用意する。
3. 本エンジンとEdaxで同一局面を評価し、比較表(局面・両エンジンの評価値・符号一致・乖離幅)を生成する。
4. 比較結果を分析し、以下の観点でレポートする:
   - **符号一致率**: 「どちらが優勢か」の判定(評価値の正負)が両エンジンで一致する局面の割合。中盤局面で著しく低い(例: 目安として70%を大きく下回る)場合は、本エンジンの評価関数に問題がある可能性が高いため、具体的にどの局面でどう食い違うかを記載し、原因の仮説(実装バグか、単に評価方針の違いによる妥当な差か)を考察する。
   - **明白な悪手の検出**: 少なくとも数局面について、明らかに悪い手(隅の隣接マスへの着手等、定跡上の既知の悪手)を本エンジンが「悪い」と評価しているか(Edaxも同様に悪いと評価するはずの手)を個別に確認する。
   - 完全一致は求めないが、**符号が系統的に逆転している、評価値のスケールが明らかに異常(桁違い)、既知の悪手を明確に良い手と評価する、といった重大な問題が見つかった場合は、それを明記した上でオーケストレーターにエスカレーションする**(本タスクの範囲では評価関数自体の修正は行わない。問題発見がゴール)。
5. 本タスクで見つかった問題点・改善提案は、作業ログとレポートの両方に記載する(将来の評価関数改善タスクの入力となる)。

### 2026-07-08 implementer: Edax入手・比較基盤構築・初回比較の実施

**Edaxの入手(背景の優先順位1を採用)**
- 開発機にはWindows上に `gcc`/`make`/`cargo` がPATH直下には無いが、`~/.cargo/bin` にRust toolchainがインストール済みで、かつWSL(Ubuntu-22.04)経由でネットワークアクセス・ビルドツールチェーンが使えることを確認した。
- 公式リリースページ(https://github.com/abulmo/edax-reversi/releases/tag/v4.6, 2024-12-18公開)を確認したところ、Windows向け配布ZIP `edax-4.6-MS-windows-x86.zip` が存在したため、これをそのまま採用(ソースビルドは不要だった)。
- `wEdax-x86-64.exe`(および AVX2非対応CPU向けの `-v2`/`-v3` 版)が単体で動作することを確認済み。付属の `data/eval.dat`(学習済み評価重み)も同梱されており、これを `-eval-file` で指定して使用した。
- ダウンロード・展開手順は `bench/edax-compare/download-edax.ps1` としてスクリプト化した(`Invoke-WebRequest` + `Expand-Archive`)。Edaxのバイナリ・zipそのものはライセンス上の再配布可否が未確認のため、`.gitignore` に `bench/edax-compare/edax-extract/` と `bench/edax-compare/*.zip` を追加してコミット対象から除外した(`git status --ignored` で除外を確認済み)。

**Edaxの呼び出し方式の調査**
- 対話モード(`-edax` プロトコル、`setboard`/`level`/`hint` 等をパイプ経由で流し込む方式)はPowerShellの標準入力パイプ経由では2行目以降のコマンドが実行されない(原因未特定、エンコーディング関連の問題の可能性)という再現性のある問題があり、信頼できる比較には使えないと判断して不採用にした。
- 代わりに、非対話のバッチモード `wEdax-x86-64.exe -solve <problem.obf> -l <level> -vv` (OBF形式の1行ファイルを渡し、深さ`level`までの反復深化の最終行から評価値を読み取る)を採用した。これはFFO公式値との一致を含め安定して動作することを確認済み(下記結果参照)。

**比較基盤の実装**
- `engine/src/bin/eval_cli.rs`(新規): 本エンジンの既存の公開API(`Engine::analyze`、T008のJSON Workerプロトコルそのもの。`eval`/`search`/`protocol` 等の非公開モジュールには一切手を加えていない)経由で局面を評価するCLI。サブコマンド:
  - `gen`: 初期局面からのランダム自己対戦(依存クレート追加を避けるための自作xorshift64/splitmix64乱数)で、指定した空きマス数範囲の局面を指定シードから再現可能に生成する。**定石書の名前付きオープニングではなく「代表的な一局面」の意図で生成したもの**であることに注意(タスク仕様の「代表的な序盤局面」を、既存の名前付き定石データが無い状態でも捏造せずに用意するための設計判断)。
  - `eval`: 局面配列(JSON)を受け取り、静的評価(depth=0)と指定深さ/完全読み設定での評価を返す。
  - `moves`: 単一局面の全合法手ランキング(`allMoves: true`、T018で追加済みのAPI)を返す。
  - `apply`: 単一局面に1手適用した後の局面を返す(悪手検出チェックで、実際にその手を打った後の局面をEdaxにも独立評価させるために使用)。
  - `cargo build --release -p engine --bin eval_cli` でビルド。既存の `cargo test -p engine`(単体テスト)には影響しない追加バイナリのみであることを確認済み。
- `bench/edax-compare/run-comparison.py`(新規): 局面生成→本エンジン評価→Edax評価(`-solve -vv` の出力をパースして読み取り)→符号一致率集計→悪手検出チェック→`report.md`/`raw_results.json` 生成、を一括実行するオーケストレーションスクリプト。Python 3.11(開発機に既存)を使用、追加インストール不要。

**トラブル: 完全読み比較でのハング(重要、再発防止策込み)**
- 初回実装時、FFO局面(#40〜#49)全問を本エンジン側で `exactFromEmpties=64`(常に完全読み)で評価する実装にしていたところ、**FFO #49 相当の局面の完全読みで長時間ハングした**。これはT009(`engine/tests/ffo_bench.rs`)で既に判明していた既知の問題の再発だった: 本エンジンの `solve_exact`(T006)はMPC・安定石カット等の高度な枝刈りを持たないプレーンなalpha-beta+TTであり、T009の実測では空きマス24以上の問題(#45〜#49)が1問あたり数分〜約31分(#48)、#49は同テスト内でも完走を確認できていない(打ち切り)。
- オーケストレーターから指摘を受け(プロセスkillはオーケストレーター側の権限では実行できなかったため、こちらのセッションで `taskkill /F /PID <pid>` を実行してハングしていた `eval_cli.exe`/`python3.11.exe` を強制終了した)、以下の対応を行った:
  1. `run-comparison.py` に `FFO_FAST_MAX_EMPTIES = 23`(T009の `FAST_MAX_EMPTIES` と同じ値・同じ根拠)を導入し、**空きマス数24以上のFFO局面(#45〜#49)は本エンジン側の完全読み評価をスキップする**ようにした(オーケストレーターからは「#40〜#48は従来通り実施してよい」との指示だったが、T009の実測記録(#45〜#48も1問数分〜31分かかる)を踏まえると#48まで含めるだけでも合計1時間近くかかりうると判断し、より安全側のT009既存の閾値(#40〜#44のみ)に合わせて縮小した。この判断の根拠と経緯は `report.md` にも明記した)。
  2. Edax側は非常に高速(#40がミリ秒オーダー)なため、#45〜#49も含めて全10問評価している(3者比較の表には残るが、本エンジンの列だけ `skipped` になる)。
  3. `run()` ヘルパーに `timeout_sec` を追加し、本エンジンのFFO完全読み呼び出し(バッチ全体)に900秒のタイムアウトを設定(想定外に重い局面が紛れ込んだ場合に無限に待たず打ち切ってエラーにするための安全策)。
  4. `print()` を `flush=True` に変更(初回ハング時、標準出力がブロックバッファリングされていて進捗が全く見えず、外部から「進行中か完全に停止しているか」の判別ができなかったため)。
- 修正後の再実行では、局面生成(8+20局面)〜本エンジン評価(opening/midgame depth10 + FFO#40-44完全読み)〜Edax評価(opening/midgame depth10 + FFO#40-49完全読み)〜悪手検出チェック(5局面)まで含めて**約12分**で完走することを確認した(`positions.json` 06:30:57生成 → `report.md` 06:43:03生成)。

**比較結果の要約(詳細は `bench/edax-compare/report.md` 参照)**
- 比較局面数: opening 8 + midgame 20 + ffo 10 = 合計38局面(うちffoは5問(#40-44)のみ本エンジン側完全読み評価、残り5問(#45-49)はEdax値とFFO公式値のみ記載)。
- 符号一致率: 全体72.7%(24/33、ffo以外はdepth10のdepth-limited探索での比較)。カテゴリ別: opening 37.5%(3/8)、midgame 80.0%(16/20)、ffo 100%(5/5)。
  - openingの低い一致率は、個別局面を確認したところ**両エンジンとも評価値がほぼ0(±1石未満)の互角局面での符号ノイズ**が原因であり、系統的な不一致ではないと判断した(「両エンジンとも|discDiff|>=1.0石」に絞ると全体90.9%まで改善。詳細はreport.md「符号一致率が低いカテゴリの原因分析」参照)。
- FFO局面(#40-44)は本エンジン・Edax・FFO公式正解値の3者が完全一致(5/5)。T009に続き終盤ソルバーの正しさが裏付けられた。Edax側は#45-49も完走しており、その値もFFO公式値と一致していた(本エンジン側は前述の理由で未評価)。
- 明白な悪手の検出: 隅隣接のX打(b2/g2/b7/g7)が合法手として存在しかつ隅が空いている局面を17件発見し、うち5件についてEdaxとの突き合わせを実施。**5件中5件で、本エンジンがそのX打を悪手と判定し(ランキング下位)、Edaxも「そのX打を打った後の局面は本エンジンの最善手より明確に悪い」と評価しており、一致した**。明確な悪手を良い手と誤判定するような重大な不具合は見つからなかった。

**エスカレーション事項(オーケストレーターへ、重要)**
- **重大なバグや符号系統的逆転は見つからなかった**が、1点、UI設計上の懸念に繋がりうる発見があった: 本エンジンのdepth-limited探索(opening/midgame)の評価値は、Edaxの値より一桁以上大きくなることが多い(例: midgame-1で本エンジン-260.8 vs Edax-43.0)。これはバグではなく `engine/src/eval.rs` の `CORNER_WEIGHT=2500`(隅1個=25石相当)・`STABLE_WEIGHT=1500`(安定石1個=15石相当)という、最終石差の予測値ではなく「探索誘導用の目安の重み」として意図的に大きく設定された値が原因(コードコメントに明記済みの設計方針通り)。
  - **懸念点**: もしアプリUI(`app/src/`)が `discDiff`(本エンジンの応答値)をユーザーに「石差」としてそのまま表示している箇所があれば、中盤局面で「+260石差」のような非現実的な表示になり、ユーザーに誤解を与えるおそれがある。本タスクのスコープでは `eval.rs` の修正もUI側の調査も行っていないため、UI側で `discDiff` をどう表示しているかの確認、および必要であれば評価関数の較正(重みを現実的な石差スケールに縮小する)を将来タスクとして検討することを推奨する。
- 上記以外はスコープ外の大きな問題は無く、`eval.rs` 自体の修正は行っていない(タスク方針通り)。
