# T146: lv10ラベル付けの所要時間見積もり

## 目的(ユーザー指示 2026-07-20)

「lv16のEdaxラベルとWTHOR石差ラベルに有意差がない(T144)なら、lv16である必要がない。lv10なら時間のかかる帯域も速くなる期待がある。lv10でどれくらい時間がかかるか見積もって」— **WTHOR全局面(約443万レコード、74,024局の全手)へのEdaxラベル付け**を level 10 で行った場合の所要時間を、実測ベースで見積もる。**計測と見積もりレポートのみ。コーパス生成・本番コード変更は行わない。**

## 背景・事実

- 現行の生成方式(T127g/i確立): warm 32親束バッチ+`wEdax-x86-64-v3.exe`(AVX2)×8シャード並列、`-n 1`決定性仕様。1レコード=1局面の**全合法手(子局面)**を評価してラベル化。exactEmptiesThreshold=20(空き20以下=完全読み、21以上=level探索)。
- 実績: expanded1m(100万レコード)の新規80万件生成に実質約5日(方式改善込み・lv16)。重い帯(空き20-29)で約4.5-5k件/h、軽い帯で140k件/h超と帯差が極端。
- 既存参考見積もり: 「WTHOR全局面400万はlv16で約8日」(2026-07-17時点、方式改善前の概算)。
- 生成パイプライン: `bench/edax-compare/gen_teacher_corpus.py`(T143で堅牢化済み。**本タスクでは変更禁止**)。A/B計測の前例: T127f/T127i(ハーネスは未コミット、ログ logs/codex-T127*系とタスクファイル作業ログに手法記録あり)。
- 全手ラベル付けの対象分布: 74,024局の全手=各局約60局面、空き1〜59にほぼ一様に分布(1局面が空きnに1個ずつ)。ただし空き20以下はexact経路なのでlevelを下げても速くならない点に注意。

## 要件

1. **サンプル実測(lv16 vs lv10)**: WTHOR対局から実局面をサンプリングし(既存コーパスの局面流用可)、空き帯別(最低: 21-25 / 26-30 / 31-40 / 41-50 / 51-59 の5帯、各20〜40局面)に、本番同等の呼び出し方(v3バイナリ、warm束方式または現行バッチ相当、`-n 1`、全合法手評価)で 1レコードあたり壁時計を lv16 と lv10 の両方で計測する。計測は専有状態で行う(現在他の重い処理はない)。
2. **exact帯の実測または実績流用**: 空き20以下のexact経路のコスト(levelに非依存)を、expanded1m生成の実績データ(シャードログ・meta等)から帯別スループットとして推定するか、少数サンプルで実測する。
3. **外挿**: 443万レコードの空き分布(74,024局×全手、空き1-59ほぼ一様=各空き約7.4万レコード)に帯別実測を掛けて、8シャード並列前提の総所要時間を見積もる。以下を分けて提示:
   - (i) lv16での総時間(現行方式の再見積もり)
   - (ii) lv10での総時間
   - (iii) 内訳: exact帯(空き≤20、level非依存)/ level帯(空き21+)の時間と、lv10化による短縮が総時間の何割か
   - (iv) 感度: exactEmptiesThreshold を 20→18 や 16 に下げた場合の追加短縮(参考値。exact帯が支配的な場合に判断材料になる)
4. **レポート**: bench/edax-compare/ に t146_lv10_estimate_report.md(+必要ならmeta json)として計測条件・生データ要約・見積もり表・仮定と誤差要因を記録し、コミット・push。計測用スクリプトはscratchpad(リポジトリ外)でよいが、再現に必要なコマンド・条件はレポートに明記。
5. 長時間実行ルール: サンプル計測は帯ごとに逐次結果をログへ書き出す(一括書き出し禁止)。総計測時間が2時間を超えそうなら、サンプル数を減らして精度とトレードオフする(見積もり用途なので±20%程度の精度で十分)。

## スコープ外(やらないこと)

- 443万ラベル付けジョブの実行・開始(見積もりのみ。実行判断はユーザー)
- gen_teacher_corpus.py等の本番コード変更、ラベル品質(lv10 vs lv16の値の差)の評価 — 品質検証は実行判断後の別タスク
- app/engine変更(Pages確認不要)
