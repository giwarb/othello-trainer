# T059: 評価値スケール異常の修正(パターン評価のクランプ欠如+終局完全読みの誤表示バグ)

## 目的

ユーザー報告: 棋譜解析モードで、終盤の評価値が「+105」「-109」のような、オセロの理論上あり得ない値(石数差の絶対値は最大64)になり、グラフも定石を外れた直後から不自然に急落したまま高原状態になる。調査の結果、根本原因が判明した(explorer調査済み)。ユーザーからは追加で「スケーリング問題は終局に限らず中盤戦でもEdaxのように最終石差予想として一貫させてほしい。レファレンス実装を読み込んでほしい」との指示があり、本タスクではEdaxの実装を再確認しつつ、中盤・終局を通じて評価値が常に「妥当な石数差予測値」の範囲に収まるよう修正する。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように、調査済みの根本原因)

1. `engine/src/search.rs`の`search_all_moves_with_eval`は、全合法手を1つの共有時間予算(棋譜解析では`timeMs:1500`)で完全読み(`solve_exact_bounded`)しようとする設計。深い終盤局面(空き17前後)では、一部の候補手だけが時間切れになり、`static_eval`(ヒューリスティック評価)にフォールバックしうる(その手だけ`is_exact=false`になる)。
2. `engine/src/search.rs`(86-90行目付近)の`static_eval`は`PatternWeights::score()`(`engine/src/pattern_eval.rs`113-123行目、T041-45で学習したパターン評価)を×100して返す。**この`PatternWeights::score()`にはクランプ(出力範囲の上限・下限)が一切無い**。T024の較正(`eval.rs`側の手作り重み)はこのパターン評価の対象外であり、学習データが薄い(WTHOR実戦棋譜にあまり出現しない)終盤寄りの局面では、線形和が理論上の石数差の範囲(±64)を大きく超える値(105等)を出しうる。
3. `app/src/analysis/analyzeGame.ts`(214-221行目付近)の`evalSource`/`isExact`は、**その局面の最善手(`best`)の評価方式だけを見て決めており、実際に打たれた手(`played`)の評価方式を見ていない**。そのため、`played`側がタイムアウトでヒューリスティックにフォールバックしていても、`best`が完全読みできていれば「終盤(完全読み確定)」と表示されてしまう(実際には未確定の近似値なのに、確定した値であるかのように誤表示する)。
4. `app/src/analysis/analyzeGame.ts`(195行目付近)の`lossDiscs = best.discDiff - played.discDiff`にクランプが無いため、上記の異常な`played.discDiff`がそのままロスに反映され、105前後の異常なロスになる。
5. T056(累積ロス方式)の`applyCumulativeEvaluation`(255-270行目付近)は、一度でも異常に大きいロスが混入すると、以降悪手0が続く限りその異常値が累積評価値に永続的に残り続ける設計(これ自体はT056の意図した設計であり、根本原因である1〜4を修正すれば正しく機能する)。

## Edaxリファレンス実装の再確認(ユーザー指示)

T048着手前の調査で、Edaxの評価関数(`src/eval.c`)がパターンベースの学習済み重み(`EVAL_N_WEIGHT=226315`、`EVAL_N_PLY=61`段階の進行度別テーブル)を使っていることを確認済みだが、**Edaxがこの重み和の出力を物理的に妥当な範囲(石数差の理論上限)にクランプしているか、学習時にどう範囲を抑えているか**は未確認だった。本タスクで`src/eval.c`(https://github.com/abulmo/edax-reversi/blob/master/src/eval.c )を再度実際に読み、以下を確認すること:
- 評価値の最終出力(`evaluate`関数等)にクランプ処理があるか、あるならその範囲・実装方法。
- 進行度別(ply)テーブルの学習が、範囲外の出力を防ぐ設計(正則化・出力範囲の学習データ由来の自然な収束等)になっているか、参考になる記述がコード中にあるか。
- 中盤・終盤を通じて評価値が一貫して「最終石差の予測値」というセマンティクスを持つよう設計されているか(進行度が上がるほど精度が上がるだけで、スケール自体は変わらない設計になっているか)。

この調査結果を踏まえ、自作エンジンのパターン評価にも同様のクランプ・整合性を導入すること。

## 変更対象

- `engine/src/pattern_eval.rs`(または`static_eval`の呼び出し元、`engine/src/search.rs`) — `PatternWeights::score()`の出力(またはその呼び出し結果)に、物理的に妥当な範囲のクランプを追加する。単純に`[-64, 64]`(石数差の絶対的な理論上限)でクランプするか、より厳密に「現在の石数差 ± 残り空きマス数」(その局面から実際に変動しうる最大幅)でクランプするか、実装しやすい方を採用してよい(後者の方がより正確だが、前者でも実用上十分)。**中盤ヒューリスティック探索・終盤タイムアウト時フォールバックの両方でこのクランプが効くよう、`static_eval`を経由するすべての経路に適用すること**(ユーザー指示「中盤戦でも」に対応)。
- `app/src/analysis/analyzeGame.ts` — `evalSource`/`isExact`の判定を、`best`だけでなく`played`(実際に打たれた手)の評価方式も考慮するよう修正する。`played`がヒューリスティックフォールバックだった場合は、`evalSource`を`'midgame'`(または相当のラベル)にすること(完全読み区間であっても、実際にタイムアウトした手は「確定」として表示しない)。
- `app/src/analysis/analyzeGame.ts` — `lossDiscs`の計算に、念のため理論上限に基づくクランプ(異常値の混入を防ぐ最終防御)を追加する(エンジン側のクランプで根本解決するが、表示層でも二重に防御しておく)。

## 要件

1. 棋譜解析モードで表示されるすべての評価値・ロスが、石数差として物理的にあり得ない値(絶対値64を超える値)にならないこと。
2. 実際に打たれた手がタイムアウト等でヒューリスティックフォールバックした場合、評価ソース表示が「終盤(完全読み確定)」ではなく正しく「中盤(探索)」相当になること。
3. 中盤ヒューリスティック探索の評価値も、Edaxのような「最終石差予測値」として一貫した妥当な範囲に収まること(クランプが中盤・終盤問わず適用されること)。
4. FFO終盤テストが引き続き全問正解すること(クランプが完全読みソルバー自体の正しさに影響しないことの確認。クランプはヒューリスティック評価`static_eval`のみに適用し、`endgame::solve_exact`系の全数探索ロジックには適用しないこと)。
5. Edax比較(`bench/edax-compare`)・自己対戦検証で、クランプ導入前後で明確な棋力低下が無いことを確認する(クランプは異常値の抑制のみが目的であり、通常の局面には影響しないはずだが、念のため確認する)。
6. `cargo test --workspace`・`npm test`が全件パスすること。
7. ユーザーが報告したシナリオ(60手前後の棋譜、定石を外れた後に評価値が105等の異常値になる)を実際に再現し、修正後は妥当な範囲に収まることを確認する。

### 判断に迷った点・オーケストレーターへの報告事項

- **T056累積評価モデル(`blackAdvantageBefore`/`blackAdvantageAfter`)自体は、個別の手の評価値・ロスが完全に理論上限内に収まっていても、多数の中程度のロス(悪手ではない誤差レベルの差)が同じ方向に積み重なるゲームでは、依然として理論上限(±64)を超えた値を表示しうる。** 今回の自己対戦テストでは弱いエンジン同士の対局(人為的な設定)でのみ顕在化し、実際のWTHOR実戦棋譜では発生しなかったため、本タスクの受け入れ基準(根本原因1〜4の修正)は満たしていると判断したが、実際のユーザーの棋譜(人間の対局、様々な棋力)ではどの程度の頻度でこの現象が起こりうるかは未検証。もし今後同様の報告が再発する場合は、T056の累積モデル自体(例えば累積値にも同様のクランプを設けるか、あるいは「現在のブラウザに表示する評価値は累積値ではなく生の局面評価値にする」等の設計変更)を再検討する必要があるかもしれない。本タスクの「やらないこと」に明記されている通り、この漸化式自体の変更は行っていない。
