# T055: 終局・パス判定の非同期タイミングを統一し「評価値が終局近くで消える」問題を解消

## 目的

ユーザー報告: 「評価値の表示が最初にできなかったり、終局近くで消えたり、いろいろと不安定」のうち、「終局近くで消える」問題に対応する(「最初に出ない」問題はT054で対応済み)。explorer調査により、以下の複数要因が判明している:

1. 中盤練習・詰めオセロモードで、パス発生時の手番反転(`sideToMove`の更新)が**別のuseEffectで非同期に**行われるため、パス直後の1レンダーだけ盤面評価オーバーレイ取得用のuseEffectが古い手番判定のまま`setOverlayMoves(null)`を呼んでしまい、オーバーレイが一瞬消えてすぐ復帰する(ちらつき)。パスは終盤ほど発生しやすいため「終局近くで消える」と体感されやすい。
2. 中盤練習モードの終局判定(`checkEnd`)が、空きマス数が一定閾値(24)以下になった**最初の呼び出しで即座に**結果画面へ遷移する設計になっており、これが「まだ打てるはずなのに評価表示が突然消えて結果画面に切り替わった」という体感を生んでいる可能性がある(意図的な設計なのか、早すぎる打ち切りなのか未確認)。
3. 盤面評価オーバーレイのロジック(`moveEvalOverlayLogic.ts`)が、候補手が0件(合法手なし=パスすべき局面、または真の終局)の場合に一律で空を返すため、「一時的にパス処理中で評価が空の状態」と「本当にゲームが終わった状態」を区別できず、両方とも同じ見た目(オーバーレイが消える)になる。
4. 対局モード(`game/gameLoop.ts`)・棋譜解析(`analysis/branchTree.ts`の`resolveMover`)は手番・終局判定を**同一レンダー内で同期的に**解決しており、上記1のような1テンポ遅れが無い。中盤練習・詰めオセロだけが非同期パターンになっている(実装の不統一)。

## 背景・コンテキスト(このリポジトリを知らない前提で読めるように)

- `app/src/midgame/PracticeMode.tsx`: 中盤練習モード本体。`checkEnd`(212-253行目付近)が空きマス数`<=24`で即座に`phase:'result'`へ遷移する。パス処理は`resolveMover`(`app/src/joseki/resolveMover.ts`または同等の共通モジュール)を使うが、その結果を反映する`setSession`呼び出しが、着手適用処理とは別のuseEffect(294-306行目付近)に分離されている。盤面評価オーバーレイ取得用のuseEffect(350-360行目付近)は`session.sideToMove === session.humanSide`を見て判定するため、パス直後の1レンダーではこの値がまだ古いままになる。
- `app/src/tsume/PlayMode.tsx`: 詰めオセロモード本体。同様のパス処理パターン(275-287行目付近)、オーバーレイ取得useEffect(333-343行目付近)を持つ。
- `app/src/game/gameLoop.ts`: 対局モードの手番・終局解決ロジック(`afterMove`、93-129行目付近)。着手適用と同期して手番反転・終局判定を行う「正しい」参照実装。
- `app/src/analysis/branchTree.ts`の`resolveMover`(64-66行目付近): 棋譜解析のフリー分岐探索でも同期解決されている。
- `app/src/components/moveEvalOverlayLogic.ts`(36行目付近): `allMoves`が空配列のとき無条件で空のMapを返す。
- T054のreviewer指摘(参考情報、本タスクと関連あり): Worker共有化によりコンポーネントのアンマウント後も進行中リクエストが解決するようになった。パス/終局処理の非同期タイミングを修正する際、古いレンダーへの`setState`呼び出しが増えないよう注意すること。

## 変更対象

- `app/src/midgame/PracticeMode.tsx` — パス発生時の手番反転処理を、着手適用と同期した処理(`game/gameLoop.ts`の`afterMove`と同じパターン)に変更し、別useEffectでの非同期反映をやめる。
- `app/src/tsume/PlayMode.tsx` — 同様に同期化する。
- `app/src/midgame/PracticeMode.tsx`の`checkEnd` — まず「空きマス数`<=24`で即座に結果画面へ遷移する」のが意図的な設計か、早すぎる打ち切りバグかを調査する。意図的な設計であれば、遷移前に一瞬「決着が確定しました」等の分かりやすい表示を挟む、または閾値・タイミングを見直す。バグであれば正しい終局判定(双方に合法手が無い)に修正する。
- `app/src/components/moveEvalOverlayLogic.ts` — 「候補手0件」の状態が「一時的なパス処理中」なのか「真の終局」なのかを呼び出し側で区別できるよう、必要であれば型・戻り値を拡張する(過剰な変更は避け、実際にちらつきの原因になっている箇所を優先して直すこと)。

## 要件

1. 中盤練習・詰めオセロモードで、パスが発生しても盤面評価オーバーレイが一瞬消えて復帰する(ちらつく)ことがないこと。
2. 中盤練習モードの終局判定について、意図的な設計かバグかを明確にした上で、ユーザーにとって唐突に感じられない遷移になっていること(調査結果と対応方針を作業ログに記録すること)。
3. 対局モード・棋譜解析モードの既存の同期的な終局・手番解決パターンとの一貫性を高めること(全モードで同じ設計パターンを使うことが望ましいが、無理に統一する必要はない。差異がある場合はその理由を作業ログに記録すること)。
4. 新規テスト: パス発生を含む対局シナリオ(中盤練習・詰めオセロ双方)で、パス直後にオーバーレイ用の状態が不必要に`null`にならないことを検証するテストを追加する。
5. 既存のテストが壊れないこと。
